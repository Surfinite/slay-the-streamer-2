# Plan B prep — smoke test design

**Date:** 2026-05-08
**Status:** approved (in brainstorming session); awaiting plan

## Purpose

Validate two architectural concerns from `notes/06-followups-and-deferred.md` *before* writing Plan B's full implementation:

1. **#6 Godot autoload from a mod assembly.** Can a mod register a `Node`-derived class to receive frame ticks at runtime? `AddAutoloadSingleton` requires a class reference resolvable at engine boot, which a runtime-loaded mod assembly can't provide. The two candidate runtime mechanisms are `Engine.RegisterSingleton(...)` + `Root.AddChild(...)`, or plain `Root.AddChild(...)` only.
2. **#7 Harmony deadlock risk.** Can a Harmony prefix safely `await Voter.Start(...).AwaitWinnerAsync()` on the Godot main thread when the underlying timer dispatches its close callback back through the same dispatcher? Plan A's `RunContinuationsAsynchronously` design *should* prevent this, but the failure mode (main-thread `.Result`/`.Wait()` blocking the dispatcher's pump) is real and we want to confirm the working path before committing to the pattern across 6+ Plan B Harmony patches.

## Scope

Two smoke vote scenarios using `FakeChatService` (in-memory, no network, no auth):

- **Smoke A** — fired from `[ModInitializer("Init")]`. Exercises the dispatcher mechanics and `AwaitWinnerAsync` from a startup context.
- **Smoke B** — fired from a `[HarmonyPatch(typeof(NMainMenu), "_Ready")]` prefix. Exercises the realistic Plan-B pattern: Harmony prefix on the main thread that awaits a vote.

Both vote scenarios:
- 3-second duration, 3 options (`["A","B","C"]`), no policy overrides.
- Inject `"#0"` once via `FakeChatService.SimulateMessage(...)` so the expected winner is index `0` (deterministic).
- Log `"Smoke X starting"` at vote start and `"Smoke X: winner=N"` at vote end via `TiLog.Sink` → `MegaCrit.Sts2.Core.Logging.Log.Info`.

## Out of scope

- Real Twitch IRC connection (`TwitchIrcChatService` — Plan B Phase 1).
- Player-input substitution patterns (the actual vote-replaces-click) — Plan B core hooks.
- Card/event/map decision Harmony patches — Plan B Phase 2+.
- UI prompts, sound, anything visible to the streamer beyond log lines.
- Configurable vote labels or durations — fixed for the smoke.

## Component contracts

### `DispatcherAutoload : Node` (Godot-side)

```csharp
public partial class DispatcherAutoload : Node {
    public void Post(Action action) =>
        CallDeferred(MethodName.Run, Callable.From(action));
    private void Run(Callable callable) => callable.Call();
}
```

Single responsibility: hop an arbitrary `Action` onto the next Godot main-thread frame via `CallDeferred`. The `MethodName.Run` constant is auto-generated by Godot's source generator.

### `GodotMainThreadDispatcher : IMainThreadDispatcher`

```csharp
public sealed class GodotMainThreadDispatcher : IMainThreadDispatcher {
    private DispatcherAutoload? _autoload;
    public void SetAutoload(DispatcherAutoload a) => _autoload = a;
    public void Post(Action action) =>
        (_autoload ?? throw new InvalidOperationException(
            "GodotMainThreadDispatcher.Post called before SetAutoload."))
            .Post(action);
}
```

Holds a reference to a registered `DispatcherAutoload`, delegates `Post` to it. Throws on missing autoload (caller bug). Plan A's `IMainThreadDispatcher` interface contract requires `Post(Action)` to schedule the action on the dispatcher thread; this delegates to Godot's frame-deferred queue.

### `ModEntry.Init()` (the `[ModInitializer]`)

```csharp
[ModInitializer("Init")]
public static class ModEntry {
    internal static FakeChatService SmokeChat = null!;   // disposable
    
    public static void Init() {
        Log.Info("[slay_the_streamer_2] mod loading...");
        
        // 1. Register dispatcher autoload (test #6)
        var autoload = new DispatcherAutoload { Name = "DispatcherAutoload" };
        bool registeredSingleton = false;
        try {
            Engine.GetMainLoop().Root.AddChild(autoload);
            Engine.RegisterSingleton("DispatcherAutoload", autoload);
            registeredSingleton = true;
            Log.Info("[slay_the_streamer_2] autoload: Engine.RegisterSingleton + Root.AddChild");
        } catch (Exception e) {
            Log.Warn($"[slay_the_streamer_2] autoload registration via RegisterSingleton failed: {e.Message}");
            // Fall back to plain Root.AddChild only (autoload may already be added)
            if (autoload.GetParent() == null) {
                Engine.GetMainLoop().Root.AddChild(autoload);
            }
            Log.Info("[slay_the_streamer_2] autoload: Root.AddChild fallback");
        }
        
        // 2. Wire IMainThreadDispatcher
        var dispatcher = new GodotMainThreadDispatcher();
        dispatcher.SetAutoload(autoload);
        
        // 3. Plan A logging passthrough (verified thread-safe in notes/03)
        TiLog.Sink = (level, msg, ex) => {
            switch (level) {
                case LogLevel.Error: Log.Error(ex is null ? msg : $"{msg} :: {ex}"); break;
                case LogLevel.Warn:  Log.Warn(msg); break;
                default:             Log.Info(msg); break;
            }
        };
        
        // 4. Smoke wiring (disposable)
        var clock = new SystemClock();
        var scheduler = new SystemTimerScheduler(clock);
        SmokeChat = new FakeChatService();
        SmokeChat.ConnectAsync("smoke", new ChatCredentials("smokebot", "abc")).GetAwaiter().GetResult();
        var coord = new VoteCoordinator(SmokeChat, clock, scheduler, dispatcher);
        Voter.Default = coord;
        
        // 5. Apply Harmony patches (game only auto-PatchAll's if no [ModInitializer] is present)
        new Harmony("slay_the_streamer_2").PatchAll(Assembly.GetExecutingAssembly());
        
        // 6. Smoke A: fire-and-forget vote from this startup context
        _ = SmokeRunner.RunSmokeA(SmokeChat);
        
        Log.Info("[slay_the_streamer_2] Init complete.");
    }
}
```

Plan-B reduction after smoke: remove the smoke section (4 + 6) and the static `SmokeChat` field; replace with the real `TwitchIrcChatService` wiring once that's built.

### `SmokeRunner` (disposable)

```csharp
public static class SmokeRunner {
    public static async Task RunSmokeA(FakeChatService chat) =>
        await Run(chat, "smoke-A", "alice", "alice-id");
    
    public static async Task RunSmokeB(FakeChatService chat) =>
        await Run(chat, "smoke-B", "bob", "bob-id");
    
    private static async Task Run(FakeChatService chat, string label, string login, string userId) {
        try {
            Log.Info($"[{label}] starting...");
            var session = Voter.Start(label, new[] { "A", "B", "C" }, TimeSpan.FromSeconds(3));
            chat.Inject(new ChatMessage(
                UserId: userId, Login: login, DisplayName: login, Text: "#0",
                ReceivedAt: DateTimeOffset.UtcNow,
                IsSubscriber: false, IsModerator: false, IsVip: false));
            var winner = await session.AwaitWinnerAsync();
            Log.Info($"[{label}] winner={winner} (expected 0)");
        } catch (Exception e) {
            Log.Error($"[{label}] FAILED: {e}");
        }
    }
}
```

Returns `Task` (not `void`) and wraps the body in `try/catch` so any awaited exception lands in `Log.Error` rather than becoming an unobserved task exception. Callers fire-and-forget via `_ = SmokeRunner.RunSmokeA(...)`. The `Inject` call uses Plan A's actual `FakeChatService` API: a `ChatMessage` record (positional ctor: `UserId, Login, DisplayName, Text, ReceivedAt, IsSubscriber, IsModerator, IsVip`).

### `SmokeMainMenuPatch` (disposable)

```csharp
[HarmonyPatch(typeof(NMainMenu), "_Ready")]
internal static class SmokeMainMenuPatch {
    private static int _fired = 0;
    
    static void Prefix() {
        if (Interlocked.Exchange(ref _fired, 1) == 1) return;
        _ = SmokeRunner.RunSmokeB(ModEntry.SmokeChat);
    }
}
```

First-time-only guard via `Interlocked.Exchange` so navigating away and back to the main menu doesn't spam votes. `void` prefix (no Harmony return) means the original `_Ready` always runs unmodified.

## Flow

### Smoke A (ModInitializer context)

```
game loads slay_the_streamer_2.dll → calls ModEntry.Init() on Godot main thread
  Init: register autoload, wire dispatcher, set Voter.Default, Harmony.PatchAll
  Init: fire-and-forget SmokeRunner.RunSmokeA(SmokeChat)
RunSmokeA: Voter.Start(...), SimulateMessage("alice", "#0"), await AwaitWinnerAsync()
    ↓ (yields; Init returns; game continues startup)
3 seconds elapse → close timer fires (System.Threading.Timer, threadpool thread)
  → _dispatcher.Post(() => CloseNowInternal(byTimer: true))
  → DispatcherAutoload.Post → CallDeferred queues for next frame
next Godot frame tick → DispatcherAutoload.Run → CloseNowInternal on main thread
  → ComputeWinner returns 0, TCS.TrySetResult(0), Closed event fires
  → RunContinuationsAsynchronously schedules RunSmokeA continuation on threadpool
RunSmokeA continuation: Log.Info("[smoke-A] winner=0 (expected 0)")
```

### Smoke B (Harmony prefix context)

```
player navigates to main menu → NMainMenu._Ready called on Godot main thread
SmokeMainMenuPatch.Prefix runs (synchronous void)
  first-time guard passes, fire-and-forget SmokeRunner.RunSmokeB(SmokeChat)
RunSmokeB: same shape as RunSmokeA
  ↓ (yields; Prefix returns; original NMainMenu._Ready continues)
3 seconds elapse → same timer/dispatcher/TCS flow as Smoke A
RunSmokeB continuation: Log.Info("[smoke-B] winner=0 (expected 0)")
```

## Success matrix

| Smoke A | Smoke B | Conclusion |
|---|---|---|
| ✅ winner=0 | ✅ winner=0 | green light for Plan B's Harmony+await pattern |
| ✅ | ❌ silent | autoload+dispatcher fine; Harmony prefix not auto-applied or filtering issue (PatchAll didn't pick it up, or NMainMenu signature mismatch) |
| ✅ | ❌ hang | main-thread deadlock — `AwaitWinnerAsync` needs revisiting (Plan A design assumption violated) |
| ❌ silent | n/a | autoload registration didn't take effect — investigate fallback path or alternative attachment |
| ❌ hang at startup | n/a | dispatcher's `CallDeferred` or autoload Node not receiving frame ticks |

The "expected 0" vs actual winner index gives extra signal: if the vote completes but winner is wrong (e.g., 1 or 2 instead of 0), the `SimulateMessage` injection didn't reach the session — likely a `FakeChatService` lifecycle issue or a parsing-policy regression.

## Risks and assumptions

- **Assumption: `Engine.RegisterSingleton` exists in Godot 4.5.1 and accepts a `Node` instance.** If not, the fallback path (`Root.AddChild` only) is the actual mechanism.
- **Assumption: a `Node` added under `Root` receives `_Process` ticks and processes its `CallDeferred` queue.** This is core Godot behavior, but we haven't verified it from a runtime-loaded mod assembly.
- **Assumption: `Harmony.PatchAll` correctly picks up `[HarmonyPatch]` attributes on internal types.** Standard Harmony behavior, but Godot's source-gen interaction with `[HarmonyPatch]` is unverified for us.
- **Assumption: `NMainMenu._Ready` is a stable patch target across Godot's source-generated method dispatch.** A Godot `_Ready` override has both a C# method AND a `MethodName._Ready` constant; Harmony patching the C# method should work, but the source-generator boilerplate could complicate it.
- **Risk: `[ModInitializer]` runs before Godot's main loop is fully initialized**, in which case `Engine.GetMainLoop()` returns null. Mitigation: catch and log; we'll see the failure in the smoke output.
- **Risk: a misconfigured manifest or build-pipeline glitch produces a mod that silently doesn't load.** Mitigation: the very first `Log.Info` in `Init` ("mod loading...") is the canary; if it doesn't appear, something is wrong before the smoke even runs.
