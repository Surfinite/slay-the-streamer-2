**One-line verdict:** The spec is well-structured and appropriately scoped for a validation smoke test, but it has a critical blind spot — Smoke B only validates fire-and-forget await, not the blocking await pattern that Plan B actually needs — and several smaller gaps that could cause avoidable friction during execution.

---

## What's good

1. **Clear purpose and tight scope.** Two smokes, two architectural questions, disposable scaffolding clearly separated from permanent. This is exactly the right level of ambition for a "prep before Plan B" validation step.

2. **Disposable vs permanent delineation is explicit.** The spec clearly marks what gets deleted after success, including the specific code blocks (steps 4+6 in `ModEntry.Init()`, the `SmokeChat` static field). This prevents smoke residue from leaking into production.

3. **Deterministic smoke design.** Hardcoded 3 options, single `#0` injection, expected winner=0 — no flakiness. The "expected 0 vs actual" diagnostic signal in the success matrix is a nice touch that gives more granular failure information than just "did it complete."

4. **Fallback autoload path.** The try/catch around `Engine.RegisterSingleton` with a `Root.AddChild`-only fallback is pragmatic. The spec correctly identifies that `Engine.RegisterSingleton` may not exist or may behave differently in Godot 4.5.1.

5. **Harmony.PatchAll in `[ModInitializer]`.** The spec correctly notes that the game only auto-runs `Harmony.PatchAll` if the assembly lacks `[ModInitializer]`, and includes the explicit `new Harmony(...).PatchAll(...)` call. This is a non-obvious interaction that would have been a painful runtime debugging session if missed.

6. **Success matrix is well-designed.** The 2x2 grid with specific diagnostic conclusions for each outcome gives the spec author a clear decision tree. The "wrong winner" signal (winner != 0) is a useful extra diagnostic dimension.

7. **Build pipeline extension is minimal and correct.** Adding `0Harmony.dll` refresh alongside `sts2.dll`, `<Private>false</Private>`, Release config, `dist/` assembly — all appropriate.

---

## Concerns & risks

### Critical

**1. Smoke B does not validate the realistic Plan B pattern.** This is the spec's most significant gap. Smoke B uses a `static void Prefix()` that fire-and-forgets the async runner via `_ = SmokeRunner.RunSmokeB(...)`. This validates that *fire-and-forget* await from a Harmony prefix doesn't deadlock. But Plan B's real pattern needs a Harmony prefix that **blocks** until the vote completes so it can substitute the player's input (e.g., "the vote picked card #2, so click that card instead of whatever the player would have chosen"). A blocking pattern would look like:

```csharp
static bool Prefix(ref int __result) {
    var winner = SmokeRunner.RunSmokeBBlocking(chat).GetAwaiter().GetResult();
    __result = winner;
    return false; // skip original
}
```

The deadlock risk is fundamentally different for blocking vs fire-and-forget. The spec acknowledges this in Section 10 question #3 but doesn't address it in the design. **As written, passing Smoke B gives false confidence about the pattern Plan B actually needs.**

### High

**2. `IMainThreadDispatcher.DrainAsync()` is not implemented.** The interface has two methods: `Post(Action)` and `Task DrainAsync()`. The spec's `GodotMainThreadDispatcher` snippet only shows `Post`. The context document (Section 7) explicitly flags this as a known gap. While nothing currently calls `DrainAsync()` from production code, shipping a permanent class (`GodotMainThreadDispatcher`) that doesn't fully implement its interface is a compile-time error — this isn't a design-time question, it's a build failure. The spec must address this before it can be executed.

**3. `FakeChatService.ConnectAsync` may not exist or may have a different signature.** The spec calls `SmokeChat.ConnectAsync("smoke", new ChatCredentials("smokebot", "abc")).GetAwaiter().GetResult()`. The context document doesn't confirm that `FakeChatService` has a `ConnectAsync` method — it describes `FakeChatService` as having `Inject(ChatMessage)` and `SimulateState(...)`. If `ConnectAsync` doesn't exist or `ChatCredentials` has a different constructor, this is a compile error. The spec author needs to validate this against the actual `FakeChatService` API.

### Medium

**4. `NMainMenu._Ready` patch target may not work as expected.** The spec patches `NMainMenu._Ready`, which is a Godot lifecycle method. Godot source generators produce `MethodName._Ready` constants, and the actual method that gets called may be wrapped by generated boilerplate. Harmony patching a C# `_Ready` override in a Godot Node subclass may fail silently if the source-generated dispatch doesn't go through the C# method directly. This is listed as an assumption in the risks section but has no mitigation beyond "we'll see if it works." Consider adding a diagnostic log line *inside the prefix itself* (before the fire-and-forget) so the streamer can distinguish "prefix never fired" from "prefix fired but smoke hung."

**5. `Engine.GetMainLoop()` may return null during `[ModInitializer]`.** The spec lists this as a risk but the catch block only catches exceptions from `RegisterSingleton`. If `Engine.GetMainLoop()` returns null, the `.Root` access will throw a `NullReferenceException` — but the catch block's fallback path also calls `Engine.GetMainLoop().Root.AddChild(autoload)`, which would throw the same `NullReferenceException` again. The fallback doesn't actually handle this case.

**6. `SmokeRunner.Run` uses `Log.Info` directly, not `TiLog`.** The spec says logging goes via `TiLog.Sink → MegaCrit.Sts2.Core.Logging.Log`, but the `SmokeRunner` snippet calls `Log.Info(...)` and `Log.Error(...)` directly. This is inconsistent — either `SmokeRunner` should use `TiLog` (to validate the passthrough), or the spec should explain why direct `Log` calls are intentional here.

### Low

**7. `SmokeMainMenuPatch` uses `Interlocked.Exchange` on an `int` field that's only ever read/written from the main thread.** `Interlocked` is unnecessary overhead for a main-thread-only guard. A simple `if (_fired) return; _fired = 1;` would suffice. Not a bug, but it's misleading — it suggests thread safety is needed when it isn't.

**8. The `MethodName.Run` constant in `DispatcherAutoload` may not exist.** Godot's source generator produces `MethodName` constants for methods that are explicitly defined as `[Signal]` callbacks or are otherwise registered. A private `Run(Callable)` method may not get a `MethodName.Run` constant generated. The spec should note this as an assumption to validate, or use a string literal `"Run"` instead.

## Suggested changes

1. **Add a Smoke C (or modify Smoke B) to validate the blocking await pattern.** The smallest addition: a second Harmony patch (or a second method on `SmokeMainMenuPatch`) that does `.GetAwaiter().GetResult()` on the vote task and logs the result. This directly validates the pattern Plan B needs. If it deadlocks, you learn that *before* writing 6+ Harmony patches that depend on it.

2. **Implement `DrainAsync()` on `GodotMainThreadDispatcher`.** The simplest correct implementation for a `CallDeferred`-based dispatcher is `public Task DrainAsync() => Task.CompletedTask;` with a comment documenting that "drain" semantics are ambiguous for a frame-deferred queue and that callers should not assume queued actions have completed when this returns. This makes the class compile and documents the semantic gap.

3. **Validate `FakeChatService.ConnectAsync` and `ChatCredentials` signatures against the actual code.** If `ConnectAsync` doesn't exist, the smoke can use `FakeChatService.SimulateState(...)` to set the connection state instead. The spec should not assume API surface it hasn't verified.

4. **Add a diagnostic log line inside `SmokeMainMenuPatch.Prefix` before the fire-and-forget.** Something like `Log.Info("[smoke-B] prefix fired; launching smoke");`. This lets the streamer distinguish "prefix never applied" from "prefix applied but smoke failed."

5. **Fix the null-`GetMainLoop()` fallback.** The catch block should handle the case where `GetMainLoop()` itself returns null:
   ```csharp
   catch (Exception e) {
       Log.Warn($"[slay_the_streamer_2] autoload registration failed: {e.Message}");
       var root = Engine.GetMainLoop()?.Root;
       if (root != null && autoload.GetParent() == null) {
           root.AddChild(autoload);
           Log.Info("[slay_the_streamer_2] autoload: Root.AddChild fallback");
       } else {
           Log.Error("[slay_the_streamer_2] autoload: cannot attach — no main loop root");
       }
   }
   ```

6. **Make `SmokeRunner` use `TiLog` instead of direct `Log` calls.** This validates the logging passthrough as part of the smoke, which is one of the things you're testing. If `TiLog.Sink` isn't wired correctly, the smoke log lines won't appear — which is useful signal.

7. **Add `// SMOKE-TEST: DELETE AFTER VALIDATION` headers to disposable files.** The spec mentions this in Section 10 question #5 but doesn't include it in the design. Add it explicitly to `SmokeRunner.cs` and `SmokeMainMenuPatch.cs`.

8. **Replace `MethodName.Run` with a string literal or validate the source generator produces it.** If the constant doesn't exist, the `CallDeferred` call will throw at runtime. Either use `CallDeferred("Run", Callable.From(action))` or verify the constant exists.

## Alternatives

1. **Skip the autoload Node entirely and use `Engine.Singleton` or a static event.** If the only purpose of `DispatcherAutoload` is to provide a `CallDeferred` target, you could instead use `Engine.GetMainLoop().CallDeferred(...)` directly from `GodotMainThreadDispatcher.Post`. This eliminates the entire autoload registration question (concern #6 from `notes/06`) in favor of a simpler pattern. However, this may not work if `GetMainLoop()` returns null at the time `Post` is called (vs at `Init` time).

2. **Use a single smoke that covers both concerns sequentially.** Instead of Smoke A (from `Init`) and Smoke B (from Harmony prefix), run the dispatcher validation first from `Init`, and if it succeeds, immediately apply a Harmony patch that runs the blocking-await smoke. This reduces the number of moving parts and ensures the Harmony patch is only tested after the dispatcher is confirmed working.

3. **Validate the autoload with a simpler test before the vote smoke.** Instead of combining autoload validation with a full vote smoke, first validate that the autoload Node receives frame ticks (e.g., log a counter every `_Process` call for 5 frames). Once confirmed, run the vote smoke. This gives cleaner failure isolation — you know the autoload works before you test votes on top of it.

## Additions

1. **Diagnostic log in `ModEntry.Init()` listing applied Harmony patches.** After `PatchAll`, log the number of patches applied and their target methods. This directly addresses Section 10 question #4 — if Smoke B is silent, the streamer can check whether the patch was applied at all.

2. **Timeout on the smoke vote.** If `AwaitWinnerAsync` never completes (deadlock), the smoke will hang forever with no diagnostic output. Add a `CancellationTokenSource` with a 10-second timeout so the smoke logs `"[smoke-X] TIMEOUT: AwaitWinnerAsync did not complete within 10s"` instead of hanging silently.

3. **Log the Godot version and `Engine.GetMainLoop()` type at startup.** This helps diagnose autoload issues: `Log.Info($"[slay_the_streamer_2] Godot {Engine.GetVersionInfo()}, main loop type: {Engine.GetMainLoop()?.GetType().Name}")`.

4. **Document the `SystemTimerScheduler` constructor signature.** The spec uses `new SystemTimerScheduler(clock)` — verify this matches the actual constructor. If `SystemTimerScheduler` takes additional parameters, the spec needs updating.

## Removals

1. **`Interlocked.Exchange` in `SmokeMainMenuPatch`.** Replace with a simple boolean flag. The `Interlocked` suggests cross-thread access that doesn't exist, and it's confusing to future readers.

2. **The `registeredSingleton` local variable in `ModEntry.Init()`.** It's assigned but never read. Either log its value or remove it.

## Minor / nits

1. The spec uses `SimulateMessage("alice", "#0")` in the flow diagram but `chat.Inject(new ChatMessage(...))` in the `SmokeRunner` code. These are different APIs — the flow diagram should match the code.

2. The `SmokeRunner.Run` method is `async Task` but the `try/catch` around `AwaitWinnerAsync` means the returned `Task` will always complete successfully (exceptions are caught and logged, not rethrown). This is fine for fire-and-forget, but the caller can't observe failure. Consider rethrowing after logging, or returning the winner index so callers can check.

3. The `Run(Callable callable)` method in `DispatcherAutoload` is private but called via `CallDeferred`. Godot's `CallDeferred` can invoke private methods, but this should be noted as an assumption.

4. The spec says `SmokeChat.ConnectAsync(...).GetAwaiter().GetResult()` — if `ConnectAsync` returns a `Task` (not `ValueTask`), this is fine. But if it returns `ValueTask`, calling `.GetAwaiter().GetResult()` on an already-completed task is still fine, just worth noting.

5. The `ChatCredentials` record is constructed inline with `("smokebot", "abc")` — verify the constructor parameter names and types match.

6. The spec's flow diagram for Smoke A says "Init returns; game continues startup" — this is correct because the smoke is fire-and-forget, but it's worth noting that `Init` is synchronous and must return quickly. The `ConnectAsync(...).GetAwaiter().GetResult()` call could block if the connection takes time, but since `FakeChatService` is in-memory, it should be instant.

## Assumptions you're making

1. **`FakeChatService` has a `ConnectAsync` method** that takes a string and `ChatCredentials`. The context document only confirms `Inject(ChatMessage)` and `SimulateState(...)`. I'm assuming `ConnectAsync` exists but haven't verified its signature.

2. **`ChatCredentials` has a constructor taking `(string, string)`.** The context document mentions `ChatCredentials` is a record but doesn't specify its constructor parameters.

3. **`SystemTimerScheduler` constructor takes `(IClock)`.** The spec uses `new SystemTimerScheduler(clock)`. I'm assuming this matches the actual constructor.

4. **`SystemClock` has a parameterless constructor.** The spec uses `new SystemClock()`.

5. **`MethodName.Run` is generated by Godot's source generator** for the private `Run(Callable)` method. I'm assuming the source generator picks up non-public methods, but this may not be the case.

6. **`NMainMenu._Ready` is a valid Harmony patch target.** I'm assuming Godot's source-generated dispatch for `_Ready` goes through the C# method directly, which may not be true.

7. **`Engine.RegisterSingleton` exists in Godot 4.5.1** and accepts a `(string, Node)` signature. The spec acknowledges this may not be true and provides a fallback.

8. **`MegaCrit.Sts2.Core.Logging.Log` has static `Info`, `Warn`, `Error` methods** with the signatures used in the spec (`Info(string)`, `Warn(string)`, `Error(string)` and `Error(string, Exception)`). The context document confirms thread safety but not exact method signatures.

9. **`Voter.Start` accepts `(string, string[], TimeSpan)`.** The spec uses this signature but the context document doesn't confirm it.

10. **`VoteCoordinator` constructor takes `(IChatService, IClock, ITimerScheduler, IMainThreadDispatcher)`** — the context document says it also takes an optional `Random?` parameter, which the spec omits. This is fine (the default is used), but worth noting.
