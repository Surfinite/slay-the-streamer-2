# Review — TI Layer Design Spec (`2026-05-08-ti-layer-design.md`)

---

## 1. One-line verdict

A disciplined, testable, well-scoped design with genuinely clean seams — but the threading model is under-specified at the edges where it matters most, the parser is one notch too strict, and the IRC implementation is missing a few Twitch-specific protocol realities that will bite at scale.

---

## 2. What’s good

**The `ChatService` / `VoteSession` boundary is genuinely clean.** `IChatService` carries no vote semantics whatsoever; `ChatMessage` is a pure data record; `VoteSession` consumes the interface without reaching into implementation details. A future mod that wants "chat picks an outfit every 5 minutes" really can build on `ChatService` alone without dragging in vote concepts. This is the hardest seam to get right and you got it right.

**The testability design is excellent.** `FakeChatService` + `FakeClock` + injectable `Random` gives deterministic coverage of every interesting path: vote changes, ties, no-voters, mid-session disconnect, validation edge cases. No `Thread.Sleep` anywhere in the unit tests. The "source-referenced, not DLL-referenced" test project is the right call for a greenfield mod that shouldn't need `InternalsVisibleTo` ceremony.

**The receipt strategy directly addresses the `#1` lag problem.** Open/periodic/close announcements are the right mechanism, 7 s default cadence is reasonable, the "no tally when all zeroes" guard prevents dead-air spam, and the close receipt explicitly announces ties and no-voter random picks rather than hiding them. This is UX thinking that many spec authors skip.

**"One vote open at a time" with loud failure (`Voter.Start` throws) is the right decision.** It surfaces misuse immediately rather than silently corrupting state. Matches StS's one-screen-at-a-time game flow perfectly. The `Dispose`-to-clear contract is clean.

**Scope discipline is admirable.** No `ChatCommandRouter`, no Twitch Extensions, no channel points, no cross-game abstraction, no premature base-mod split, no NuGet dependency sprawl. Everything in the "out of scope" and "non-goals" sections is correctly deferred. The spec defines exactly what needs to be built for v0.1 and nothing more.

**The extraction story is credible.** `Ti/Chat/` has zero vote dependencies, zero Godot dependencies, and only one StS2-specific dependency (`Log`). A future lift genuinely is a file move + a tiny logging shim. The spec doesn't oversell this — it acknowledges `Ti/Ui/` might stay behind, and that the lift is "not a refactor." Honest.

## 3. Concerns & risks (ranked by severity)

### 3.1 — Threading model has unaddressed failure modes (HIGH)

The spec says `GameThreadDispatcher` "queues actions and flushes them via Godot's `CallDeferred`." Three problems:

1. **`CallDeferred` ordering is FIFO per-call but not guaranteed across rapid successive calls.** If two IRC messages arrive close together and their dispatches get queued, Godot guarantees they execute in order on the main thread — but *only* if they land in the same frame's deferred queue. The spec doesn't specify whether the dispatcher batches or calls `CallDeferred` individually. If individual, a frame boundary between two queued actions could interleave with game code, which matters for `TallyChanged` → `Closed` ordering (a tally update and a close event firing in the wrong order would confuse `VoteOverlayControl`).

2. **Disposal during shutdown.** `ModEntry` constructs the service and fires `ConnectAsync` non-blocking. What happens if the game shuts down while the IRC read loop is mid-flight? The spec says "the OS cleans up the socket" but Godot finalizers on the Mono runtime can hang or crash on orphaned `Task`s. You need a `CancellationToken` propagated through the entire read loop and a `DisposeAsync`-style teardown that awaits the read loop's graceful exit.

3. **The "long-lived autoload node" for `CallDeferred` is non-trivial in a mod context.** Godot autoloads are configured in `project.godot`, which a mod doesn't control. You can register an autoload programmatically via `AddAutoloadSingleton` but that requires a `Godot.Object` (C# class extending `Node`), and the mod's assembly needs to be loaded before you can register it. The spec handwaves this. This isn't a design flaw per se, but it's an implementation detail that could force the threading model to get messier if the autoload approach doesn't pan out.

**Severity**: Threading bugs are the hardest to reproduce and fix. These gaps need resolution before implementation, not during.

### 3.2 — Twitch RECONNECT and NOTICE messages are unhandled (HIGH)

Twitch's IRC gateway sends `RECONNECT` notices when it wants the client to gracefully disconnect and reconnect (server maintenance, load balancing). Ignoring this means your connection gets severed unexpectedly and you fall into the retry loop — which "works" but causes a 5+ second gap from the exponential backoff floor, when a `RECONNECT`-triggered immediate reconnect would be near-seamless.

Twitch also sends `NOTICE * :Login authentication failed` and `NOTICE * :Error logging in` on bad credentials. The spec's infinite-retry loop on wrong oauth would spam connection attempts forever. The current design defers this to an "open item" — but it's worth specifying even a minimal bound (e.g., "don't retry auth failures, only transport failures").

### 3.3 — Message rate limiting is handled only by tally cadence, not by a send queue (MEDIUM)

The spec says "Safe under Twitch's 100 msg / 30 s rate limit" because 7 s cadence = ~4 messages/30s for tallies plus open/close = ~6 messages/30s. This is true for the *steady state*.

But a rapid open→close cycle (e.g., a 5-second vote that opens, sends one tally at 3.5 s, and closes) could send 3 messages in ~5 seconds. Multiple short votes in sequence (event→card reward→shop in under a minute) could pile up. And if the periodic tally fires at the exact same time as the close receipt, that's two rapid-fire messages.

Twitch's rate limit is per-30-seconds sliding window, and exceeding it gets you *global* message suppression (not just per-channel). A simple token-bucket rate limiter (~20 lines) would prevent this entire class of bug.

### 3.4 — Strict `#N` parser is too strict for real Twitch chat (MEDIUM)

The design rationale is sound: fewer false positives. But the "false negative" cost is higher than the spec assumes:

- New viewers naturally type `!1` (conditioned by other stream-bot commands using `!`).
- Mobile Twitch users often omit special characters entirely and type bare `1` or `1 `.
- During heated moments, people type `VOTE 1` or `I pick 1`.

Every one of these is a lost vote. In a small stream (10-20 active chatters), losing 3-4 votes to parser strictness is a significant fraction of the electorate. The spec should at minimum acknowledge this trade-off with data from the StS1 mod: what fraction of valid votes did robojumper's parser catch with `#N` vs `!N` vs bare `N`?

The spec flatly rejects `!N` as a "false positive" risk, but `!N` in the StS1 base mod is the *primary* accepted form (`#N` is secondary). This is a deliberate divergence from established convention that deserves stronger justification than "reduces false positives."

### 3.5 — User-identity dependence on IRC tags is fragile (MEDIUM)

`VoteSession` tracks voters by "Twitch user-id (from CAP tags)." The spec requests `twitch.tv/tags` capability, which does include `user-id`. But:

1. Not all IRC clients send tags (though Twitch's web chat and most mobile clients do). A viewer on a bare IRC client won't have a `user-id` tag and their vote will either be discarded silently (bad) or tracked by display-name (bad: impersonation trivial). The spec should specify the fallback behavior.

2. `twitch.tv/tags` also includes `display-name`, `color`, `badges`, `emotes`, etc. The IRC parser should explicitly extract `user-id` and `display-name`, and the spec should be clear that `ChatMessage.User` is the `user-id` tag value (not the display name, not the IRC nick).

### 3.6 — The parsing regex `#(\d+)` has an edge case with multi-digit numbers (LOW)

"More than 9 options" is rejected by `Voter.Start` validation. Good. But the regex `#(\d+)` would match `#10`, `#999`, etc. in chat. The spec says "out-of-range index → silently ignored," which handles this. However, someone typing `#12` when there are valid `#1` and `#2` options gets silently ignored rather than counted as `#1`. This is the correct behavior per spec but may surprise viewers. Worth a note in the spec that this is intentional.

## 4. Suggested changes (specific, actionable)

### 4.1 — Specify the `GameThreadDispatcher` contract precisely

Add to the spec:

```
GameThreadDispatcher guarantees:
- Actions enqueued in order A, B are executed in order A, B on the main thread.
- Actions are delivered no later than the next idle frame after enqueue.
- If the dispatcher is disposed, enqueued-but-unexecuted actions are silently dropped
  (logged at Warn level).
- The dispatcher exposes a `Task DrainAsync()` that blocks until the queue is empty;
  used during shutdown before the autoload node is freed.
```

This forces the implementation to batch into a single `CallDeferred` per frame (or use a `ConcurrentQueue` drained by a `_Process` override) rather than individual `CallDeferred` calls.

### 4.2 — Add explicit RECONNECT handling

In `TwitchIrcChatService`:

```
On receiving `RECONNECT` (Twitch command):
  1. Send QUIT gracefully.
  2. Close the socket.
  3. Reconnect immediately (no backoff delay).
  4. Re-join the channel.
  5. Do NOT re-emit ConnectionStateChanged — from the subscriber's perspective,
     the connection was uninterrupted.
```

On receiving `NOTICE * :Login authentication failed` or `NOTICE * :Error logging in`:
```
Do not retry. Set a terminal error state. Log at Error level.
Fire ConnectionStateChanged(false) once with no subsequent recovery.
```

### 4.3 — Add a minimal message-rate limiter

Add to `Ti/Internal/`:

```csharp
internal sealed class MessageRateLimiter {
    // Enforces max `count` messages per `window`, globally across all channels.
    // If the limit would be exceeded, the message is queued and sent when budget allows.
    // If the queue exceeds `maxQueue`, oldest queued messages are dropped (logged at Warn).
    Task EnqueueAsync(Func<Task> sendAction, CancellationToken ct);
}
```

`TwitchIrcChatService.SendMessageAsync` routes through this. Default config: 90 msg/30s (headroom below the 100 limit). This is ~15 lines with a `SemaphoreSlim` + `Channel<T>`.

### 4.4 — Hard-spec the tag mapping

In the `ChatMessage` record:
```csharp
public sealed record ChatMessage(
    string User,          // Twitch user-id from `user-id` tag. Null if tag absent
                          // (e.g., third-party IRC client). VoteSession skips these.
    string DisplayName,   // From `display-name` tag. Falls back to IRC nick if absent.
    string Text,          // The PRIVMSG body, after the ` :` delimiter.
    DateTime ReceivedAt   // UTC, set by TwitchIrcParser from `tmi-sent-ts` tag if present,
                          // else local clock.
);
```

The spec currently says `User` is a `string` with no nullability annotation. It should be `string?` or explicitly documented as nullable for non-tagged clients.

### 4.5 — Add a `#1` → `!1` config toggle (but default `#N` only)

Keep the strict `#N` parser as default, but add a `VoteCommandStyle` enum:

```csharp
public enum VoteCommandStyle {
    HashOnly,     // #1, #2 only (default, strict)
    HashOrBang,   // #1 or !1 accepted
    // Bare numbers deliberately NOT supported — too many false positives
}
```

Pass it through `Voter.Start`. This costs ~3 lines of regex alternation (`#(\d+)|!(\d+)`) and defuses the "too strict" criticism without opening the floodgates to bare-number false positives. The default stays strict; streamers who observe false-negative pain can toggle.

### 4.6 — Specify shutdown lifecycle

Add to the spec (in the Lifecycle section):

```
Shutdown sequence (ModEntry.Dispose or game-exit hook):
  1. Dispose any open VoteSession (triggers early close with receipt).
  2. Call ChatService.DisconnectAsync() with a 3-second timeout CancellationToken.
  3. Await GameThreadDispatcher.DrainAsync() (ensures all queued events are processed).
  4. Dispose GameThreadDispatcher (drops any remaining queued actions).
  5. Dispose the IRC socket and CancellationTokenSource.
```

The spec currently says "OS cleans up the socket" — that's not sufficient for a managed runtime.

## 5. Alternatives worth considering

### 5.1 — `ConcurrentQueue` + `_Process` poll instead of `CallDeferred`

Instead of an autoload node with `CallDeferred`, the `GameThreadDispatcher` could be a simple static class with a `ConcurrentQueue<Action>`. The `VoteOverlayControl` (or any Godot node that wants to receive events) calls `GameThreadDispatcher.Flush()` in its `_Process`. This avoids the autoload registration problem entirely and gives the consumer control over when events are delivered. Trade-off: if no UI node is present, events pile up and never get delivered. Mitigation: the `VoteSession` itself could own a hidden `Node` that does the flush.

I don't think this is strictly *better* than `CallDeferred`, but it's worth noting as an escape hatch if the autoload approach hits Godot-mod-integration friction.

### 5.2 — Use `TimeProvider` from .NET 8+ instead of custom `IClock`

.NET 8+ ships `System.TimeProvider` (abstract class) and `FakeTimeProvider` (from `Microsoft.Bcl.TimeProvider` NuGet, or you can write a trivial one). This removes the custom `IClock` / `SystemClock` / `FakeClock` abstraction and replaces it with a BCL type. Trade-off: adds one NuGet dep (`Microsoft.Bcl.TimeProvider`), but it's from Microsoft, tiny, and well-maintained. Your call whether the ~30 lines of custom clock code is worth keeping over a standard type.

### 5.3 — Cheaper tie-break announcement

Currently the spec says: `"Tied between #<a> <label_a> and #<b> <label_b> — chose #<n> <label_n> by random pick."` This is two lines of chat text. For 3+ way ties it grows. A shorter format: `"Tie! Chose #<n> <label_n> randomly among <k> tied options."` keeps it to one line and avoids bulk. Minor.

### 5.4 — Vote confirmation via reply-thread

Twitch supports `@reply-parent-msg-id` tags on PRIVMSG. The open receipt could stash its message ID, and each voter's `#N` could get a threaded reply: `"@viewer, counted #2."` This gives individual confirmation without the lag problem. Trade-off: doubles outbound message volume (~1 reply per unique voter per vote change). Probably not worth it for v0.1, but worth flagging as a future quality-of-life feature since it directly addresses the `#1` lag problem at the individual level.

## 6. Additions (things missing that should be there)

### 6.1 — Connection health metric

`IChatService` should expose:

```csharp
DateTime? LastMessageReceivedAt { get; }  // null if never connected
```

This lets the `VoteOverlayControl` (or game-side UI) show "Chat: connected (last msg 3 s ago)" vs "Chat: disconnected." Essential for the streamer to diagnose during a vote whether chat input is actually flowing. Costs one `DateTime` field updated per received message.

### 6.2 — Anonymous-mode `SendMessageAsync` behavior

The spec says `SendMessageAsync` fails (logs warning) in anonymous mode. But the receipt announcement system *needs* to send messages for votes to make sense. If the connection is anonymous, `Voter.Start` should probably throw or at least log an error — a vote with no receipts and no chat confirmation defeats the purpose. Add a validation check in `Voter.Start`:

```
if (!chat.IsConnected) → warn but allow (local-only vote).
if (chat is connected but in anonymous/read-only mode) → throw ArgumentException
  ("ChatService is in anonymous/read-only mode; cannot run votes with receipts.
   Provide ChatCredentials to enable sending.")
```

Or, alternatively, make anonymous mode skip receipts silently and let the vote run without announcements. But the spec should pick one explicitly.

### 6.3 — IRC test-fixture generator

Add to the testing strategy:

```
A small console tool in the test project that connects to Twitch IRC
(channel optional, e.g., a low-traffic channel), subscribes to all
incoming lines, and writes raw IRC lines to a JSON file. This file
becomes the test corpus for TwitchIrcParser tests. Run it once per
major Twitch IRC update to refresh the corpus.
```

The spec says "Inputs: real captured Twitch IRC lines" for parser tests, but those don't exist yet. A capture tool guarantees the test corpus matches reality. ~30 lines of code.

### 6.4 — `ToString` override on `VoteSession` for debugging

```csharp
public override string ToString() =>
    $"VoteSession[{Id}] '{Label}' {(IsClosed ? "CLOSED" : $"{TimeRemaining.TotalSeconds:F0}s")} " +
    $"[{string.Join(", ", Options.Select(o => o.Label))}]";
```

Trivial, invaluable when printf-debugging complex vote flows.

### 6.5 — Explicit `Dispose` safety on `VoteSession`

The spec says callers must `Dispose` the previous session before calling `Voter.Start` again. Add to `VoteSession.Dispose`:

```
If IsClosed is false, Dispose triggers an early close (same logic as
Duration expiration: compute winner, fire Closed, send close receipt,
set Voter.CurrentSession = null). Calling Dispose twice is a no-op.
```

This makes `Dispose` the universal "I'm done with this vote" gesture regardless of whether it closed naturally.

## 7. Removals (things that shouldn't be there)

### 7.1 — `VoteOption.Index` is redundant

```csharp
public sealed record VoteOption(int Index, string Label);
```

The `Index` is always the position in the `Options` list. It's a source of inconsistency (what if someone constructs `VoteOption(5, "hello")` when it's the first item?). The spec elsewhere uses 1-based indexes for `#N` commands. Keep the list positional, store options as `IReadOnlyList<string>`, and compute the index from position. The `VoteOption` record adds complexity with no benefit.

### 7.2 — `VoteReceiptPolicy` might be over-parameterised for v0.1

Three booleans/nulls: `AnnounceOnOpen`, `PeriodicTallyEvery`, `AnnounceOnClose`. Two static presets: `Default` and `Silent`. For v0.1, the only consumer will call `Voter.Start(...)` without passing a custom `VoteReceiptPolicy`. The full configurability is YAGNI. Keep the policy concept but reduce it to an enum for v0.1:

```csharp
public enum VoteReceiptMode { Full, Silent }
```

Add the fine-grained policy struct in v0.2 when the settings UI demands it. The spec already says "expose configuration when the broader settings UI is designed."

## 8. Minor / nits

- **`Ti` vs `TI` vs `TwitchIntegration`**: `Ti` reads like the chemical symbol for titanium. `TI` or `Twitch` would be more immediately readable. Not a hill to die on, but every reader will do a double-take.
- **"Chat picked" phrasing**: "The chat has chosen #1: Defend" or "Chat chose #1: Defend" reads more naturally than "Chat picked #1: Defend." "Picked" has a slightly juvenile/random connotation; "chose" or "voted for" better conveys the mechanism.
- **Section "Concurrency" in VoteSession**: could add a note that `Tallies` snapshot is safe for UI iteration on the main thread without locking (since it's a snapshot, not a live reference).
- **`TimeSpan Duration` comparison in validation**: `Duration < TimeSpan.FromSeconds(1)` catches zero and negative durations, which is good. But a 0.5-second vote duration, while technically allowed, would close before the first IRC message arrives. Consider a softer warning: "below 5 s is allowed but will likely produce no-voter results."
- **`ChatCredentials` record**: the `OauthToken` field is misspelled as `OauthToken` instead of `OAuthToken` or `OauthToken` — minor, but get the casing right in the actual code.
- **Spec mentions "9 or fewer options" but the regex is `#(\d+)` which matches `#0`**. Should `#0` be a valid vote? If options are 1-indexed, `#0` should be out-of-range and silently ignored. Worth an explicit note.
- **`AwaitWinnerAsync` with cancellation**: what happens if the `CancellationToken` fires before the vote closes? The spec doesn't say. Does it throw `OperationCanceledException`? Return a default? Document the behavior.

## 9. Assumptions I'm making (validate these)

1. **Godot autoload works from mod assemblies.** I'm assuming you can register a C# autoload node from a mod DLL at runtime. If Godot requires autoloads to be declared in `project.godot` (which a mod can't modify), the `CallDeferred` approach may need a fallback. Validate this before implementing `GameThreadDispatcher`.

2. **`MegaCrit.Sts2.Core.Logging.Log` is thread-safe.** The spec routes IRC-thread logs through it. If it's not thread-safe, you'll get corrupted log output or crashes from the background IRC task. Validate.

3. **StS2 exposes a shutdown/teardown hook for mods.** The spec says "if the game exposes a shutdown hook; otherwise the OS cleans up the socket." I'm assuming there's *some* mechanism (even if it's just the mod's assembly unload event or a `Dispose` pattern on registered models) that allows graceful teardown. If not, the IRC read loop will throw `ObjectDisposedException` on game exit and you'll get unclean shutdowns. Validate early.

4. **Twitch IRC CAP REQ tags still work identically to documented behavior.** Twitch has been stable on this for years, but I'm assuming no breaking changes between now and implementation. The tags spec (specifically `user-id`, `display-name`, `tmi-sent-ts`) is assumed available as documented.

5. **The 100 msg / 30 s rate limit applies to mod/broadcaster accounts.** Twitch documentation historically gives broadcasters and mods a higher limit (100/30s) vs regular users (20/30s). I'm assuming this is still accurate. Validate — if the limit is lower, the 7 s cadence may need adjustment.

6. **`FakeChatService.Inject` on a non-Godot thread is safe for tests.** Since `FakeChatService` doesn't use `GameThreadDispatcher` (it's in-memory), tests that inject messages synchronously and immediately assert on `Tallies` should work. But if `VoteSession` internally routes through the dispatcher even for fake services, you'll need to flush it in tests. The spec doesn't clarify whether `VoteSession` always dispatches or only dispatches when the `ChatService` indicates it's on a background thread. Validate this interaction.

7. **Harmony patches can call `await Voter.Start(...).AwaitWinnerAsync()` without deadlocking the game.** The Harmony prefix that intercepts a card-reward screen needs to *block* the game's original method until the vote resolves. If the Harmony patch runs on the Godot main thread and `AwaitWinnerAsync` also dispatches to the main thread, you have a deadlock. The spec's threading model implies this works because the IRC messages arrive already-dispatched to the main thread, so `AwaitWinnerAsync` completion also fires on the main thread — but a blocking `await` from a Harmony prefix on the main thread requires the continuation to run somewhere else. This is a known C# `SynchronizationContext` problem. The spec should address it explicitly in the "Game-side glue" section even if that section is out of scope for this design pass — at minimum, flag it as a dependency.

## Summary of must-fix vs. nice-to-have

| Must-fix before implementation | Nice-to-have |
|---|---|
| Threading model specification (ordering, disposal, shutdown) | `IClock` → `TimeProvider` |
| RECONNECT + auth-failure NOTICE handling | `VoteReceiptPolicy` → `VoteReceiptMode` enum |
| Message rate limiter | `VoteOption.Index` removal |
| Shutdown lifecycle | Vote confirmation via reply-threads |
| `ChatMessage.User` nullability for untagged clients | `#1` → `!1` toggle (or just add `!N` support by default) |
| Anonymous-mode receipt gap | `ToString` on `VoteSession` |
| Harmony deadlock risk (at minimum flag it) | IRC test-fixture generator |
