## 1. One-line verdict

Strong overall architecture and a good seam between IRC and voting, but the spec has several implementation-level contradictions and missing failure-mode definitions that will bite immediately unless tightened before coding.

---

## 2. What’s good

- **The two-tier split is the right core shape.** `IChatService` as chat IO and `VoteSession` / `Voter` as voting semantics is a clean and useful boundary. It avoids baking vote logic into IRC, and it gives future StS2 mods a plausible reuse point.

- **Rejecting a generic `ChatCommandRouter` for v0.1 is correct.** You have only one consumer: voting. A command router would add abstractions before you know the actual second use case.

- **One open vote at a time is a good invariant for StS.** It matches the UX and dramatically simplifies state, UI, receipts, and game integration. Preserve this.

- **Latest vote wins is the right interaction model.** It is forgiving, easy to explain, and matches the “chat can react to visible tallies” loop.

- **The receipt policy is valuable.** Open / periodic / close messages solve a real Twitch-lag problem. This is not just polish; it is a correctness/UX feature for viewers.

- **Fake chat service + fake time + seeded random is exactly the right testability strategy.** The spec is unusually test-oriented for an early design doc; keep that.

- **The optional UI is appropriately isolated.** `VoteOverlayControl` consuming `VoteSession` events rather than being owned by `VoteSession` is the right direction.

- **The handcrafted IRC client is defensible for this scope.** Given the “no TwitchLib” constraint, implementing the small subset is reasonable, provided the spec becomes much more explicit about edge cases.

---

## 3. Concerns & risks, ranked by severity

### 1. `ChatMessage` lacks the `UserId` that `VoteSession` depends on

The spec says tallies are backed by:

> `Dictionary<string, int>` keyed by Twitch user-id from CAP tags

But `ChatMessage` is currently:

```csharp
public sealed record ChatMessage(
    string User,
    string DisplayName,
    string Text,
    DateTime ReceivedAt);
```

There is no `UserId`. This is a direct mismatch between the lower-tier API and upper-tier voting requirements.

If you key by login name instead, name changes are a theoretical issue but probably acceptable inside one session. If you key by display name, that is bad: display names are not stable identifiers and may differ in casing/localization. If you require tags, expose the tag-derived ID.

**Action:** add `UserId` or define an explicit stable key field.

Suggested shape:

```csharp
public sealed record ChatMessage(
    string UserId,
    string Login,
    string DisplayName,
    string Text,
    DateTimeOffset ReceivedAt);
```

If anonymous/untagged messages can lack a user ID, define fallback behavior explicitly.

---

### 2. `Ti/Internal/GameThreadDispatcher` contradicts the allowed dependency rule

The spec says:

> `Ti/*` may reference: BCL, Godot in `Ti/Ui/*` only, `MegaCrit.Sts2.Core.Logging.Log`

But architecture puts this in `Ti/Internal`:

> `GameThreadDispatcher.cs` — marshal IRC → Godot main thread

and the threading section says it uses Godot `CallDeferred`.

That means `Ti/Internal` depends on Godot, violating the spec’s own dependency rule.

This matters because `Ti/Chat` and `Ti/Voting` depend on `GameThreadDispatcher`, so the supposed “UI-only Godot dependency” leaks into the whole TI layer.

**Action:** choose one of these:

1. Allow Godot in `Ti/Internal` and update the dependency rule.
2. Better: define a Godot-free dispatcher interface in `Ti/Internal`, and put the Godot implementation in the mod/bootstrap layer.

For example:

```csharp
public interface IMainThreadDispatcher
{
    void Post(Action action);
}
```

Then implementations:

- `GodotMainThreadDispatcher` in `Ti/Godot` or `Game/Infrastructure`
- `ImmediateDispatcher` for tests
- maybe `QueuedDispatcher` for smoke harnesses

That keeps `Ti/Chat` and `Ti/Voting` testable without Godot.

---

### 3. Fake time and `System.Threading.Timer` are inconsistent

The spec says `VoteSession` is tested with `FakeClock`, but also says:

> Internal timer (`System.Threading.Timer`) ticks

A fake clock does not control `System.Threading.Timer`. If `VoteSession` actually uses `System.Threading.Timer`, your tests will either use real time or require awkward sleeps.

**Action:** introduce a scheduling abstraction, not just a clock abstraction.

Options:

- Use .NET `TimeProvider` if available cleanly in the Godot/.NET 9 environment.
- Or define:

```csharp
public interface ITimerScheduler
{
    IDisposable Schedule(TimeSpan delay, Action callback);
    IDisposable SchedulePeriodic(TimeSpan interval, Action callback);
}
```

`FakeClock` can then own the scheduler and run due callbacks deterministically.

This is one of the most important implementation details to settle before coding tests.

---

### 4. `Voter` static singleton will hurt tests and future extraction

`Voter.CurrentSession` as a static convenience is fine for game glue, but it should not be the core owner of voting state.

Problems:

- Test isolation becomes fragile.
- Static state can survive failed tests.
- Future base-mod extraction may want one coordinator per channel/service.
- It is harder to inject `Random`, clock, dispatcher, logger, and receipt formatter.
- It encourages global access from Harmony patches, which can become painful.

**Action:** make an instance coordinator the real API, and keep `Voter` as a thin static facade if you want convenience.

For example:

```csharp
public sealed class VoteCoordinator
{
    public VoteSession? CurrentSession { get; }

    public VoteSession Start(
        IChatService chat,
        string label,
        IReadOnlyList<string> options,
        TimeSpan duration,
        VoteReceiptPolicy? receipts = null);
}
```

Then `Voter.Start(...)` can delegate to a singleton `VoteCoordinator` configured by `ModEntry`.

---

### 5. Close/dispose semantics are underspecified and potentially wrong

The spec says:

> When `Duration` elapses or `Dispose` is called early

then it computes a winner.

That means `Dispose()` is not merely cleanup; it is a semantic “close and pick winner” operation. That is surprising. In .NET, `Dispose()` usually releases resources and unsubscribes, not “make a gameplay decision.”

You likely need separate concepts:

- Natural close: duration elapsed, pick winner.
- Force close: caller wants current winner now.
- Cancel/abandon: caller wants to end session without choosing, e.g. screen disappeared, run ended, patch failed, streamer backed out.

**Action:** make the methods explicit.

Possible API:

```csharp
public Task<int> AwaitWinnerAsync(CancellationToken ct = default);
public int CloseNow();
public void Cancel();
public void Dispose();
```

Then define:

- `CloseNow()` computes/announces winner.
- `Cancel()` ends without winner and completes waiters as canceled.
- `Dispose()` is idempotent cleanup, probably equivalent to `Cancel()` unless already closed.

If you deliberately want `Dispose()` to close and pick, call that out loudly and explain why.

---

### 6. `AwaitWinnerAsync` cancellation semantics are missing

What happens if the caller passes a cancellation token and it fires?

Possibilities:

1. Only that waiter is canceled; vote continues.
2. The whole session is canceled.
3. The session closes immediately and picks a winner.
4. The token is ignored after registration.

The spec does not say. This matters because Harmony/game-side code may cancel on screen change, run end, or mod unload.

**Action:** define it explicitly.

Recommended:

- `AwaitWinnerAsync(ct)` cancellation cancels only the awaiting caller.
- Session lifetime is controlled by `CloseNow`, `Cancel`, duration expiry, or coordinator disposal.
- If the session itself is canceled, all pending waiters complete canceled.

---

### 7. Receipt ordering can race with `Closed` and new sessions

The close sequence says:

1. compute result
2. set winner/closed
3. fire `Closed`
4. send close receipt
5. set `Voter.CurrentSession = null`

This creates several issues:

- A `Closed` handler cannot start the next vote because `CurrentSession` is still occupied.
- If you clear `CurrentSession` before firing `Closed`, then a `Closed` handler may start a new vote and its open receipt could race ahead of the previous close receipt.
- If `SendMessageAsync` is awaited on the main thread, close handling can stall on network.
- If `SendMessageAsync` is fire-and-forget, message order is undefined unless `ChatService` serializes outgoing messages.

**Action:** specify a single outgoing message queue in `TwitchIrcChatService`, and define close ordering around that.

Recommended model:

- `SendMessageAsync` enqueues into a serialized outgoing queue.
- Queue enforces rate limits and preserves order.
- `VoteSession` enqueues close receipt before exposing the session as fully closed, or the coordinator enforces receipt order.
- Then clear `CurrentSession`.
- Then complete waiters/fire events.

Or, if gameplay responsiveness matters more than chat receipt order, explicitly accept that receipts may appear after the next open message. I would not accept that for v0.1; it will look broken.

---

### 8. No outgoing rate limiter is specified

The spec relies on the 7s cadence being below Twitch limits, but `IChatService.SendMessageAsync` is public and reusable. Future consumers or bugs can spam it.

Also, current Twitch chat limits include not just 20 messages / 30s for regular accounts and 100 / 30s for broadcaster/mod/VIP, but also a 1 message/sec/channel limit for regular accounts. Your default receipts are probably safe, but the service should still enforce a conservative send queue.

**Action:** add an outgoing queue/rate limiter to `TwitchIrcChatService`.

Minimum policy:

- Serialize all sends.
- Enforce at least 1 second between sent messages unless you have confidently detected broadcaster/mod/VIP status.
- Bound the queue.
- Drop or coalesce low-priority periodic tally messages when backed up.
- Never drop close receipts unless disconnected.

Suggested priority model:

- High: close receipt
- Medium: open receipt
- Low: periodic tally

Periodic tallies should be skippable if there is already a newer periodic tally waiting.

---

### 9. `IChatService` needs more state than `IsConnected`

`IsConnected` is too coarse.

You need to distinguish:

- Not connected
- Connecting
- Connected read-only
- Connected authenticated read/write
- Reconnecting
- Auth failed
- Banned / cannot join
- Disposed

Also, anonymous mode means `IsConnected == true` but `SendMessageAsync` cannot work. `VoteSession` default receipts will spam warnings unless it knows sending is unavailable.

**Action:** add at least:

```csharp
bool CanSend { get; }
ChatConnectionState State { get; }
Exception? LastError { get; }
```

Or use a richer state event:

```csharp
event EventHandler<ChatConnectionStateChangedEventArgs> ConnectionStateChanged;
```

Where the event includes old state, new state, reason, and offline duration if relevant.

---

### 10. Twitch protocol handling is under-specified

The spec covers PING/PONG and `PRIVMSG`, but Twitch IRC has several important cases that should be explicitly handled even in a minimal client:

- `RECONNECT`: Twitch asks client to reconnect.
- `NOTICE`: auth failure, invalid token, banned channel, duplicate login, etc.
- `CAP ACK` / `CAP NAK`: especially since tags require commands.
- `ROOMSTATE`: useful for detecting slow/sub/follower/emote-only modes.
- `USERSTATE`: confirms join/send state; useful for display name and sent-message behavior.
- `CLEARCHAT` / `CLEARMSG`: can be ignored, but say so.
- IRCv3 tag escaping.
- Message prefix parsing.
- CTCP ACTION `/me`.

**Action:** add an “IRC command handling matrix” to `TwitchIrcChatService`.

Example:

| IRC command | v0.1 behavior |
|---|---|
| `PING` | send `PONG` immediately |
| `PRIVMSG` | parse to `ChatMessage` |
| `RECONNECT` | disconnect and enter retry loop immediately |
| `NOTICE` | log; auth/join failures transition to failed state |
| `CAP ACK` | record capabilities |
| `CAP NAK` | log warning/error and continue/fail depending on missing capability |
| `ROOMSTATE` | optionally record room settings |
| `USERSTATE` | ignore except maybe bot display name |
| `CLEARCHAT`, `CLEARMSG`, `USERNOTICE` | ignore for v0.1 |

---

### 11. Anonymous read mode needs validation

The spec says:

> `creds == null` → connect with `NICK justinfan{rand6}`

Historically this works, but Twitch’s current public docs emphasize authenticated tokens for IRC/chat. If you want anonymous read mode, validate it manually and document it as “best effort / unsupported by official docs” if needed.

Also, if anonymous mode does not provide tags/user-id, your vote-keying strategy must fall back to login names.

**Action:** add an implementation note:

- Anonymous mode is best-effort.
- If `user-id` tag is absent, use lowercased login as `VoterKey`.
- If both are absent, ignore the message for voting and log at debug/warn once.

---

### 12. Command parser is “strict” but the regex is not

The spec says strict `#N` only, but the proposed regex:

```regex
#(\d+)
```

will match:

- `abc#1`
- `C#1`
- `#123456789`
- `#1foo`
- URLs/fragments containing `#1`

If the goal is fewer false positives, this regex is too loose.

**Action:** use a stricter boundary-aware pattern.

For 1–9 only:

```regex
(?<!\S)#([1-9])(?!\d)
```

This means:

- start of string or whitespace before `#`
- single digit 1–9
- not followed by another digit

You may want punctuation allowed after the digit:

```regex
(?<!\S)#([1-9])(?=$|\s|[.,!?;:])
```

Then explicitly test:

- `#1`
- ` #1 `
- `#1!`
- `C#1` ignored
- `abc#1` ignored
- `#10` ignored
- `#01` ignored
- `#1#2` probably ignored or first match, but define it

## 4. Suggested changes

### A. Update `ChatMessage`

Replace:

```csharp
public sealed record ChatMessage(
    string User,
    string DisplayName,
    string Text,
    DateTime ReceivedAt);
```

with something closer to:

```csharp
public sealed record ChatMessage(
    string VoterKey,
    string? UserId,
    string Login,
    string DisplayName,
    string Text,
    DateTimeOffset ReceivedAt);
```

Where:

- `UserId` is Twitch `user-id` tag if present.
- `Login` is lowercased Twitch login.
- `DisplayName` is cosmetic.
- `VoterKey` is the stable key used for voting: preferably `UserId`, fallback to `login:<login>`.

This avoids making every consumer duplicate fallback logic.

---

### B. Add `CanSend` and richer connection state to `IChatService`

Suggested interface:

```csharp
public interface IChatService : IDisposable
{
    ChatConnectionState State { get; }
    bool IsConnected { get; }
    bool CanSend { get; }

    event EventHandler<ChatMessage>? MessageReceived;
    event EventHandler<ChatConnectionChangedEventArgs>? ConnectionStateChanged;

    Task ConnectAsync(
        string channel,
        ChatCredentials? creds = null,
        CancellationToken ct = default);

    void Disconnect();

    Task SendMessageAsync(string text, CancellationToken ct = default);
}
```

Also consider `ValueTask` if you care, but `Task` is fine.

---

### C. Specify send behavior when disconnected

Right now `SendMessageAsync` behavior is only specified for anonymous mode.

Define all cases:

- Anonymous mode: reject/log/complete?
- Not connected yet: queue until connected or fail?
- Reconnecting: queue high-priority messages or drop?
- Disconnected permanently: fail?
- Disposed: throw `ObjectDisposedException`?

Recommended:

- `SendMessageAsync` returns a failed task if `CanSend == false`, except periodic receipts may be silently skipped by `VoteSession`.
- `VoteSession` should check `CanSend` before sending receipts to avoid warning spam.
- The service owns an outgoing queue only while connected/authenticated.

---

### D. Add a receipt formatter and sanitizer

Outgoing messages include labels from game content. Those labels may be long or contain punctuation/newlines/control chars.

Add a small internal formatter:

- Strip CR/LF.
- Collapse whitespace.
- Truncate to safe byte length.
- Ensure IRC line length is respected.
- Avoid accidental command injection.
- Maybe avoid sending `.`-empty messages.

This is especially important because IRC line length limits are byte-based, not character-based.

---

### E. Define option index base unambiguously

`VoteOption.Index`, `WinnerIndex`, and `Tallies` need one clear convention.

The spec currently implies 1-based indices because viewers type `#1`, but many C# APIs expect 0-based indices.

**Action:** explicitly state:

- `VoteOption.Index` is 1-based.
- `WinnerIndex` is 1-based.
- `Tallies` keys are 1-based.
- Callers receiving `WinnerIndex` must subtract 1 to index into the original `options` list.

Or change the API to expose both:

```csharp
public int WinnerOptionNumber { get; } // 1-based, chat-facing
public int WinnerOptionIndex { get; }  // 0-based, list-facing
```

I would strongly consider this because game-side code will eventually index lists.

---

### F. Fix close ordering

Suggested sequence:

1. Stop accepting votes.
2. Compute result.
3. Mark session closed.
4. Unsubscribe from chat and connection events.
5. Enqueue close receipt.
6. Clear coordinator `CurrentSession`.
7. Complete `AwaitWinnerAsync`.
8. Fire `Closed`.

This allows `Closed` handlers to start a new vote while preserving outgoing message order if `SendMessageAsync` is queued.

---

### G. Replace static `Voter` core with coordinator

Keep the static API if you like, but make it a facade.

Spec should say:

> `Voter` is a convenience wrapper over a process-wide `VoteCoordinator` configured by `ModEntry`; tests should normally instantiate `VoteCoordinator` directly.

This makes the future lift cleaner.

---

### H. Add an explicit protocol handling table

As noted above, this will make the handcrafted IRC client much less risky.

Especially include `RECONNECT`, `NOTICE`, `CAP ACK`, and `CAP NAK`.

---

### I. Add auth failure handling

Infinite retry is fine for transient network failures, but not for permanent auth failures.

Add logic:

- Invalid token / login auth failed → transition to `AuthFailed`; stop retrying until credentials change.
- Cannot join channel / banned → transition to `JoinFailed`; stop or use slower retry.
- Network failure → retry indefinitely with capped backoff.

This prevents “wrong OAuth token retries forever” from becoming the first support complaint.

---

### J. Add tests for message formatting and parser boundaries

Current testing list is good but incomplete.

Add tests for:

- `C#1` ignored
- `abc#1` ignored
- `#10` ignored
- `#01` ignored
- `#1!` accepted or rejected, whichever you choose
- duplicate same vote does not fire `TallyChanged`
- receipt truncation
- labels with newline/control chars
- send attempts in anonymous mode
- close receipt ordering relative to next vote
- `AwaitWinnerAsync` cancellation
- `Dispose`/cancel semantics
- `RECONNECT`
- `NOTICE` auth failure
- `CAP NAK`

---

## 5. Alternatives

### Alternative 1: Instance-first API, static convenience second

Instead of:

```csharp
Voter.Start(...)
```

as the real owner, use:

```csharp
var coordinator = new VoteCoordinator(...);
coordinator.Start(...);
```

Then expose `Voter.Start` only for game glue. This is my preferred alternative.

---

### Alternative 2: Keep `ChatService` thread-neutral

Rather than guaranteeing `IChatService.MessageReceived` is raised on the Godot main thread, make `IChatService` thread-neutral and let `VoteCoordinator` marshal.

Pros:

- Cleaner chat layer extraction.
- Easier console harness.
- Fewer hidden Godot dependencies.
- Tests become simpler.

Cons:

- Consumers of raw `ChatService` must care about threading unless wrapped.

Middle path:

- `TwitchIrcChatService` raises internally on its read task.
- `MainThreadChatServiceDecorator` wraps any `IChatService` and marshals events.
- `ModEntry` wires the decorated service into game code.

This preserves the “consumers never touch threads” goal without baking Godot into the IRC client.

---

### Alternative 3: Use `TimeProvider` instead of custom `IClock`

Since you are on .NET 9, `TimeProvider` may give you a standard abstraction for time and timers. If it works cleanly under Godot, prefer it over a bespoke `IClock`.

If Godot/.NET integration makes that awkward, your custom scheduler is fine.

---

### Alternative 4: Configurable parser, strict default

You fixed strict `#N` as non-negotiable, which is fine. But you can still design the parser as configurable without enabling alternatives in v0.1.

For example:

```csharp
public sealed record VoteCommandParserOptions(
    bool AcceptHashCommands = true,
    bool AcceptBangCommands = false,
    bool AcceptBareNumbers = false);
```

Default remains strict. Future settings UI can expose `!N` later without parser surgery.

---

### Alternative 5: `VoteSession` should not send receipts directly

Another architecture option:

- `VoteSession` only emits domain events.
- `VoteReceiptAnnouncer` subscribes to session events and sends chat messages.

Pros:

- Voting logic is pure and easier to test.
- Receipt policy is modular.
- Silent/local votes are natural.
- Future non-Twitch receipt channels are possible.

Cons:

- More moving pieces for v0.1.

Given your YAGNI stance, direct receipt sending from `VoteSession` is acceptable, but if it starts complicating tests, split it.

---

## 6. Additions

### Add a “State machine” section for `VoteSession`

Define states:

- Created/open
- Closing
- Closed with winner
- Canceled
- Disposed

And legal transitions. This will eliminate ambiguity around `Dispose`, cancellation, late votes, and event firing.

---

### Add a “ChatService state machine” section

Define:

- Disconnected
- Connecting
- ConnectedReadOnly
- ConnectedReadWrite
- Reconnecting
- AuthFailed
- JoinFailed
- Disposed

This will clarify `IsConnected`, `CanSend`, reconnect behavior, and UI status.

---

### Add offline-duration accounting details

Spec says:

> close receipt notes the gap if one occurred

Define exactly how that is calculated.

Questions:

- If vote starts while chat is disconnected, does the entire initial disconnected period count?
- If reconnect is attempted but auth fails, does that count?
- If anonymous read is connected but send unavailable, is that “offline”?
- If `ConnectionStateChanged(false)` fires multiple times during retry, how do you avoid double-counting?

Add tests for this.

---

### Add channel normalization

Define how `ConnectAsync(string channel, ...)` handles:

- `"foo"`
- `"#foo"`
- `"https://twitch.tv/foo"`
- uppercase names
- whitespace

At minimum, accept `foo` and `#foo`, normalize to lowercased login, and reject URLs unless you deliberately parse them.

---

### Add outgoing message priority/coalescing

Periodic tally messages are low value once stale. The service or receipt announcer should drop stale tallies if the queue is backed up.

This protects against weird rate-limit/backlog cases.

---

### Add manual validation checklist

The manual harness section is good. Add a checklist:

- connect with valid OAuth
- connect with bad OAuth
- connect anonymously
- join nonexistent channel
- join while banned/timed out if testable
- receive tags/user-id
- receive display names with non-ASCII chars
- send open/tally/close receipts
- disconnect network mid-vote
- receive Twitch `RECONNECT` if you can simulate via mock server

---

### Add minimal privacy/security handling for tokens

The settings/onboarding design is deferred, but the TI spec should still say:

- Never log OAuth token.
- Redact token from exceptions/log lines.
- `ChatCredentials.ToString()` should not expose token.
- Accept either raw token or `oauth:`-prefixed token, but normalize safely.
- Token requires `chat:read` and `chat:write` if sending receipts.

---

### Add source reference caution for tests

The spec says tests are “source-referenced” to avoid `InternalsVisibleTo`.

I would reconsider this. Source-referencing a Godot mod project can create a different compilation environment from the actual mod assembly. `InternalsVisibleTo` is not that painful and tests the built assembly more honestly.

If source-reference is kept, explicitly state why it will not diverge from production compilation.

---

## 7. Removals

### Remove or soften “handcrafted IRC is ~200 LOC”

This estimate may be misleading. A robust minimal Twitch client with TLS, reconnect, CAP handling, parser, send queue, rate limit, cancellation, and tests is probably not 200 LOC unless very compressed.

Say “small handcrafted client” instead of anchoring to a line count.

---

### Remove “Safe under Twitch’s 100 msg / 30 s rate limit” as the primary rate-limit claim

It is incomplete because regular accounts also have lower limits, including 20 messages / 30s and 1 message/sec/channel. Your cadence is still likely safe, but the spec should not imply 100/30 is the relevant default.

Replace with:

> Default receipts are below Twitch’s regular-account chat limits, but `TwitchIrcChatService` still serializes outgoing messages and applies a conservative rate limiter.

---

### Remove direct dependency on `MegaCrit.Sts2.Core.Logging.Log` from every TI class

Not necessarily from the whole layer, but do not sprinkle it everywhere. Use a tiny `ITiLogger` or static `TiLog` shim.

This makes later extraction and tests easier, even if the shim implementation calls MegaCrit logging.

---

### Remove “Dispose chooses a winner” unless you really mean it

As noted above, this is surprising and likely to cause bugs. Prefer explicit `CloseNow` / `Cancel`.

---

## 8. Minor / nits

- `DateTime ReceivedAt` should probably be `DateTimeOffset ReceivedAt`.

- `ChatCredentials(string Username, string OauthToken)` should specify whether `OauthToken` includes the `oauth:` prefix. Better: accept either but normalize internally and never double-prefix.

- `Username` should probably be `Login` and normalized to lowercase.

- `VoteReceiptPolicy.Default => new(true, TimeSpan.FromSeconds(7), true)` creates a new instance each time; fine, but static readonly would also be fine.

- `PeriodicTallyEvery = null` means disabled; good. Also validate it is positive if provided.

- Validate option labels: non-empty after trimming, no duplicates maybe not required, max display length.

- More than 9 options: good restriction. But if parser is single-digit, regex should enforce single-digit too.

- `TallyChanged` should not fire when a user repeats the same vote; the spec says deduped, good. Add a test.

- `Tallies` should probably include zero entries for every option, not only voted options. The examples imply `#1=0 #2=0`, etc. Specify this.

- `TimeRemaining` should be clamped at zero.

- `WinnerIndex` nullable is fine, but after cancel you need a separate state to distinguish “not closed yet” from “closed without winner.”

- `FakeChatService.Inject` synchronous delivery is useful, but if production dispatches events asynchronously through a dispatcher, tests may miss ordering bugs. Consider letting fake use the same dispatcher abstraction.

- If `SendMessageAsync` logs warning in anonymous mode and `VoteSession` sends open/periodic/close by default, anonymous-read votes will produce repeated warnings. Better to skip receipts when `CanSend == false`.

- “Read-only-ish IRC” is understandable in prose, but specs should avoid “ish” where possible. Say “IRC read plus outgoing chat messages.”

- `Ti` is okay, but `TwitchIntegration` is clearer. If you expect other contributors, prefer clarity over brevity. If the robojumper echo matters to you, `Ti` is acceptable.

- The close receipt examples for ties only show two tied options. Define formatting for 3+ tied options.

- The open receipt says `Type #1, #2, #3` but with many options and long labels maybe viewers need option labels in chat too. The overlay may show labels, but due stream lag, chat-only viewers might not know what `#2` is. Consider open receipt including compact labels when possible:
  - `Vote: Card reward — #1 Bash, #2 Defend, #3 Strike — 30s.`
  This is more useful than just listing numbers.

---

## 9. Assumptions I’m making

- I assume game-side Harmony patches can tolerate an async wait or can be structured to pause/hold the decision screen until `AwaitWinnerAsync` completes. If not, the TI API may need a callback-first shape.

- I assume Twitch IRC tags are available in the intended connection mode. If anonymous `justinfan` mode does not reliably provide `user-id`, you need the fallback key strategy.

- I assume `Godot.CallDeferred` is available only from Godot-bound objects/nodes, hence my concern about `Ti/Internal` depending on Godot.

- I assume the optional overlay can be implemented code-only without a `.pck`. If Godot requires packaged scene/resources for your chosen UI approach, `has_pck: false` may need revisiting.

- I assume current Twitch IRC sending via raw IRC `PRIVMSG` remains usable in your target environment, even though current Twitch docs increasingly point developers toward EventSub/Send Chat Message API. Validate this early with the manual harness.

- I assume v0.1 does not need moderator/subscriber vote filtering, but adding `UserId`, `Login`, and possibly raw tags now will make that future feature much easier.

- I assume the “one vote at a time” invariant is global per mod instance/channel, not per run/combat/screen. That is fine for StS, but should be explicit.

- I assume tests can run outside Godot. If they cannot, the dispatcher/time abstractions become even more important, not less.
