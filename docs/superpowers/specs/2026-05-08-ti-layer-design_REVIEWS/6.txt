Genuine review, not just praise — though there's a lot to praise.

**What's working well:** the two-tier seam is in the right place. `ChatService` is the natural extraction unit, and you've correctly identified that a `ChatCommandRouter` middle tier is YAGNI until a second consumer appears. The non-goals section is the most underrated part of the doc — explicitly excluding overlays, channel points, and the command router will save you weeks of scope creep. The thread model is unambiguous (background reads → dispatcher → main-thread events), the test matrix has the right shape (`FakeChatService` + `FakeClock` + seeded `Random` is exactly how this should be testable), and the receipt strategy honestly addresses the streamer-delay gap rather than pretending it doesn't exist.

**Three things worth pushing back on before you start coding:**

*Multiplayer breaks the `Voter` singleton.* StS2 has co-op multiplayer. If two co-op players each run Slay the Streamer with their own Twitch channels, you've got two `ChatService` instances and conceptually two concurrent voters. A static `Voter.CurrentSession` can't model that. The cheap fix now: make `Voter` an instance type, with each `ChatService` owning its own. `IChatService.StartVote(...)` rather than `Voter.Start(chat, ...)`. The "one vote at a time" rule then enforces per-streamer rather than globally, which is what you actually want anyway. v0.1 can still be single-streamer-only behaviourally, but the API stops painting you into a corner. This is a 30-minute refactor now and a 3-day refactor later.

*Subscriber/mod-only filters being deferred is risky.* You've correctly identified this as a v0.1 cut, but streamers will ask for it on day one — brigading and coordinated raids are real. Worth at least making the design admit it: have `IChatService.MessageReceived` deliver `ChatMessage` records that include badges/role tags from CAP, so a future filter is a where-clause, not a re-architecture. You're requesting `twitch.tv/tags` already so this is basically free — just make sure `ChatMessage` exposes `IsSubscriber`, `IsModerator`, `IsVip` from the start, even if the vote engine ignores them in v0.1.

*Handcrafted IRC client is a defensible choice but acknowledge what you're trading.* TwitchLib handles a long tail of edge cases (IRCv3 tag escaping rules, reconnect-on-RECONNECT command, rate limit awareness, NOTICE-based error reporting). Your ~200 LOC will cover the happy path; the question is what happens 6 months in when Twitch ships an IRCv3 quirk and TwitchLib gets a patch but you don't. Given your stated love for understanding systems and the small surface area you actually need, handcrafting is fine — but I'd hide it behind `IChatService` so well that you could swap to TwitchLib in a day if you change your mind. From the doc, it looks like you have. Good. Just make sure `TwitchIrcParser` tests include the genuinely weird stuff (escaped semicolons in tags, ACTION messages, multi-byte usernames, malformed CAP responses), not just clean cases.

**Smaller things worth a sentence in the doc:**

- The 9-option cap (`#1`–`#9`) is fine for most decisions but breaks on card-removal-from-deck and similar 10+ cases. Note this as a v0.1 limitation that will need a different parser/UI for "select from large set" votes later.
- Show the rate-limit math in the receipts section. 4 messages per vote (open + 2 tallies + close) on a 15s vote = ~16 messages/minute, well under 100/30s, but the doc just asserts "safe" without showing the work. Future-you will thank present-you for the explicit calc when the policy is configurable.
- Disconnect tolerance handles partial-vote outages well, but if the IRC connection is down for the *entire* vote duration, you silently fall through to the no-voter random pick. Probably the right behaviour, but worth either a config knob ("extend duration if disconnected") or at least an explicit note that this case exists.
- "Private test channel" in the manual harness — Twitch doesn't have private channels anymore. Use a throwaway account; same effect.

**One nit I'm flagging only because it'll bite you fast:** `TallyChanged` firing on every vote replace will be noisy. 50 viewers all changing votes in the same second = 50 events. The doc says receipts are deduped if vote == old vote, but a viewer flipping `#1` → `#2` → `#1` produces 3 events. Fine for the dispatcher, but make sure `VoteOverlayControl._Process` (which already runs once per frame) drives the redraw rather than reacting to every event individually. Otherwise a chat brigade will pin a CPU.

The doc is in great shape — these are tightening passes, not "stop and rethink" issues. Multiplayer is the only one I'd actually fix before writing code; the rest can be addressed during implementation when they bite.
