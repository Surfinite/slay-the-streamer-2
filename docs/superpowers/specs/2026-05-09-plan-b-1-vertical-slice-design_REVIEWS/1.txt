# Review: Plan B.1 Vertical Slice Spec

## 1. One-line verdict

This is a well-structured, appropriately scoped vertical slice spec that correctly inherits Plan A's architectural constraints and the smoke's deadlock verdict, but it has several concrete issues around the two-flag re-entry guard, the `VoteTallyLabel` lifecycle, the `ModEntry.Dispatcher` public static, and Twitch rate-limit accuracy that should be addressed before implementation begins.

## 3. Concerns & risks (ranked by severity)

### HIGH: Twitch rate-limit constants are wrong

The context doc says "Twitch's send limit is 90/30s" and the spec's `OutgoingMessageQueue` (already built in Plan A) presumably uses this number. But the actual Twitch rate limits are **20 messages/30s for regular accounts** and **100 messages/30s for mods/VIPs/broadcasters** (with verified bots getting 7500/30s). The 90/30s figure doesn't match any documented tier. If the bot account is a moderator in the streamer's channel (which is typical for bot accounts), the limit is 100/30s. If it's not a mod, it's 20/30s. The spec and the already-built `OutgoingMessageQueue` need to use the correct limit. This is high severity because an incorrect limit means either (a) the bot gets temp-banned for exceeding the real limit, or (b) the queue is overly conservative and receipts are delayed.

**Action**: Verify the token-bucket capacity and refill rate in `OutgoingMessageQueue` against the actual Twitch limits. If the bot will be a mod in the streamer's channel, use 100/30s. If not, use 20/30s. Document which tier the mod targets.

### HIGH: `VoteTallyLabel` double-free risk

The spec attaches the Label as a child of `NEventRoom` via `parent.AddChild(label)`, then subscribes `session.Closed` and `session.Cancelled` to call `label.QueueFree()`. If the `NEventRoom` is freed by the game (e.g., the streamer escapes the Neow screen, or the room transitions), Godot will free all children â€” including the Label. If the vote is still running, the `Closed` event fires later and calls `QueueFree()` on an already-freed node. This is a use-after-free in Godot that can crash or produce undefined behavior.

The spec acknowledges the streamer-escape sharp edge but doesn't address the double-free. Even without escape, there's a race: if the vote closes on the same frame the room is destroyed, both `QueueFree` and the room's child cleanup could fire.

**Action**: Either (a) don't parent the Label under `NEventRoom` â€” parent it under `GetTree().Root` or a persistent overlay node and position it absolutely, or (b) use `SetProcess(false)` + null-check the parent before calling `QueueFree`, or (c) use Godot's `Object.Dispose` pattern with a freed-check guard. Option (a) is cleanest.

### MEDIUM: `ModEntry.Dispatcher` as public mutable static

The spec promotes `dispatcher` from a local to `public static GodotMainThreadDispatcher Dispatcher` so the patch can read it. This is the simplest plumbing but it's a public mutable static field â€” any code can replace it at any time. For B.1 with one patch, the risk is low. But it sets a precedent: B.2 will have 4+ patches all reaching for this field, and if any test or cleanup code accidentally nulls it, the patches silently break.

**Action**: Make it a get-only property with a private setter, or better, expose it through `VoteCoordinator` (add a `public IMainThreadDispatcher Dispatcher { get; }` property). The patch already accesses `Voter.Default` â€” reading the dispatcher from the coordinator is one extra property access but eliminates the second public static. This is a minor change that pays dividends in B.2.

### MEDIUM: Two-flag re-entry guard â€” the "fast streamer click" window

The spec acknowledges the sharp edge: between the resume's `room.OptionButtonClicked(winnerOption, winnerIndex)` returning and `_voteInProgress` being reset to 0 (the very next line in the same `Post` action), a player click could land. But the spec says "both inside the same dispatcher.Post action" â€” this is correct for the reset, but the issue is that `OptionButtonClicked` calls `EventSynchronizer.ChooseLocalOption(index)` which eventually calls `TaskHelper.RunSafely(option.Chosen())`. The `Chosen()` method is async and runs fire-and-forget. So after the resume's `OptionButtonClicked` returns, the game is still applying the blessing's effects. A player click during this window would see `_voteInProgress == 0` (already reset) and `_resumeInProgress == 0` (already reset), so it would pass through to the original â€” triggering a *second* `OptionButtonClicked` while the first blessing's effects are still being applied.

The spec says this is "B.2-deferrable" but I think it's more serious than that. If the streamer clicks fast enough (and streamers do), they could trigger a second vote while the first blessing's effects are mid-application. This could result in two blessings being applied, or an exception in StS2.

**Action**: Move `_voteInProgress = 0` to *after* the blessing's effects have settled. Since `Chosen()` is fire-and-forget, you can't await it. Instead, keep `_voteInProgress = 1` for an additional ~1 second after the resume, then reset it via a delayed `dispatcher.Post`. Or, more simply, don't reset `_voteInProgress` inside the resume Post at all â€” reset it in the `HandleVoteAsync` catch/finally block *after* the Post has been enqueued, using a separate `dispatcher.Post` with a small delay. This ensures the resume's re-entry sees `_voteInProgress == 1` (correctly suppressed by `_resumeInProgress`) and any post-resume clicks during effect application are also suppressed.

### MEDIUM: `VoteTallyLabel` uses `_Process` polling instead of event-driven updates

The spec says `_Process` polls `session.TallyChanged` state each frame. But `VoteSession` already has a `TallyChanged` event. Polling in `_Process` works but means the label redraws every frame even when nothing changed (wasteful, minor) and â€” more importantly â€” the label won't update in response to `TallyChanged` events that fire between frames. In practice this is fine because `_Process` picks up the new tallies on the next frame, but it's architecturally inconsistent with the event-driven design of the TI core.

**Action**: Subscribe to `session.TallyChanged` and update the label text in the event handler. Keep `_Process` only for the `TimeRemaining` countdown (which genuinely needs per-frame updates). This is a minor code change that aligns with Plan A's event-driven patterns.

### LOW: `VoteTallyLabel` uses `Label` but blessing titles may contain BBCode

The spec notes this in the context doc's Key Question 4(c): "long blessing labels with BBCode markup (e.g., `[color]` tags from the game's localisation strings) might render strangely in plain Label." The spec says `Label` (or `RichTextLabel`) but doesn't commit. If `EventOption.Title.GetRawText()` returns BBCode (Godot's rich text format), then `Label` will render the raw BBCode tags as literal text, which looks broken.

**Action**: Use `RichTextLabel` and set `BbCodeEnabled = true`. Confirm at implementation time whether `GetRawText()` returns BBCode or plain text. This is a one-line change if caught early, a visual bug if not.

### LOW: `OutgoingMessageQueue` rate-limit doesn't account for the 1 msg/s per-channel minimum

Twitch enforces a per-channel minimum slow-mode of 1 second for non-moderator accounts. The spec's `OutgoingMessageQueue` uses a token-bucket at 90/30s (or whatever the correct rate is), but doesn't mention a per-channel 1-second gap between messages. If the queue sends two messages within 1 second, Twitch will silently drop the second one. For a single-channel bot this is unlikely with the volume described, but during reconnect bursts (sending the "connected" receipt + open receipt in quick succession), it could happen.

**Action**: Add a minimum inter-message gap of 1 second to the `OutgoingMessageQueue`, or at minimum document that the queue assumes messages are spaced >1s apart and the callers must ensure this.

## 4. Suggested changes

2. **Fix `VoteTallyLabel` double-free**: Parent the Label under a persistent node (e.g., `GetTree().Root` or a dedicated overlay), not under `NEventRoom`. If keeping it under `NEventRoom`, add a freed-check guard before `QueueFree`.

3. **Fix `_voteInProgress` reset timing**: Don't reset `_voteInProgress` in the same `Post` action as the resume. Reset it after a short delay (~1s) via a separate `dispatcher.Post`, or reset it in `HandleVoteAsync`'s finally block after the resume Post is enqueued.

## 5. Alternatives

**Alternative to suspend-and-resume: patch `EventSynchronizer.ChooseLocalOption` instead of `OptionButtonClicked`**

The context doc's Key Question 2 raises this. Patching at the `EventSynchronizer` layer would be more multiplayer-aware (the sync layer already handles `IsShared`) and more isolated from UI changes. The trade-off: `ChooseLocalOption` takes only an `int index`, not the `EventOption` object, so the patch would need to look up the option from the index. Also, `ChooseLocalOption` is deeper in the call stack â€” if it throws, the error surface is larger. For B.1, `OptionButtonClicked` is fine. For B.2, consider moving the patch target to `ChooseLocalOption` for the non-UI votes (card reward, boss relic, map path).

**Alternative to JSON credentials: environment variables**

Environment variables would avoid the plaintext-on-disk issue and are easy to set in a streamer's launch script. Downside: harder to document for non-technical streamers, and B.2's settings UI would need to read/write env vars instead of a JSON file. The JSON approach is better for B.2 compatibility.

**Alternative to `IIrcTransport` test seam: mock `TcpClient`/`SslStream` directly**

Instead of introducing a new `IIrcTransport` interface, mock `TcpClient` and `SslStream` using a `MemoryStream` or a custom `Stream` subclass. This avoids adding a new interface but requires more setup in each test. The `IIrcTransport` approach is cleaner and more maintainable â€” keep it.
