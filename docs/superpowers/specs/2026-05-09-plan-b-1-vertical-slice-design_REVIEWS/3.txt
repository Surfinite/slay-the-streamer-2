# Review — Plan B.1 vertical slice spec

## 1. One-line verdict

**Solid, well-bounded vertical slice that correctly inherits the smoke's hard-won lesson; main risks are around the resume re-entry guard's "fast click" race, public mutable static dispatcher plumbing, and the settings file path's Godot/.NET path-resolution mismatch — none of which should block B.1 but a few of which deserve cheap mitigations now rather than B.2 deferrals.**

## 2. What's good (keep as-is)

- **Decision #1–4 (vertical slice scope, Neow-first, `OptionButtonClicked` target, suspend-and-resume).** The reasoning chain is clean and the smoke evidence is correctly load-bearing. Choosing Neow specifically because it's single-shot, no-skip, no-stacking is the right call — it strips every confounding variable from the architectural validation.
- **Two-flag re-entry guard with explicit single-flag-bug callout in §"Decisions" #4.** Catching this in self-review and documenting *why* the simpler version is wrong is exactly the kind of artifact future-you (and B.2's patch authors) will thank present-you for.
- **Failure-mode table in §"Failure modes & graceful degradation".** Concrete, ranked, with explicit graceful-degradation contracts. The "mod stays loaded; game keeps running; votes silently no-op" promise is the right product stance.
- **Sequence diagram in `NeowBlessingVotePatch`.** Worth its weight in gold for B.2's patch authors. Keep this format for every subsequent patch spec.
- **Explicit "Why this never deadlocks" property statement.** Re-articulates the smoke's lesson at the exact site where future drift could re-introduce it. Good.
- **§"Open items" calling out `SystemTimerScheduler()` no-arg ctor.** Tiny but exactly the kind of footgun that wastes hours. Keep.
- **Decision #8's verification-against-Tempus's-source paragraph.** Replaces vibes-based scope with evidence-based scope. The "card reward came from base mod, not the original mod itself" finding is non-obvious and worth preserving.

## 3. Concerns & risks (ranked)

### High

**H1. `ModEntry.Dispatcher` as a public mutable static field** (§"`ModEntry` extensions"). Promoting the dispatcher from a local to `public static` is the simplest plumbing but it's a footgun precedent: B.2 will have 4 more patches all reaching into `ModEntry.Dispatcher`, and the moment something accidentally writes to it (test? second `Init` on hot-reload?) the whole mod silently breaks. **Cheap fix**: expose the dispatcher off `VoteCoordinator` (`coordinator.Dispatcher` get-only property) and have the patch read `Voter.Default!.Dispatcher`. This also matches the existing dependency direction (the coordinator already owns the dispatcher). Failing that, at minimum make it `internal static GodotMainThreadDispatcher? Dispatcher { get; private set; }`. The spec's §10 Q6 already flags this — answer is "yes, it sets a bad precedent; fix it now."

**H2. The "fast streamer click during async `option.Chosen()` settling" race is downplayed** (§"Risks & assumptions" bullet 2). The spec's mitigation is "if observed, keep `_voteInProgress = 1` until the resume's `option.Chosen()` task completes — but that requires awaiting a task we don't currently capture." That's not actually hard: the original `OptionButtonClicked` is `void`, but inside the resume's dispatcher.Post action we control everything. We can't await Chosen() from a void Harmony prefix on the main thread (would deadlock again), but we *can* either (a) disable input on the room (`room.Layout.DisableEventOptions()` per the spec's own suggestion in §"Risks & assumptions" bullet 4 — already mentioned, just not promoted to mitigation) immediately after the resume's `OptionButtonClicked` call, or (b) keep `_voteInProgress = 1` and reset it from a follow-up `dispatcher.Post` queued after a small frame delay. **Recommendation**: pick (a) — disable input on the resume — and validate operator-side. This eliminates a class of bugs B.2 would otherwise hit on every patch.

**H3. Settings file path will likely diverge between `Environment.SpecialFolder.ApplicationData` and Godot's `user://` resolution.** Spec's `GetSettingsPath` (§"`ModSettings`") proposes resolving via .NET's special folders. Godot 4 resolves `user://` through `OS.GetUserDataDir()` which on Windows is `%APPDATA%\Godot\app_userdata\<project_name>` — *but* the project name comes from `project.godot`'s `application/config/name`, not from a hardcoded "Slay the Spire 2" string. If StS2's project.godot ever shifts (or a beta branch differs), the .NET-computed path will silently miss. **Cheap fix**: ModEntry.Init has Godot context; resolve the path via Godot (`OS.GetUserDataDir()` or `ProjectSettings.GlobalizePath("user://slay_the_streamer_2.json")`) and pass the resolved path *into* `ModSettings.Load(string path)`. Keeps `ModSettings` BCL-only/testable, gets the right path in production. Spec currently has `ModSettings` resolving the path itself, which forces it to know about Godot's quirks.

### Medium

**M1. JSON schema has no version field** (§"`ModSettings`" JSON shape). B.2 will add settings-UI fields (vote duration policy? receipt verbosity? theme?) and probably want to read v0.1's file. Adding `"schemaVersion": 1` now is free; retrofitting it later means handling "no-version means v1" forever. Add it.

**M2. The "connected" PRIVMSG to chat is unconditional** (§"Decisions" #7, §"`ModEntry` extensions"). Two issues: (a) some streamers don't want unsolicited bot chatter (especially if their chat overlay is busy); (b) on a flapping connection (auth-temp-fail-then-success, or network blip), `OldState is not ConnectedReadWrite` will trigger a connect message every successful (re)connect. **Recommendations**: include the mod version (`"slay-the-streamer-2 v0.1.0-alpha connected — votes will go to #foo"`) so streamers know what's running, and only post once per *process lifetime* (track a `_connectAnnounced` bool), not once per ConnectedReadWrite transition. Reconnect messages are noise.

**M3. `EventOption.Title.GetRawText()` may include BBCode/markup** (§"`NeowBlessingVotePatch`" `HandleVoteAsync`). StS2's `LocString` likely contains color/keyword markup for in-game rendering (e.g., `[color=...]Curse[/color]`). Sending that raw to chat will surface as literal markup; rendering in `Label` will show as plain literal too. **Two action items**: (1) verify what `GetRawText()` actually returns for a Neow option (decompile-search task — already partially flagged in §"Open items" #4 but not fleshed out); (2) consider `RichTextLabel` with `BbcodeEnabled = true` for `VoteTallyLabel` regardless, since it's a near-free upgrade and matches the game's likely formatting.

**M4. `IsNeowEvent` reflection isn't cached** (§"`NeowBlessingVotePatch`" — implementation deferred). The patch fires on every `OptionButtonClicked` invocation across every event type; doing `AccessTools.Field(typeof(NEventRoom), "_event")` per call is wasteful but more importantly fragile (silent NRE if the field name changes). **Cheap fix**: cache the `FieldInfo` in a static `Lazy<FieldInfo>` and log-and-skip-patching at `Prepare` time if it doesn't resolve. This also gives an early-exit signal during version drift instead of "votes silently never fire."

**M5. `VoteCoordinator.Start`-can-throw is a sharp edge with no real handling.** Spec §"Failure modes" row 7 says "player click is lost; flag reset; player re-clicks; second click works. Bad UX but not a crash." But the prefix has *already* returned `false`, suppressing the original — so the player's first click did nothing visible (no button highlight, no animation, no sound). UX is worse than "did nothing": the streamer assumes their click registered. **Cheap fix**: wrap `Voter.Default.Start` in try/catch *inside* the prefix (before the `return false`), and if it throws, log + reset `_voteInProgress` + `return true` to let the original run (so the streamer's click applies normally). This makes "Voter is too sad to vote" degrade to vanilla, matching the rest of the failure-mode story.

**M6. Operator-validation steps don't exercise an empty-vote (no chat input) path.** If chat sends zero `#N` messages during the 30s window, what happens? `AwaitWinnerAsync` presumably returns *something* — Plan A's tiebreak/random/no-vote behavior. The spec's fallback (`winnerIndex < 0 || winnerIndex >= options.Count` → use `playerClickIndex`) suggests "no votes returns -1 or out-of-range," but if Plan A returns `0` (default first option) on no votes, the streamer's click is *silently ignored* without operator-validation noticing. **Add to step 2**: explicitly run a Neow vote without any chat input and document what the spec expects to happen.

### Low

**L1. `VoteTallyLabel.QueueFree` on Closed/Cancelled may double-free** (§"`VoteTallyLabel`"). If the label is a child of `NEventRoom` and the room frees first (e.g., scene transition), Godot frees the label automatically. Subsequent `Closed`/`Cancelled` handlers calling `QueueFree()` on a freed Node are mostly fine (Godot is forgiving), but the C# wrapper still holds a reference and `_Process` may run on a dead Node. **Cheap fix**: guard with `GodotObject.IsInstanceValid(this)` before any Node ops, and unsubscribe from `Closed`/`Cancelled` in `_ExitTree`.

**L2. `_Process` polling once per frame is fine but mixes time + tally update** (spec §10 Q4). For B.1 it's right; flagging only because B.2's `VoteOverlayControl` should subscribe to `TallyChanged` for tally and use `_Process` only for the timer. Worth a one-line comment in `VoteTallyLabel` saying "polling here is intentional for B.1; subscribe-to-events when polish lands."

**L3. No commit/version embedded anywhere observable to the streamer.** Mod log says "Init complete" but no version. Connect message has no version (M2 already mentions). Streamers reporting bugs will have no idea what they're running. **Cheap fix**: log the assembly's `InformationalVersion` in ModEntry and include it in the connect receipt.

**L4. The "connect to throwaway test channel before in-game test" step (§"Step 1") is missing setup detail.** What's "Surfinite's testbot oauth"? Where does someone reproduce this? Either link to a private notes file or document the bot account creation steps inline.

## 4. Suggested changes (specific & actionable)

1. **§"`ModEntry` extensions"**: replace `public static GodotMainThreadDispatcher Dispatcher` with `coordinator.Dispatcher` (add a get-only property to `VoteCoordinator` that returns its already-held `IMainThreadDispatcher`). Update `NeowBlessingVotePatch.HandleVoteAsync` to read `Voter.Default!.Dispatcher`. (H1)
2. **§"`NeowBlessingVotePatch`" sequence diagram + code**: add `room.Layout.DisableEventOptions()` (or equivalent) immediately *after* the resume's `OptionButtonClicked` re-call, before resetting `_voteInProgress`. Keep this as the production mitigation rather than the operator-validation contingency. (H2)
3. **§"`ModSettings`"**: change signature to `SettingsResult Load(string path)`; have `ModEntry` resolve the path via `OS.GetUserDataDir()` (Godot side) and pass it in. Keep `ModSettings` testable and Godot-free. (H3)
4. **§"`ModSettings`" JSON shape**: add `"schemaVersion": 1` field; have `Load` reject unknown future versions with `Malformed`. (M1)
5. **§"Decisions" #7 + §"`ModEntry` extensions"**: gate the connect PRIVMSG on a `_connectAnnounced` static bool; include mod version in the message. (M2, L3)
6. **§"`NeowBlessingVotePatch`"**: add a static `Lazy<FieldInfo?>` for the `_event` field; make `Prepare` fail loudly (return false + Error log) if it doesn't resolve. (M4)
7. **§"`NeowBlessingVotePatch`" Prefix**: wrap `Voter.Default.Start(...)` in try/catch; on throw, log Error, reset `_voteInProgress`, return `true` to let original run. (M5)
8. **§"Testing strategy" Step 2**: add explicit "no-chat-input" sub-step with documented expected behavior. (M6)
9. **§"`VoteTallyLabel`"**: add `IsInstanceValid(this)` guard in `_Process`; unsubscribe from `Closed`/`Cancelled` in `_ExitTree`; add comment that polling-only is intentional for B.1. (L1, L2)
10. **§"Open items" #4**: upgrade from "verify what call returns the right text" to a hard-blocker pre-implementation task — what `Title.GetRawText()` returns directly affects both chat and `VoteTallyLabel` rendering. (M3)

## 5. Alternatives worth considering

- **Patch at `EventSynchronizer.ChooseLocalOption(int index)` instead of `NEventRoom.OptionButtonClicked(option, index)`** (spec §10 Q2). Pros: one layer below UI, so it survives UI refactors; multiplayer-aware layer (cleaner for v0.2+ co-op design); eliminates the `IsNeowEvent` reflection (you'd filter on `RunManager.Instance.EventSynchronizer.CurrentEvent` shape). Cons: less obvious "where the click happens"; harder to access the option list at this layer; harder to disable input from this layer. **Verdict**: probably not worth changing for B.1, but worth a 30-min decompile-confirmation pass before locking it in. If `ChooseLocalOption` has a clean way to read the options list, this is the better long-term seam.

- **Vote on the click *before* committing to suspend-and-resume**: i.e., the prefix could quickly check "is chat available + would a vote be useful here?" and if not, return `true` immediately without setting flags. The spec already does this (chat-not-connected → `return true`), but consider also short-circuiting if `options.Count == 1` (no real choice to make). Cheap and avoids a pointless 30s wait on degenerate Neow loadouts (rare but possible).

- **Embed a tiny in-mod IRC test server for the unit test seam.** Spec proposes `IIrcTransport` interface for unit tests, leaving real TLS as operator-only. An alternative is a ~150 LOC `TcpListener`-based fake IRC server in tests/ that speaks just enough of the protocol to drive lifecycle tests. This is a one-time cost that would also unblock B.2+ from "TwitchIrcChatService bug? guess we'll see in operator." **Verdict**: probably overkill for B.1 alone, but cheap insurance given B.2 stacks four more patches on top. Worth a 1-hour spike to estimate cost before deciding.

## 6. Additions

- **`ModSettings.SettingsResult.Success`** should also surface a list of *warnings* (e.g., "channel name had `https://twitch.tv/` prefix; normalised to `foo`"). Helps streamers debug typos.
- **Operator-validation step 0**: build + install + launch with no settings file, *no Twitch involvement at all*, and reach Neow. Confirm the mod is fully invisible in the vanilla game. This is the "did we accidentally break the game with the patch attached?" baseline. (Currently buried inside Step 3's "no settings" sub-case but worth promoting to a smoke-baseline step.)
- **`notes/06-followups-and-deferred.md` should be opened *during* B.1 implementation, not after.** Use it as the running log of "things I want to fix in B.2 but skipped in B.1" so the meta-review for B.2 has them ready.
- **Explicit test for `IsNeowEvent` returning `false` on non-Neow events.** Easy to do via a fake or by patching `_event` to a non-Neow `EventModel` subclass; otherwise the filter is operator-only-validated and silent breakage on a future event type is invisible.
- **Add a `--smoke-only` ModEntry flag (or env var) that skips Harmony patching entirely.** Lets you re-run the smoke setup (Plan B prep) without Plan B.1's patches active for diagnosing whether a regression is patch-side or scaffolding-side. Cheap and useful for B.2's debugging.

## 7. Removals

- **§"Open items" #7 (`SystemTimerScheduler` constructor signature)**: doesn't belong in the spec — it's an implementation footgun, not a design decision. Move to `notes/06-followups-and-deferred.md` or a dev-onboarding doc.
- **§"Open items" #2 (`EventModel.CurrentOptions` accessibility)**: already verified per the spec text ("Confirmed from NEventRoom.cs:200: public. No reflection needed."). It's not an open item, it's a closed one. Delete or move to a "verified facts" subsection.
- **The `RichTextLabel` parenthetical in §"`VoteTallyLabel`"** ("// or RichTextLabel for richer formatting"): pick one before commit. Leaving it as a code-comment ambiguity guarantees the implementer flips a coin. (My suggestion: pick `RichTextLabel`; see M3.)

## 8. Minor / nits

- §"Architecture" tree uses 🆕/✅/✏️ emoji legend without a key. Add a one-line legend.
- §"Decisions" #2 says "verified from `Neow.GenerateInitialOptions` returning 1 curse + 2 positive" — worth adding the exact decompiled file path for B.2 reuse.
- §"`NeowBlessingVotePatch`" Prefix's filter ordering puts `IsLocked || IsProceed` *after* `_resumeInProgress` but *before* `Voter.Default is null`. The order is fine but the comment "// Filter: don't intercept locked / proceed options (they bypass the vote layer)" is slightly misleading — they don't *bypass* the vote, they're not eligible for one. Reword.
- §"Failure modes" table row "Mid-vote disconnect" assumes Plan A's `EnglishReceipts` already supports a "(chat was offline 8s during voting)" annotation. Worth confirming this is in v2.3 and not a retroactive ask.
- §"What 'B.1 done' unlocks": the bullet "TI core is in production" is slightly aggressive — `TwitchIrcChatService` is new code that's only operator-validated against TLS, not battle-tested. Soften to "TI core is functionally complete; production-tested by B.2's wider patch surface."
- The numeric LOC estimates in §"Architecture" (~1,000 source / ~230 tests) feel low for `TwitchIrcChatService` at 600 LOC + 150 tests when you factor in the IRC protocol matrix (PING/PONG, RECONNECT, NOTICE branches, CAP ACK/NAK, USERSTATE, ROOMSTATE, CLEARCHAT/CLEARMSG, USERNOTICE, jitter/backoff state, dispose ordering). I'd budget ~800 LOC source / ~250 LOC tests for the IRC service alone, ~1,300 LOC source / ~330 LOC tests total. Doesn't change the design — just helps estimate B.1 calendar.
- "0-indexed vote options" is restated several times across spec + context; one canonical statement is enough.

## 9. Assumptions I'm making (please validate against codebase)

1. **`VoteCoordinator` already holds an `IMainThreadDispatcher` reference** that could be exposed via a get-only property. Plan A's contract description in the context doc shows the constructor takes one, so this should be true — but the spec author should confirm exposing it doesn't violate any encapsulation invariant Plan A established.
2. **`AwaitWinnerAsync` returns -1 (or throws / returns out-of-range) when no chat votes are received.** I'm inferring from the spec's fallback code (`if (winnerIndex < 0 || winnerIndex >= options.Count)`). If Plan A actually picks index 0 by default on no votes, M6's concern is materially worse and the spec's fallback is dead code.
3. **`EventOption.Title` is a `LocString` with potential BBCode/color markup.** I'm inferring from "Plan A receipts are English-only via `EnglishReceipts`" + general Godot/StS2 conventions for in-game text. If `Title.GetRawText()` already returns clean plain text, M3 is moot.
4. **`room.Layout.DisableEventOptions()` (or equivalent) exists.** The spec itself proposes this as a fallback in §"Risks & assumptions" bullet 4 — I'm assuming the API actually exists at the time the spec was written. If not, H2's mitigation needs a different mechanism (e.g., reflective `_event.CurrentOptions[i].IsLocked = true` on every option).
5. **Godot's `OS.GetUserDataDir()` and `ProjectSettings.GlobalizePath("user://...")` are accessible from a `[ModInitializer]` static method context** (i.e., before main scene is fully ready). If not, H3's fix needs to be deferred to first idle-frame.
6. **`TaskHelper.RunSafely(option.Chosen())` is fire-and-forget without an awaitable handle exposed back to `OptionButtonClicked`'s caller.** The context doc says so; I'm trusting the decompile read. If there's any way to reach the underlying Task from the patch, H2's mitigation could be sharpened to "await Chosen() on a continuation thread, then reset flags."
7. **The current `ModEntry.cs` skeleton has the diagnostic patch-count log already wired** as the spec's "existing patch-count diagnostic logging stays as-is" implies. If not, the spec under-states the ModEntry changes.
8. **Plan A v2.3's `EnglishReceipts` includes the disconnect-gap annotation** ("...chat was offline 8s..."). Spec mentions it without citation; if it was deferred past v2.3, the failure-mode contract overstates current capability.

---

Overall this is a well-thought-out spec; the suspend-and-resume pattern is correctly load-bearing, and the scope discipline (one patch, JSON config, plain Label) is exactly right for a vertical slice. The fixes I'd push hardest on are H1 (kill the public mutable static), H2 (promote input-disable from contingency to mitigation), and H3 (resolve the settings path Godot-side). Everything else is polish.
