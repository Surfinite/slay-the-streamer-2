## One-line verdict

v2 is a major improvement and is close to implementation-ready, but I would make one more targeted revision before planning: fix the token validation/scopes wording, the root-parented label lifecycle claim, the post-`Start` failure/resume path, the rate limiter’s 1-msg/sec Twitch constraint, and a likely `VoteCoordinator` constructor-order bug.

---

## What got much better

- **The `Voter.Default.Start` synchronous-in-prefix change is right.** This eliminates the worst v1 lost-click path.
- **The stricter `ConnectedReadWrite` gate is correct.** Avoids read-only/anonymous “silent vote” UX.
- **JOIN timeout is a good addition.** This prevents “forever Connecting” on quiet JOIN failure.
- **Resume-time validity checks are the right direction.** Re-reading current options before applying the winner is safer than using captured `EventOption` objects.
- **The connect-once receipt avoids reconnect spam.**
- **The Twitch rate-limit correction to 20/30s is directionally right.**
- **The operator-validation matrix is much stronger.** Step 0 vanilla baseline is especially valuable.
- **Removing `ModEntry.Dispatcher` as a public mutable static is a good cleanup.**

---

## Must-fix concerns

### 1. OAuth token regex is too strict; make it warning-only or remove it

This line is risky:

> Validated against `^[a-z0-9]{30}$`

I would not hard-fail on exact length/charset. Twitch examples often show 30 lowercase alphanumeric tokens, but Twitch’s own authentication docs also show access-token examples with uppercase characters, and docs do not promise a stable exact regex shape. Hard-failing here can reject valid tokens before Twitch gets to authenticate them.

Suggested change:

- Strip optional `oauth:`.
- Require non-empty and maybe “no whitespace / no control chars”.
- If it does not match the common pattern, return `Success` with a warning, not `Malformed`.
- Let Twitch IRC auth decide validity.

Example spec wording:

> `oauthToken`: accepts either `oauth:<token>` or bare `<token>`. `Load` rejects empty/whitespace/control-character tokens. If the token does not match the common observed lowercase-alphanumeric shape, `Load` returns `Success` with a warning rather than `Malformed`; Twitch authentication remains the source of truth.

This also makes the setup less brittle when users paste tokens from newer auth flows.

---

### 2. Scope wording is probably wrong/inconsistent for IRC

v2 says required scopes are:

- `chat:read`
- `chat:write`

Current Twitch docs are annoyingly inconsistent across IRC-vs-new-chat pages:

- IRC scope docs list `chat:read` and `chat:edit` for IRC.
- Newer chatbot/EventSub/API docs list `user:read:chat` and `user:write:chat`.
- Some Twitch IRC/auth pages mention `chat:write`.

For B.1, do not hard-document one send scope as canonical unless you have tested the actual token generator you recommend.

Suggested README/spec wording:

> For Twitch IRC, the token must be a user access token for the bot account. Use Twitch’s current IRC/chat scopes for reading and sending chat messages. Depending on the generator/API path, these may be shown as `chat:read` + `chat:edit`, or as the newer chat scopes `user:read:chat` + `user:write:chat`. Operator validation against a real test channel is the acceptance source of truth.

If you want a simple B.1 path, say:

> The tested B.1 setup uses `<specific generator>` with scopes `<exact names Surfinite tested>`.

Then don’t generalize beyond that.

---

### 3. The rate limiter still misses Twitch’s 1-message/sec-per-channel limit

v2 corrects the global/default bucket to `20 messages / 30 seconds`, but Twitch also documents:

> 1 message per second per channel for non-broadcaster/mod/VIP accounts.

A token bucket with `capacity: 20, window: 30s` can still burst multiple messages instantly. That can violate the per-channel 1/sec limit even if the 20/30s total is respected.

For B.1, simplest fix:

- Add an explicit minimum send spacing of about 1.1–1.5s for the conservative unprivileged mode, or
- Configure the queue as effectively non-bursting for B.1: capacity 1, refill every 1.5s, or
- Add dual limits: `20/30s` plus `1/sec/channel`.

Because B.1 sends low-volume receipts, non-bursting is fine.

Suggested spec change:

> B.1 conservative send policy enforces both Twitch limits for ordinary accounts: no more than 20 messages per 30s and no more than 1 message per second to the joined channel. Implementation may use a dual limiter or a non-bursting 1.5s minimum spacing queue.

Also add a unit test:

- queueing connected receipt + vote-open receipt back-to-back does not write both within the same second.

### 4. `VoteCoordinator` constructor call in `ModEntry` looks wrong

Context document says constructor shape is:

```csharp
VoteCoordinator(
    IChatService chat,
    IMainThreadDispatcher dispatcher,
    ITimerScheduler scheduler,
    IClock clock,
    ...
)
```

v2 pseudocode uses:

```csharp
var coordinator = new VoteCoordinator(chat, clock, scheduler, dispatcher);
```

That appears reversed and would either fail compilation or silently bind wrong if overloads exist.

Fix to:

```csharp
var coordinator = new VoteCoordinator(
    chat,
    dispatcher,
    scheduler,
    clock);
```

If the actual codebase constructor differs, update the context/spec so reviewers are not seeing a false mismatch.

---

### 5. Root-parenting `VoteTallyLabel` contradicts the escape-cancellation claim

v2 says:

> Parented under `GetTree().Root`, NOT under `NEventRoom`.

Good for avoiding double-free.

But then it says:

> `_ExitTree` cancels the session if still open — prevents `HandleVoteAsync` from trying to resume into a stale scene.

That does **not** follow. If the label is parented under root, it will usually **not** exit the tree when `NEventRoom` is freed or the room changes. It will remain alive until:

- the session closes and calls `QueueFree`, or
- the whole scene tree/root shuts down, or
- something explicitly frees the label.

So streamer escape is not cancelled by `_ExitTree`; instead the vote continues until timeout, then resume validity checks drop it. That may be acceptable for B.1, but the spec currently claims stronger behavior than the design provides.

Fix one of these ways:

#### Option A — keep root-parenting, correct the claim

Say:

> Root-parenting means the label is not automatically freed on room transition. If the streamer escapes mid-vote, the vote may continue until its normal close; resume-time validity checks then drop safely. `_ExitTree` only cleans up event subscriptions if the label itself is freed.

This is easiest.

#### Option B — add explicit room-lifetime cancellation in `Game/*`

Keep `Ti/Ui` room-agnostic, but in the patch pass a cancellation token or register a room-validity watchdog from `Game/DecisionVotes`.

For example:

```csharp
private static async Task WatchRoomValidityAsync(
    NEventRoom room,
    VoteSession session,
    IMainThreadDispatcher dispatcher,
    ITimerScheduler scheduler,
    IClock clock)
```

Every frame or short timer tick:

- dispatch to main thread,
- check `GodotObject.IsInstanceValid(room)` and `IsNeowEvent(room)`,
- if invalid, cancel session.

This is more work and probably unnecessary for B.1.

Recommendation: choose Option A for B.1. The current safe-drop behavior is enough, but document it accurately.

### 6. Post-`Start` exceptions can still lose the click and leave buttons disabled

v2 fixes `Voter.Default.Start` throwing. Good.

But after `Start` succeeds, the code disables event options and then `HandleVoteAsync` can still hit its outer catch:

```csharp
catch (Exception ex) {
    TiLog.Error("[neow-vote] HandleVoteAsync threw", ex);
    try { Voter.Default?.Dispatcher.Post(() => Interlocked.Exchange(ref _voteInProgress, 0)); }
    catch { Interlocked.Exchange(ref _voteInProgress, 0); }
}
```

At that point:

- prefix already returned `false`,
- original click was skipped,
- options may be disabled,
- no fallback resume occurs.

So v2’s “no lost player click in every failure path” is not quite true.

Maybe you think this path is practically unreachable, but the spec should not rely on that. The easy fix is to factor a safe fallback resume and use it in the outer catch.

Suggested shape:

```csharp
catch (Exception ex) {
    TiLog.Error("[neow-vote] HandleVoteAsync threw; falling back to player click", ex);
    SafePostResumeFallback(room, playerClickIndex);
}
```

And inside the posted fallback:

- set `_resumeInProgress = 1`,
- verify room/event/current options,
- if `playerClickIndex` valid, call `OptionButtonClicked(currentOptions[playerClickIndex], playerClickIndex)`,
- otherwise drop with Warn,
- always reset both flags.

That gives the spec’s “no lost click” claim actual coverage beyond `Start`.

---

### 7. `ResumeOnMainThread` should use `playerClickIndex` fallback instead of dropping on changed options

The method takes `playerClickIndex` but currently does not use it.

Current drop path:

```csharp
if (currentOptions is null || winnerIndex >= currentOptions.Count) {
    TiLog.Warn(...);
    return;
}
```

If the winner index is invalid but the player’s original click index is valid, the better fallback is to apply the player’s original click.

Suggested logic:

```csharp
var applyIndex = winnerIndex;
if (currentOptions is null) {
    drop;
}

if (applyIndex < 0 || applyIndex >= currentOptions.Count) {
    TiLog.Warn(...);
    applyIndex = playerClickIndex;
}

if (applyIndex < 0 || applyIndex >= currentOptions.Count) {
    TiLog.Warn("[neow-vote] resume: neither winner nor player index valid; dropping");
    return;
}

room.OptionButtonClicked(currentOptions[applyIndex], applyIndex);
```

This makes the fallback behavior consistent.

### 8. `DisableEventOptions` may create an unrecoverable disabled UI if resume drops

This is a direct consequence of the above.

If you call `DisableEventOptions()` and later decide to drop resume because the room/event/options are invalid, that is fine if the room is gone. But if the room is still visible and options are merely changed or the resume path fails, the streamer may be left with disabled buttons.

Mitigations:

- Prefer fallback to `playerClickIndex` whenever the room is still valid.
- If dropping while the room is still valid, attempt to re-enable/rebuild options if the API exists.
- If no re-enable API exists, document that `DisableEventOptions` is acceptable only because every valid same-room failure path resumes/fallbacks rather than drops.

Spec should say:

> Resume is only dropped when the room/event is no longer valid. If the room is still valid but the winning index is unusable, fallback to `playerClickIndex`. This avoids leaving disabled Neow buttons on-screen.

## Should-fix concerns

### 9. Capture `VoteCoordinator` / dispatcher instead of repeatedly reading `Voter.Default`

v2 reads `Voter.Default` in async continuations:

```csharp
Voter.Default!.Dispatcher.Post(...)
...
Voter.Default.Dispatcher.Post(...)
```

For B.1 no reload path probably exists, but it is cleaner and safer to capture the coordinator in the prefix:

```csharp
var coordinator = Voter.Default;
if (coordinator is null) return true;
...
session = coordinator.Start(...);
_ = HandleVoteAsync(__instance, coordinator.Dispatcher, session, optionsSnapshot, index);
```

Then `HandleVoteAsync` uses the captured dispatcher. This avoids weirdness if B.2 adds shutdown/reload and sets `Voter.Default = null`.

---

### 10. `RichTextLabel` with `BbcodeEnabled = true` needs label sanitization policy

v2 says use:

```csharp
var labels = optionsSnapshot.Select(o => o.Title.GetFormattedText()).ToList();
```

and:

```csharp
label.BbcodeEnabled = true;
```

This may be right if StS2’s formatted text is Godot BBCode-compatible. But you need two label forms:

1. **UI label** — formatted/BBCode-safe for `RichTextLabel`.
2. **Chat receipt label** — plain text for Twitch chat.

If `VoteCoordinator.Start(...)` receives the formatted label, English receipts may send `[color]...[/color]` tags to Twitch chat. That is ugly.

Suggested design:

```csharp
var voteOptions = optionsSnapshot
    .Select(o => new {
        Plain = o.Title.GetRawText(),
        Formatted = o.Title.GetFormattedText(),
    })
    .ToList();
```

But Plan A’s `VoteSession` probably stores only one label. For B.1, choose plain labels for the session/receipts and let the UI display plain labels too. RichTextLabel can still render plain text.

Recommendation for B.1:

- Use `GetRawText()` for `VoteCoordinator.Start`.
- Use `RichTextLabel` with BBCode disabled unless you explicitly sanitize/verify formatted text.
- Treat formatted labels as B.2 polish.

If you keep `BbcodeEnabled = true`, add:

- BBCode escaping for plain text, or
- explicit decompile/operator verification that `GetFormattedText()` returns valid BBCode and does not make receipts ugly.

---

### 11. Root-level UI may need a `CanvasLayer` or overlay container

A `Control` under `SceneTree.Root` may work, but root is typically a `Window`/`Viewport`, not the same as being inside the game’s UI container. Anchors likely work relative to the viewport, but z-order and scaling may surprise you.

B.1 can discover this operator-side, but I would add a fallback:

- Try `tree.Root.AddChild(label)`.
- If it appears behind the game or does not anchor correctly, create a `CanvasLayer` named `SlayTheStreamerOverlayLayer` under root and add the label there.

Better B.1 spec:

> `VoteTallyLabel.AttachTo` attaches to a root-level overlay container. Implementation may create/reuse a `CanvasLayer` under `SceneTree.Root` to guarantee visibility above room UI.

This is still Godot-only `Ti/Ui`, no StS2 dependency.

---

### 12. JOIN timeout should probably not be terminal for all no-confirm cases

v2 says:

> On timeout without confirmation, transition to `JoinFailed`.

That is good for nonexistent channels, but if Twitch is slow or there is transient network weirdness, terminal `JoinFailed` may be too final. Since B.1’s promise is “network failure retries forever,” consider distinguishing:

- socket/read failure → retry,
- explicit banned/suspended/auth failure → terminal,
- JOIN no-confirm timeout → maybe terminal for now, but document why.

For B.1 it is acceptable as terminal, but I would soften the wording:

> Treat JOIN confirmation timeout as `JoinFailed` for v0.1 because the streamer can fix channel config and restart. If real-world false positives occur, B.2 can reclassify as retryable.

---

### 13. CAP NAK fallback weakens self-echo guard

v2 says:

> CAP NAK fallback path (no tags, falls back to login-by-prefix only).

Good. But self-echo guard currently says:

> drops `parsed.UserId == self.UserId`

If tags are unavailable, `UserId` may be unavailable. Add explicit behavior:

- With tags: compare `user-id`.
- Without tags: compare login/nick case-insensitively if available.
- If neither is available, do not attempt self-echo suppression.

Add a test for self-echo in no-tags mode by login.

---

### 14. Scope creep: anonymous operator validation may be optional

v2 adds anonymous mode operator validation. That path is explicitly not exercised by B.1 and the patch now bails out for read-only. Unit-test coverage is enough unless it’s cheap.

I’d demote anonymous live validation from acceptance gate to “nice if easy.” Otherwise you can block B.1 on a path not used by the vertical slice.

## Minor / nits

- `Prepare` signature check should use `typeof(EventOption)` rather than `ParameterType.Name == "EventOption"` if the patch assembly references the type anyway.

- Multiplayer warn log should be throttled. If a co-op streamer clicks repeatedly, Warn spam is possible.

- `schemaVersion` missing → `Malformed` is okay but slightly unfriendly. For B.1 that is acceptable; settings UI will handle later.

- `SettingsResult.Success(... Warnings)` is good, but include username lowercasing as a warning if you normalize it.

- `TwitchIrcChatService` constructor now takes `scheduler`; ensure it is not the same scheduler used by `VoteCoordinator` in ways that create surprising timer coupling. Shared scheduler is probably fine if stateless/system, but tests with `FakeTimerScheduler` may prefer separate instances.

- “close receipt notes disconnect gap” is okay if verified, but operator-validation should not fail solely because exact wording differs.

- `VoteTallyLabel._Process`: clamp negative seconds to 0 so it does not briefly display `-1s left`.

- `RichTextLabel.FitContent = true` plus anchors may produce odd sizing. Operator validation will catch; not a design blocker.

---

## Suggested minimal v3 patch list

If you want the smallest possible v2.1/v3 revision before implementation, I’d do only these:

1. **Remove hard `^[a-z0-9]{30}$` OAuth rejection.** Make it non-empty/no-whitespace only, optional warning for unusual shape.
2. **Fix OAuth scope docs wording.** Mention IRC legacy/new scope names or document the exact tested generator/scopes.
3. **Enforce 1-message/sec send spacing** in addition to 20/30s.
4. **Fix `VoteCoordinator` constructor call order** in `ModEntry` pseudocode.
5. **Correct `VoteTallyLabel` root-parent lifecycle wording.** `_ExitTree` does not detect `NEventRoom` escape when root-parented.
6. **Add fallback resume in `HandleVoteAsync` outer catch.**
7. **Use `playerClickIndex` fallback in `ResumeOnMainThread`.**
8. **Clarify label text policy:** raw/plain labels for `VoteSession` receipts; formatted/BBCode only if separately sanitized for UI.

After those, I’d call the spec implementation-ready.
