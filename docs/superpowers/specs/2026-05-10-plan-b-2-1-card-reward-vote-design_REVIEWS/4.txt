One-line verdict
The spec is thorough and coherent but contains avoidable duplication, unresolved run-id details, and several omissions that should be tightened before merging to main.

What’s good (keep as-is)
- Vote-on-click model, multi-message linear receipts, and two-flag re-entry guard (B.1 pattern) are solid — reuse exactly.
- Selector logic (card list extraction from NCardRewardSelectionScreen._options and run/act indexing) is well described and follows the code map.

- Separate per-act and per-run skip budgets are sensible and aligned with intended UX.

Concerns & risks (ranked by severity)
1) Run-ID naming/type is underspecified (blocker for resume correctness).
   - Spec says “verify in Prepare” but doesn’t specify the field name, type, or where to read it at runtime. If wrong, run-id guard silently fails and the guard becomes ineffective.
   - Action: identify exact RuntimeState property or field that gives a stable unique run ID (likely `RunManager.Instance.DebugOnlyGetState()?.Id` or similar). Verify shape in `CardRewardVotePatch.Prepare` and store in a static `_runIdType` field; advertise type/accessor in spec.

2) RewardSkippedFrom(Control) matching is fragile.
   - Reflection on `_rewardButtons` must include exact field name and element type; if it changed in the game version targeted, the entire skip gate fails open.
   - Action: add a `Prepare` that stores `FieldInfo _rewardButtons` and a method `static IEnumerable<Control> GetRewardButtons(NRewardsScreen)`. If shapes mismatch, log and skip gate open (fail-open). Add unit tests using a fake screen mock to verify the reflection result.

3) Act change detection timing is not deterministic.
   - Spec notes act-index discovery may be “lazy (one frame off)”. That means skip budget reset may lag one rewards screen. If streamer changes act and immediately attempts a skip, the budget may appear stale.
   - Action: define explicit rule (e.g., on _Ready, resolve current act index once and cache). Document whether index is `Acts.Count - 1` or a dedicated property. Add tests asserting reset behavior matches chosen rule.

4) No explicit nullability discipline for static state.
   - Static `_activeLabel` and `_lastSeenRunId` should be null when no screen active. If reused across runs without reset, stale values can mis-gate.
   - Action: add `Requires` assertion or explicit null checks in every consumer (e.g., `UpdateText` should early-return if label is null). Add tests that simulate run change → label recreated.

5) Receipt format relies on EnglishReceipts contract.
   - `EnglishReceipts` may not accept a 4-parameter skip receipt signature. If receipt format mismatches, the mod either throws or silently omits receipts.
   - Action: add a `SkipReceipt(int actUsed, int actLimit, int runUsed, int runLimit)` helper to `EnglishReceipts` (or a local format string in the patch). Verify via reflection that receipts render legible strings.
