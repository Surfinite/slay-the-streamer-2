## Review of Plan B.2.1 ‚Äî Card Reward Vote

---

### 1. One-line verdict

The spec is a solid vertical slice that correctly reuses B.1's production-validated pattern and makes a genuinely insightful design call in chat-vs-streamer asymmetry, but it has several interaction gaps between the vote patch and skip gate patch, a likely TI/Game seam violation in receipt formatting, an underspecified thread-safety model for skip counters, and a gameplay-pacing concern that deserves explicit acknowledgment even if it's deferred.

---

### 2. What's good

- **Chat-vs-streamer asymmetry framing (Author's note #1).** This is the single strongest design insight in the spec. "Skip is never a chat-vote option" is not obvious ‚Äî it's easy to imagine a naive implementation where skip is option #3 ‚Äî and the reasoning is clear and well-documented. Preserve this framing prominently; it will generalize to B.2.2 boss relic and beyond.

- **Leveraging vanilla `DisallowSkipping()` (Decision 3).** Smaller patch surface than intercepting `OnProceedButtonPressed`. The fact that vanilla already has this public method and the spec found it is good research. This is the right call.

- **Rule of Three for helper extraction (Decision 13).** Correct and disciplined. The ~200 LOC duplication is cheap insurance against a wrong abstraction. Preserve this ‚Äî don't let anyone pressure you into extracting early.

- **Run-ID guard (Decision 6).** Cleanly closes the B.1 race window noted in `notes/06`. The implementation is ~5 lines per patch with fail-open semantics. Good risk/effort ratio.

- **Independent failure isolation between vote and skip gate patches.** Failure mode table entries 1‚Äì2 show that if one patch's `Prepare` fails, the other still works. This is the right decomposition ‚Äî the vote and the gate are separable features.

- **Fail-open posture on every new failure mode.** The failure-mode table is thorough and consistent with B.1's philosophy. Every path degrades to vanilla. Good.

- **Copy-paste-modify discipline.** The spec is explicit about what changes vs. B.1 (7 numbered differences) and what stays the same. This makes the diff review tractable.

---

### 3. Concerns & risks

#### Severity: HIGH

**C1. Skip gate's `DisallowSkipping()` persists for the entire rewards-screen lifecycle ‚Äî even after the card is claimed.**

`DisallowSkipping()` sets vanilla's `_skipDisallowed = true`. There is no `AllowSkipping()` call anywhere in the spec. So if the rewards screen has a card reward AND a potion, and the streamer claims the card (via chat vote), the potion can no longer be skipped either. The streamer is forced to claim it. In strict mode (`cardSkipsPerAct: 0`), this means every rewards screen with a card reward also locks in all other reward types.

This is arguably consistent with the adversarial framing (chat doesn't want you to skip anything), but it's a side effect the spec doesn't acknowledge. If a streamer wants to skip a potion they don't want, they'll be confused about why Proceed is disabled. This is a UX friction point that should be explicitly documented and tested in the acceptance gate (currently no step tests a multi-reward-type screen post-card-claim).

**C2. Thread-safety model for skip-counter state is inconsistent.**

`CardRewardSkipGatePatch` shows `Interlocked.Increment(ref _actSkipsUsed)` in the `RewardSkippedFrom` postfix, but `_Ready` postfix resets `_actSkipsUsed = 0` and `_runSkipsUsed = 0` via plain assignment (no `Interlocked.Exchange`). If `_Ready` and `RewardSkippedFrom` could ever contend (even theoretically ‚Äî Godot signals are main-thread, so probably not), you'd get torn reads. But more importantly, the spec is *internally inconsistent*: it uses `Interlocked` for increments but not for resets and reads. Either commit to `Interlocked` for all accesses (including reads in the gate-evaluation logic) or document that all accesses are main-thread-only and `Interlocked` is unnecessary.

My recommendation: since all `NRewardsScreen` callbacks run on the Godot main thread, document that these fields are main-thread-only and remove `Interlocked.Increment` ‚Äî it's misleading about the threading model. Or, if you want to be defensive, use `Interlocked` consistently everywhere.

**C3. `EnglishReceipts.SkipReceipt` would violate the TI/Game seam.**

The spec says (in the Receipt format note): "add a `SkipReceipt(...)` static helper to `EnglishReceipts`" as option (a), preferred. But `EnglishReceipts` lives in `src/Ti/Voting/`. A "card reward skip" receipt is game-domain knowledge. Adding it to the TI core violates the load-bearing TI/Game seam (architectural decision #5). Option (b) ‚Äî inline in `CardRewardSkipGatePatch` ‚Äî is the correct choice for v0.1, or create a `Game/Receipts/GameReceipts.cs` helper if you want structure. Do not modify `EnglishReceipts` with game-specific receipt formats.

#### Severity: MEDIUM

**C4. Card reward votes happen ~16√ó per act; 30-second hardcoded duration is a pacing problem.**

B.1 hardcodes 30s for Neow (once per run). B.2.1 copies this for card rewards (once per combat). In a typical act with 16 combats, that's 8 minutes of pure waiting. This will feel very slow for viewers and streamer alike. The spec correctly defers settings-driven duration to B.2.4, but should explicitly acknowledge this pacing concern as a known issue, and consider whether a shorter default (e.g., 15s or 20s) for card rewards would be appropriate even before the settings knob exists.

**C5. No mechanism to undo `DisallowSkipping()` after card is claimed.**

Related to C1. The spec calls `DisallowSkipping()` in `_Ready` but never re-evaluates after the card reward is claimed via the vote patch. If vanilla has an `AllowSkipping()` method or if `_skipDisallowed` gets reset by vanilla's `RewardCollectedFrom`, the gate might self-correct ‚Äî but the spec doesn't confirm this. If vanilla doesn't reset it, the streamer is locked out of skipping all reward types for the entire screen. The spec should patch `RewardCollectedFrom` as well to re-evaluate `HasUnclaimedCardReward` and conditionally re-enable skipping, or explicitly document that "gate persists for screen lifecycle" is the intended behavior.

**C6. `_activeLabel` static reference becomes a dangling pointer when `NRewardsScreen` is freed.**

The label is parented under `NRewardsScreen`, so Godot frees it when the screen exits the tree. But `_activeLabel` still holds the managed reference. If the next `_Ready` postfix checks `_activeLabel != null` and calls `_activeLabel.UpdateText(...)`, it could operate on a freed Godot object (which throws or returns default values in Godot 4.x). The spec doesn't describe nulling `_activeLabel` on `_ExitTree` or checking `IsInstanceValid` before use. This is a real bug.

**C7. `ModEntryState.Settings` accessor is undefined but used in the skip gate.**

The pseudo-code references `ModEntryState.Settings` but this accessor doesn't exist in the codebase map. B.1's Neow patch doesn't need settings at all. If this requires adding a static settings accessor to `ModEntry` or a new `ModEntryState` class, that's a change to `ModEntry.cs` ‚Äî contradicting the spec's claim that "ModEntry.cs: no functional change." Either define the accessor or acknowledge the `ModEntry` change.

#### Severity: LOW

**C8. Vote winner ‚Üí card holder mapping at resume time is ambiguous.**

The spec says "bounds check on `winningCardHolder` against current card-holder list" but doesn't specify the mapping. The vote winner is an index (0, 1, 2). At resume time, you need to convert that index back to a card holder to re-call `SelectCard`. The spec should clarify: is the resume action `__instance.SelectCard(currentHolders[winnerIndex])` (re-derive holder from index) or `__instance.SelectCard(capturedCardHolder)` (use captured reference)? The former is correct (handles display-order changes); the latter is fragile (captured reference may be freed by reroll).

**C9. Card names may not be unique within a reward set.**

The spec uses `Card.Name.GetText()` for receipt labels and vote option names. Two cards could share a display name (e.g., a base card and its upgraded version, or two different-color Strikes if a relic adds off-color cards). Chat typing `#1` for "the second Strike" is ambiguous in receipts. Low severity for v0.1 (rare in practice), but should be documented as a known limitation.

### 4. Suggested changes

**S1. Add explicit documentation of the `DisallowSkipping()` scope persistence (addresses C1, C5).**

In the `CardRewardSkipGatePatch` section, add a subsection:

> **Gate scope**: `DisallowSkipping()` is called once in `_Ready` and persists for the entire `NRewardsScreen` lifecycle. After the card reward is claimed, skipping remains disabled for all reward types on the same screen. This is intentional for v0.1 ‚Äî the adversarial framing means chat's card-claim should also prevent streamer from skipping other rewards. If this proves frustrating in operator validation, a `RewardCollectedFrom` postfix can re-evaluate `HasUnclaimedCardReward` and conditionally call vanilla's skip-reenable path in a follow-up.

**S2. Consistent thread-safety model for skip counters (addresses C2).**

Either:
- (a) Document that all `CardRewardSkipGatePatch` callbacks run on the Godot main thread, remove `Interlocked.Increment`, use plain increments, and add a comment: `// Main-thread only; Godot signals are dispatched on main thread.`
- (b) Use `Interlocked` consistently for all reads, writes, and resets.

Recommend (a) ‚Äî it's simpler and accurate for Godot's execution model.

**S3. Choose option (b) for skip receipt formatting ‚Äî inline in `CardRewardSkipGatePatch` (addresses C3).**

Change the receipt format note to explicitly reject option (a) with the TI/Game seam rationale. If you want structure, add a small `static string FormatSkipReceipt(...)` private method within `CardRewardSkipGatePatch`.

**S4. Fix `_activeLabel` lifecycle ‚Äî null on screen exit (addresses C6).**

Add a `NRewardsScreen._ExitTree` Postfix (or connect to the screen's `TreeExiting` signal) that sets `_activeLabel = null`. Alternatively, check `IsInstanceValid(_activeLabel)` before every use. This is a 3-line fix but important for correctness.

**S5. Define `ModEntryState.Settings` accessor or acknowledge `ModEntry` change (addresses C7).**

Either:
- (a) Add a `internal static ModSettings.SettingsData Settings { get; private set; }` to `ModEntry`, set during initialization, and update the codebase map to show `ModEntry.cs` as ‚úèÔ∏è.
- (b) Have `CardRewardSkipGatePatch` read settings directly via `ModSettings.Load()` each time (wasteful but simple).
- (c) Add a new `ModEntryState` static class in `Game/Bootstrap/`.

Recommend (a) ‚Äî minimal change, clear ownership.

**S6. Clarify resume-action mapping: re-derive holder from index (addresses C8).**

In the `CardRewardVotePatch` section, specify:

> Resume action: `dispatcher.Post(() => ResumeOnMainThread(...))` where `ResumeOnMainThread` re-derives the card holder from the vote-winner index: `var holders = _cardHoldersField.GetValue(__instance) as ...; __instance.SelectCard(holders[winnerIndex]);`. This is more robust than capturing the card holder reference, which may be freed by a reroll.

**S7. Add a multi-reward-type scenario to the acceptance gate (addresses C1).**

Add to Step 1 or Step 2: "Rewards screen with card + potion: after card is claimed via vote, verify Proceed behavior for the remaining potion (claim required, skip unavailable)."

**S8. Acknowledge pacing concern for 30s card-vote duration (addresses C4).**

Add to Open Questions or Non-goals: "Card rewards occur ~16√ó per act vs. Neow's 1√ó per run. The 30s hardcoded vote duration may feel slow at this frequency. B.2.4's settings-driven duration will address this; consider a shorter default (15‚Äì20s) for card rewards when the settings knob is added."

### 5. Alternatives

**A1. Extend `VoteTallyLabel` with a skip-counter line instead of a separate `CardSkipCounterLabel`.**

The current spec creates a new Godot `Label` node parented under `NRewardsScreen` near the Proceed button. This requires managing Godot node lifecycle, anchor/position calculations relative to `_proceedButton`, and z-ordering. An alternative: add a line to the existing `VoteTallyLabel` (which is already parented under `SceneTree.Root` and has `_Process` polling). The skip counter is most relevant during votes, so co-location makes sense. This eliminates an entire new Godot node class (`CardSkipCounterLabel`), removes the anchor-positioning problem, and avoids the `_activeLabel` dangling-pointer issue (C6).

Trade-off: the counter would only be visible during votes (when `VoteTallyLabel` is attached to a session). Between combats, the streamer wouldn't see their remaining skips. But the counter is only *actionable* during votes, so this might be fine. If persistent visibility is needed, a separate label is better ‚Äî but consider whether v0.1 actually needs persistent visibility.

**A2. Single budget mode (`cardSkipsPerAct` only) for v0.1.**

The dual budget (per-act + per-run) adds a setting key, a counter, label complexity (`act + run` display), and receipt format complexity. The per-run limit is defense-in-depth: if per-act is 1, you get 3 total skips per run across 3 acts, which is already quite restrictive. Per-run adds marginal value. Consider: `cardSkipsPerAct` only for v0.1, add `cardSkipsPerRun` in B.2.4 if streamers request it. This simplifies the label, the receipt, and the settings parsing.

**A3. Patch `RewardCollectedFrom` to re-evaluate gate after card claim.**

Instead of letting `DisallowSkipping()` persist for the screen lifecycle, add a third postfix on `RewardCollectedFrom` that checks `HasUnclaimedCardReward` and, if no unclaimed card remains, calls vanilla's skip-reenable path (if one exists ‚Äî needs verification). This gives the streamer more freedom to skip non-card rewards after claiming the card. Trade-off: more patch surface, more complexity, needs verification of vanilla's re-enable mechanism. Probably a v0.2 refinement.

### 6. Additions

**A4. Document the interaction between `NCardRewardSelectionScreen` and `NRewardsScreen` lifecycles.**

The spec describes two patches on two different screens but doesn't explicitly state the screen lifecycle: (1) `NRewardsScreen._Ready` fires ‚Üí gate activates, (2) streamer clicks card button ‚Üí `NCardRewardSelectionScreen` opens (is `NRewardsScreen` hidden? paused? still in tree?), (3) vote happens on sub-screen, (4) card claimed ‚Üí sub-screen closes, (5) `NRewardsScreen` reappears. A brief lifecycle diagram would clarify when each patch fires and what state each screen is in during a vote.

**A5. Add `RewardCollectedFrom` behavior to the acceptance gate.**

No current step tests what happens after a card is claimed via vote ‚Äî does the Proceed button change state? Does the counter label update? Does the rewards screen close or remain open for other rewards? Add a scenario: "Rewards screen with card reward + gold: claim card via vote, verify gold can still be claimed, verify Proceed becomes available after all rewards handled."

**A6. Add a note about alternate-card-options on the sub-screen.**

The spec says alternates are not patched, but doesn't confirm that `SelectCard` is *not* called when an alternate is clicked. If `SelectCard` IS called for alternates (gold-instead-of-card), the vote would fire for a non-card option, which would confuse chat. Verify in `Prepare` or document the assumption that alternates use a different code path.

---

### 7. Removals

**R1. Remove the `EnglishReceipts.SkipReceipt` option (a) from consideration.**

The spec says "(a) preferred" but this violates the TI/Game seam. Removing it eliminates the ambiguity and prevents a seam violation during implementation.

**R2. Consider removing `CardSkipCounterLabel` as a separate class if A1 (extend `VoteTallyLabel`) is adopted.**

This would eliminate ~70 LOC and the Godot node lifecycle management. The `CardRewardSkipGatePatch` would still own the counter state; it would just communicate it to `VoteTallyLabel` via the session snapshot or a different mechanism.

### 8. Minor / nits

- **Receipt format for infinity**: The spec says "rendering `-1` as `‚àû`". Make sure the actual code uses the Unicode infinity sign `‚àû` (U+221E), not a custom string. Minor but worth being explicit.

- **`_multiplayerWarnFired` flag**: The `CardRewardVotePatch` pseudo-code includes this flag but the spec doesn't explain it. It's presumably copied from B.1's Neow patch (one-time warning log). Consider noting this explicitly so the implementer knows to copy it.

- **`DebugOnlyGetState()` method name**: The spec references `RunManager.Instance.DebugOnlyGetState()?.Id`. The "DebugOnly" prefix suggests this might not be available in release builds. If StS2 strips debug-only APIs in non-modded/release mode, this could break. Verify that this method is available when `ModManager.IsRunningModded()` is true (which it always is for mods). Flagging because I can't verify from the decompiled source.

- **Acceptance gate Step 2 receipt format**: Says `Streamer skipped a card reward (1/1 act, 0/‚àû run)`. After the skip, the counter should show `1/1 act used`, meaning remaining = 0. The receipt format shows used/limit, not remaining/limit. The label format shows remaining/limit. These are opposite conventions ‚Äî confirm this is intentional (receipt = "how many used", label = "how many left"). If so, document the distinction to avoid implementer confusion.

- **Failure mode 4 characterization**: "Counter not decremented" ‚Äî the counter is incremented, not decremented, in `RewardSkippedFrom`. If the postfix throws after `Interlocked.Increment`, the counter is correct. If it throws before, the counter undercounts. The spec should say "Counter not incremented" (undercount = gate is less strict than intended, which is fail-open and acceptable).

- **The `Lazy<FieldInfo?>` pattern for reflected fields**: The spec mentions this convention from B.1 but doesn't show it in the `CardRewardSkipGatePatch` pseudo-code. The implementer should use the same pattern for `_rewardButtons`, `_proceedButton`, etc. Worth a note.

### 9. Assumptions I'm making

1. **All `NRewardsScreen` callbacks (`_Ready`, `RewardSkippedFrom`, `RewardCollectedFrom`) run on the Godot main thread.** I'm assuming Godot's signal dispatch is single-threaded. If this is wrong, the thread-safety model in C2 needs revisiting and `Interlocked` should be used everywhere.

2. **Vanilla's `DisallowSkipping()` has no `AllowSkipping()` counterpart, and `_skipDisallowed` is not reset by vanilla after a reward is collected.** If vanilla does reset it in `RewardCollectedFrom` or `TryEnableProceedButton`, then C1/C5 are less severe than I've characterized. The implementer should verify.

3. **`NCardRewardSelectionScreen` is a child/sub-screen of `NRewardsScreen` that opens modally and returns to the parent screen when closed.** I'm assuming the parent rewards screen stays in the scene tree (possibly hidden) while the card sub-screen is open. If the rewards screen is freed when the sub-screen opens, the skip-counter label would be freed too, and `_activeLabel` would definitely dangle.

4. **`SelectCard(NCardHolder)` is NOT called for alternate options (gold-instead-of-card).** I'm assuming alternates use a different signal/method. If `SelectCard` is called for alternates, the vote would fire inappropriately (see A6).

5. **`NCardRewardSelectionScreen` doesn't expose a `DisableCardHolders()` or equivalent method** (analogous to `DisableEventOptions` on the Neow screen). The spec notes this as an intentional omission. I'm assuming this is correct and that the `_voteInProgress` re-entry guard is sufficient.

6. **`RunManager.Instance.DebugOnlyGetState()` is available at runtime when modded.** The "DebugOnly" name suggests it might be conditional. I'm assuming it works in modded mode (which the spec relies on heavily).

7. **The `_proceedButton` field on `NRewardsScreen` is accessible via reflection and has a stable position/anchor.** The label positioning depends on this. If `_proceedButton` is renamed or restructured in a game update, the label falls back to the rewards screen root at a fixed offset (spec documents this). I'm assuming this fallback is acceptable for v0.1.

8. **Card names returned by `Card.Name.GetText()` are unique within a single card-reward set.** I'm assuming the game doesn't offer duplicate card names in a single 3-card reward. If it does, chat receipts are ambiguous (see C9).
