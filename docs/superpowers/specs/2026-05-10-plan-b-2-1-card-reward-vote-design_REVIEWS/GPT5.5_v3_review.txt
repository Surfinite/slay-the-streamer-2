## Final verdict

v3 is substantially improved and close to implementation-ready, but I would fix a handful of remaining contradictions before generating the implementation plan ŌĆö mainly around skip-gate activation, run-ID fallback semantics, reroll-cancel receipts, and self-contained patch details.

## Must fix before implementation

### 1. Contradiction: run-ID guard failure disables patch vs only disables guard

In **Decision 6**, you say:

> if shape mismatch, log Warn and disable the run-ID check ŌĆö vote still works without guard

But in **`CardRewardVotePatch` ŌåÆ Reflected members verified by `Prepare`**, you say:

> If ANY check fails: log Error, return false from `Prepare`, patch silently skips.

That includes `RunManager.Instance`, `DebugOnlyGetState()`, and `RunState.Id`, so the two sections conflict.

Recommended fix:

Split `Prepare` requirements into two tiers:

```text
Hard Prepare requirements:
- NCardRewardSelectionScreen exists
- SelectCard(NCardHolder) exists
- _options exists and is readable
- card-holder collection exists and maps to _options
- required player-count / MP-bail access exists, if the patch depends on it

Soft Prepare requirements:
- run-state accessor exists
- run ID exists

If soft run-ID checks fail, patch still registers with run-ID guard disabled.
```

Apply the same distinction to `NeowBlessingVotePatch`.

### 2. Skip gate still activates when card voting is not actually operational

v3 fixes missing/malformed settings:

> Skip gate detects `Settings` is `Missing`/`Malformed` and degrades to vanilla.

Good.

But there are still cases where the skip gate may enforce even though chat voting cannot happen:

- `CardRewardVotePatch.Prepare` fails, but `CardRewardSkipGatePatch` still works.
- settings are valid, but Twitch is disconnected/read-only.
- `Voter.Default` is null or not usable.
- chat credentials are valid but connection never succeeds.

In those cases, the streamer may be forced to claim cards manually, but chat is not actually choosing. That weakens ŌĆ£degrade to vanilla.ŌĆØ

You need an explicit policy.

Recommended policy:

```text
Skip gate enforces only when card-vote infrastructure is available:
- SettingsResult.Success exists.
- CardRewardVotePatch prepared successfully.
- Voter.Default exists.
- Chat service is configured.

Temporary Twitch disconnects during a run do NOT disable the gate, because reconnect/backlog is expected.
Permanent no-settings/malformed-settings/no-voter/vote-patch-not-prepared degrade to vanilla.
```

Implementation can expose:

```csharp
internal static bool CardRewardVotePatch.PreparedSuccessfully { get; private set; }
```

Then skip gate checks it before enforcing.

If you intentionally want skip gate to enforce with valid settings even when chat is disconnected, state that directly.

### 3. Reroll receipt claim is overconfident without patching reroll

Decision 12 says:

> If streamer clicks reroll mid-vote, chat receives a `vote cancelled ŌĆö streamer rerolled` receipt.

But reroll is explicitly not patched. If you only detect resume failure via holder snapshot mismatch, you cannot reliably know the cause was reroll. It could also be:

- screen closed;
- run abandoned;
- card screen rebuilt;
- holder invalidated by some other UI transition;
- alternate reward path;
- game update changed structure.

So either:

#### Option A ŌĆö make receipt generic

Use:

```text
vote cancelled ŌĆö card reward changed or closed
```

or:

```text
vote cancelled ŌĆö card selection no longer valid
```

This matches what the code can actually prove.

#### Option B ŌĆö patch reroll or screen-change event

Out of scope per Non-goals, so I would not do this in B.2.1.

Recommended fix: replace ŌĆ£rerolledŌĆØ receipt with generic ŌĆ£card selection changedŌĆØ wording everywhere, including Acceptance Step 6.

### 4. Acceptance Step 6 assumes ŌĆ£look + back outŌĆØ is possible

Mode B validation says:

> open card sub-screen, see cards, return to rewards screen WITHOUT picking, click Proceed

This is only valid if vanilla actually supports backing out from `NCardRewardSelectionScreen` to `NRewardsScreen` without selecting.

If the card reward sub-screen has no back/cancel path, Mode B still exists in theory only if some UI path closes it. But the acceptance step would be impossible.

Recommended fix:

Change the acceptance wording to:

```text
If vanilla provides a back/cancel path from the card-selection sub-screen:
  open card sub-screen, inspect cards, return to rewards screen without picking, click Proceed.
  With cardSkipsPerAct: 1, skip is allowed.

If no such path exists:
  record that Mode B is only relevant to any future/rare vanilla path that returns from card selection without picking.
```

Or validate this in Task 1 before locking the acceptance gate.

### 5. `SkipBudgetTracker` assumes `Guid`, but spec says run ID type is uncertain

The tracker uses:

```csharp
private Guid? _lastSeenRunId;
public void ObserveRunAndAct(Guid? runId, int? actIndex)
```

But several parts of the spec say the run ID type must be pinned and is only assumed to be `Guid`.

If the actual type is not `Guid`, youŌĆÖll either rewrite the tracker during implementation or disable run detection unnecessarily.

Recommended fix:

Make tracker independent of the concrete run-ID type:

```csharp
internal sealed class SkipBudgetTracker
{
    private string? _lastSeenRunId;

    public void ObserveRunAndAct(string? runId, int? actIndex)
    {
        ...
    }
}
```

Then the Harmony layer converts the actual run ID to a stable string:

```csharp
string? runId = GetRunId(runState)?.ToString();
```

Or use `object?` plus `Equals`, but string is simpler and test-friendly.

Same for the vote guard: compare normalized run-ID strings unless the real type is definitely known.

### 8. `RewardSkippedFrom` record-before-settings could mutate tracker during degraded mode

### 13. Failure mode #8 may be too strong

## Final recommendation

IŌĆÖd call v3 **implementation-ready after a small v3.1 cleanup**, not a full redesign.

The most important edits are:

1. Resolve hard-vs-soft `Prepare` checks for run ID.
2. Define skip-gate activation when card voting/chat is unavailable.
3. Make reroll receipt generic unless reroll is actually patched.
4. Make v3 self-contained for `CardRewardVotePatch`.
5. Use a run-ID representation that does not assume `Guid`.

Everything else is polish or implementation-plan detail. The core design ŌĆö card vote via `SelectCard`, skip gate via `DisallowSkipping()`, per-act budget, pure `SkipBudgetTracker`, separate skip label, and Neow run-ID hardening ŌĆö is sound.
