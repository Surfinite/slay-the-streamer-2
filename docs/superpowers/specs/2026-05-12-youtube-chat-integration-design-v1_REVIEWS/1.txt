## 1. One-line verdict

Strong overall architecture and the right isolation boundary, but the spec is not implementation-ready yet because several contract mismatches around `MultiChatService`, skip-gate state, per-platform UI semantics, YouTube backlog/duplicate handling, and receipt/state reasoning could produce subtle production bugs.

---

## 2. What’s good

- **The high-level seam is right**: putting YouTube behind `IChatService` and preserving `VoteCoordinator` as a single-chat-service consumer is the correct architectural move. The aggregator pattern is a good fit for the existing Plan A surface.

- **Keeping YouTube fragility isolated under `src/Ti/Chat/YouTubeChat/` is exactly right.** The scraper will break eventually; containing all HTML/JSON/continuation assumptions in `YouTubeLiveChatScraper` and `YouTubeLiveBroadcastDiscovery` is the most important maintainability decision in the spec.

- **Read-only YouTube is the correct v1 constraint.** D3 is pragmatic. OAuth + Google verification would turn this from a shippable mod feature into an onboarding/product problem.

- **D9’s `"yt:"` prefix is a good low-impact compatibility move.** It avoids a `ChatMessage` schema change and gives deterministic cross-platform non-collision. Keep this.

- **The spec is unusually explicit about failure modes.** The Failure modes table is valuable and should be preserved, though several rows need tightening.

- **The single shared vote window is a reasonable v1 trade-off.** YouTube lag is real, but per-platform timers would multiply complexity and receipt UX problems.

- **Acceptance Step 0 is excellent.** The Twitch-only regression path is the most important thing to validate because this slice changes the chat plumbing even when YouTube is disabled.

- **No new Harmony patches is the right scope boundary.** This should stay a chat-layer slice.

---

## 3. Concerns & risks, ranked by severity

### Critical 1 — `ShouldEnforceSkipGate()` may be broken by aggregate state masking Twitch terminal failure

The spec repeatedly says `ShouldEnforceSkipGate()` is unchanged and that the YouTube child has no influence. But `ModEntry` now hands `VoteCoordinator` a `MultiChatService`, whose aggregate `State` is “best-of-children.”

This creates a dangerous ambiguity:

- Twitch: `AuthenticationFailed`
- YouTube: `ConnectedReadOnly`
- `MultiChatService.State`: likely `ConnectedReadOnly`

Acceptance Step 1 expects:

> Card-skip gate is degraded per Decision 21 amendment — terminal Twitch state disables the gate.

But if the gate currently consults the same chat service now hidden behind `Voter.Default` / `VoteCoordinator`, the aggregate state will hide the terminal Twitch failure and the gate may incorrectly remain enforced.

This is the most important thing to validate against the actual codebase.

Actionable fix:

- Add an explicit implementation note under `ModEntry wiring` / `Failure mode #21`:
  - Either `ShouldEnforceSkipGate()` must continue to consult the concrete Twitch service or a Twitch-specific state provider, not `MultiChatService.State`.
  - Or `ModEntry` must retain and expose the Twitch child separately for gate checks.
- Add a unit/integration test if possible:
  - Twitch terminal failure + YT connected ⇒ skip gate disabled.

Do not rely on “unchanged” here without code validation.

---

### Critical 2 — `YouTubeChatService.IsConnected` treats `Reconnecting` as connected

The spec says:

```csharp
public bool IsConnected => _state is
    ChatConnectionState.ConnectedReadOnly or
    ChatConnectionState.Reconnecting;
```

This is wrong or at least highly misleading. `Reconnecting` means the service is not currently reading chat. If `MultiChatService.IsConnected` is:

```csharp
_children.Any(c => c.IsConnected)
```

then a YouTube service with no live broadcast could make the multi appear “connected.”

Consequences:

- Vote infrastructure may think chat is available when it is not.
- YT-only with no live broadcast may look connected.
- Aggregate connection checks become semantically unreliable.
- Existing Twitch conventions may be violated.

Suggested fix:

```csharp
public bool IsConnected => _state is
    ChatConnectionState.ConnectedReadOnly or
    ChatConnectionState.ConnectedReadWrite;
```

If the author wants a separate “trying/recoverable” concept, expose it internally as `IsRecovering` or inspect `State == Reconnecting`. Do not overload `IsConnected`.

---

### Critical 3 — `VoteSession.TalliesByPlatform` semantics contradict the spec and acceptance gate

The spec says D10 requires split per-platform lines “when YT is enabled.” Acceptance Step 1 says YT-only smoke should show a YouTube line:

> In-game tally label renders YouTube line for any vote.

But the proposed accessor returns `null` unless more than one platform has been observed:

```csharp
public IReadOnlyDictionary<(string Platform, int OptionIndex), int>? TalliesByPlatform =>
    _observedPlatforms.Count > 1 ? new Dictionary<...>(_talliesByPlatform) : null;
```

That means:

- Twitch-only ⇒ single-line original rendering.
- YouTube-only ⇒ also single-line original rendering.
- The UI cannot distinguish Twitch-only from YouTube-only.
- Acceptance Step 1 cannot pass as written.
- Failure mode #21 says “YouTube-only single-platform rendering,” but Step 1 says “YouTube line.”

You need to choose one rule.

Recommended rule:

- If only Twitch is configured/observed: preserve original single-platform rendering.
- If YouTube is configured or any YouTube vote has ever been observed in this session: render platform-labeled lines.
  - YT-only: `YouTube: 0=1, 1=0, 2=0`
  - Dual: both rows.
  - Twitch-only with YouTube configured but no YT votes yet: either:
    - show only `Twitch:` row, or
    - show both rows with zeroes.

The cleanest implementation probably requires passing platform availability/configuration into the UI or `VoteSession`, not deriving everything from `_observedPlatforms.Count > 1`.

At minimum, fix the contradiction.

### Critical 4 — YouTube backlog / duplicate message handling is underspecified and could corrupt votes

The spec sets:

```csharp
ReceivedAt: _clock.UtcNow
```

for YouTube messages. That is fine for local receipt time, but the scraper may return backlog/history on initial connection or reconnect. If YT connects mid-vote, old chat messages containing `#0`, `#1`, etc. could be raised as if they arrived during the current vote.

Risk scenarios:

- Stream starts before the mod connects. Initial `get_live_chat` returns recent messages. Some contain vote-like strings from unrelated context. They get counted.
- YT disconnects/reconnects during an active vote. The first poll after reconnect returns messages from before the reconnect or duplicates from the previous continuation.
- Noita-pattern collision becomes worse: delayed `#1` from a previous vote appears during the next vote.

The spec needs an explicit policy:

Options:

1. **Cursor-establishing initial poll**
   - On initial connection, parse continuation but do not emit messages from the first response.
   - Start emitting only from subsequent polls.
   - This may drop legitimate messages during the reconnect window, but it avoids stale vote pollution.

2. **Message ID de-duplication**
   - Extract YouTube action/message IDs where available.
   - Maintain a bounded recent-ID cache per service.
   - Do not emit duplicates.

3. **Timestamp filtering**
   - Parse YouTube message timestamps if available/reliable.
   - Drop messages older than service connection time or vote start time.
   - This may be difficult because YouTube timestamps and local clocks are not always aligned.

Recommended v1 minimum:

- Do not emit messages from the initial cursor-establishing poll.
- Add a bounded duplicate cache if message/action IDs are available.
- Add tests specifically for:
  - initial poll contains `#1` but should not be counted;
  - duplicate message returned across two polls should only be counted once;
  - reconnect during vote does not replay already-seen messages.

This is important enough that I would not ship without addressing it.

### Critical 5 — `MultiChatService.ConnectionStateChanged` cannot compute “before aggregate” from current child state unless it caches state

The spec says:

> Computes aggregate before and after the child transition; raises iff different.

But by the time the child’s `ConnectionStateChanged` event is received, the child has probably already mutated its `State`. Unless `MultiChatService` maintains a cached `_lastAggregateState`, it cannot reliably compute the previous aggregate state from the children.

Required implementation detail:

```csharp
private ChatConnectionState _lastAggregateState;

private void OnChildConnectionStateChanged(...) {
    var before = _lastAggregateState;
    var after = AggregateState();

    ChildConnectionStateChanged?.Invoke(...);

    if (after != before) {
        _lastAggregateState = after;
        ConnectionStateChanged?.Invoke(...);
    }
}
```

Also make this thread-safe. Child services may raise events from different task/timer contexts.

Add a `MultiChatService` test:

- Child A changes state but aggregate remains unchanged ⇒ aggregate event does not fire.
- Child B changes state and aggregate changes ⇒ aggregate event fires exactly once with correct old/new aggregate states.

### High 6 — State and event threading model is under-specified

`YouTubeChatService` has mutable fields touched by:

- `ConnectAsync`
- `Task.Run` discovery
- poll loop task
- retry timer
- `Disconnect`
- `Dispose`
- dispatcher-posted message events

The pseudocode uses:

```csharp
private bool _disposed;
private ChatConnectionState _state;
```

But the context says the codebase convention is interlocked-int flags for cross-thread state, not raw `bool`.

Risks:

- Timer fires after `Dispose`.
- Poll loop transitions to `Reconnecting` after `Disconnect`.
- `ConnectionStateChanged` fires from a background thread while consumers assume main thread.
- `MessageReceived` is dispatcher-posted, but state changes may not be.
- `_cts` disposed while still used by the loop.
- double reconnect timers.

Suggested spec changes:

- Define whether `ConnectionStateChanged` is raised on the main thread. Ideally yes, matching `MessageReceived`.
- Use an explicit lock or interlocked state machine.
- Make `TransitionTo` the only place state changes happen.
- `ArmReconnect()` must cancel/dispose any existing retry timer before scheduling a new one.
- Timer callback must no-op if disposed, disconnected, or token cancelled.
- Add a test for `Dispose()` racing with an armed reconnect timer.

---

### High 7 — D8 receipts require reason information, not just state

Startup receipt logic says:

- YT `Connecting` or `Reconnecting` ⇒ “no live broadcast found, retrying.”

But `Reconnecting` could mean:

- no live broadcast;
- channel typo;
- HTTP 429;
- scraper shape regression;
- network failure;
- live ended;
- YouTube returned 500.

The spec knowingly accepts generic disconnect wording later, but the startup receipt example is too specific. It will lie in common cases.

Suggested fix:

Add a small internal reason enum or status string on `YouTubeChatService`, for example:

```csharp
public string? LastStatusReason { get; }
```

or:

```csharp
public YouTubeChatStatusReason LastStatusReason { get; }
```

Possible values:

- `NotConfigured`
- `NoLiveBroadcastFound`
- `LiveBroadcastEnded`
- `NetworkError`
- `RateLimited`
- `ScraperParseFailed`
- `UnknownError`

Then D8 receipts can be generic but not false:

- Startup:
  - `YouTube: not connected yet, retrying.`
  - `YouTube: no live broadcast found, retrying.`
  - `YouTube: temporarily unavailable, retrying.`
- Mid-session disconnect:
  - `YouTube disconnected; will retry automatically.`
  - Avoid saying “live broadcast ended” for network kills.

This also improves logs and operator diagnostics without changing public interfaces.

### High 8 — Fixed 60s retry is mostly fine, but HTTP 429 should respect `Retry-After`

D7’s “single cadence, no permanent/transient distinction” is acceptable for v1, but 429 is not the same as “no live broadcast.”

Current spec:

> YouTube ratelimits us ... retry in ~60s. No backoff.

This is unnecessarily risky. A tiny special case for 429 does not undermine D7.

Suggested compromise:

- Keep 60s ± jitter for all ordinary failures.
- If response is HTTP 429:
  - Respect `Retry-After` if present.
  - Else use a longer backoff, e.g. 5 minutes ± jitter.
- Log Warn once when entering rate-limited state, then Debug for repeated same-state retries.

This avoids hammering YouTube while preserving the simple state model.

### High 9 — `MultiChatService.ConnectAsync` throwing is a design smell

The spec accepts:

```csharp
public Task ConnectAsync(...) =>
    throw new NotSupportedException(...)
```

This violates the substitutability of `IChatService`. The exact consumer today may not call `ConnectAsync`, but future code or tests reasonably could.

Given the non-negotiable “`IChatService` interface unchanged,” I would not demand a large interface split now. But the spec should reduce the footgun.

Better options:

1. **Construct `MultiChatService` after children are connected, but make this explicit and test it.**
   - Keep the throw, but add a warning in the class XML comment and spec.
   - This is the minimal path.

2. **Add a static factory with connected children.**
   - `MultiChatService.FromConnectedChildren(...)`
   - Makes lifecycle expectations visible.

3. **Support `ConnectAsync` only for single-child multi.**
   - In Twitch-only deployment, it can delegate.
   - In multi-child mode, throw.
   - This is slightly weird but improves the one-child passthrough claim.

4. **Future cleanup: split interfaces.**
   - `IChatReader` / `IChatSender` / `IChatLifecycle`.
   - Not for this slice unless the codebase is already close.

At minimum, add tests that no production path calls `MultiChatService.ConnectAsync`.

### High 10 — Super Chat handling contradicts the non-goals

Non-goals say:

> Super Chat / Super Sticker handling. Reads the messages like normal chat; no monetary special-casing.

But `YouTubeLiveChatScraper` tests say:

> A response with a `liveChatPaidMessageRenderer` ... verify these are skipped without error.

That is a contradiction. Super Chat messages can contain text, and if a viewer sends `#1` as a Super Chat, the stated non-goal says it should be treated like normal chat.

Suggested fix:

- Treat `liveChatPaidMessageRenderer` with text as a normal chat message.
- Skip non-text membership/system items.
- Update tests:
  - paid message with text `#1` is extracted;
  - membership item without vote text is skipped;
  - sticker-only item is skipped.

### Medium 11 — The “1–2 LOC fix” scraper maintenance claim is overconfident

The spec says redesign fixes are “Most likely: 1–2 LOC change.” That undersells the risk.

Things that can break simultaneously:

- `INNERTUBE_API_KEY` extraction;
- continuation location;
- required client context;
- required visitor data;
- renderer type names;
- message text shape;
- author ID path;
- live redirect shape;
- bot/consent/rate-limit behavior.

Keep the optimistic maintenance model, but remove the “1–2 LOC” claim. Replace with:

> Expected maintenance is localized to `YouTubeLiveChatScraper` / `YouTubeLiveBroadcastDiscovery`; typical fixes should be small, but redesigns may require refreshing multiple extraction paths.

This is more honest and still reassuring.

### Medium 12 — VoteSession per-platform side-dict is acceptable, but the invariant needs stronger implementation guidance

I agree with the side-dict choice over a separate collaborator for v1. Co-locating latest-wins logic avoids duplication.

But the pseudocode is too informal around prior-vote handling:

```csharp
if (existing) {
    var priorKey = (platform, prior);
    ...
}
```

This is safe only if the same `VoterKey` can never change platform. That is true under D9, but the invariant should be explicit:

- The platform for a voter key is derived from the voter key and is stable.
- If future platforms are added, `PlatformOf(VoterKey)` must remain deterministic.
- Do not derive platform from display name or mutable fields.

Also add an invariant assertion/test:

- After every accepted vote:
  - sum of merged tallies == number of unique voters;
  - sum of per-platform tallies == sum of merged tallies when per-platform is exposed.

### Medium 13 — Per-frame dictionary allocation in `VoteTallyLabel`

This accessor allocates a new dictionary:

```csharp
new Dictionary<(string, int), int>(_talliesByPlatform)
```

`VoteTallyLabel._Process` calls it every frame. This is probably not catastrophic, but it is needless churn in Godot.

Suggested alternatives:

- `VoteSession.GetSnapshot()` returns an immutable/copy snapshot only on `TallyChanged`.
- `VoteTallyLabel` updates on `TallyChanged` instead of polling every frame, if existing design allows.
- Cache the rendered tally text until `TallyChanged`.

Not a blocker, but easy to avoid.

### Medium 14 — Discovery should not collapse all exceptions to `null` without preserving reason

`YouTubeLiveBroadcastDiscovery` says DNS failure, timeout, 404, no live, redirect loop all return `null` with Debug log.

For the service state, collapsing is fine. For diagnostics and D8 receipts, it is too lossy.

Suggested change:

- Keep the public-ish service behavior as “Reconnecting.”
- Internally return a result object:

```csharp
public sealed record LiveBroadcastDiscoveryResult(
    string? VideoId,
    YouTubeDiscoveryFailureReason? FailureReason);
```

If that feels too heavy, at least set `LastError` / `LastStatusReason` in `YouTubeChatService`.

### Medium 15 — Acceptance Step 3b conflicts with Failure mode #9 / UI behavior

Acceptance Step 3b says:

> Subsequent votes show single-platform rendering until YT reconnects.

Failure mode #9 says:

> `VoteTallyLabel` continues showing the YouTube line for the rest of the vote window ... v0.2 may collapse to single-platform if YT disconnects mid-vote.

Those can both be true if Step 3b means subsequent new votes, not the current vote, but the wording is ambiguous.

Clarify:

- During the current active vote after YT disconnects: keep stale YouTube row or collapse?
- On later votes after YT disconnected: single-platform or still show YouTube zero row because YT is configured?

This ties back to the larger D10 UI semantics issue.

### Medium 16 — `ChildConnectionStateChanged` API shape needs to be pinned

The spec shows both:

```csharp
new MultiChatService(params IChatService[] children)
```

and:

```csharp
new MultiChatService(("twitch", twitch), ("youtube", youtube));
```

Then says constructor shape can be pinned during implementation.

This is too loose for a design spec because D8 depends on child names.

Suggested pin now:

```csharp
public sealed record ChatServiceRegistration(
    string Name,
    IChatService Service);

public MultiChatService(params ChatServiceRegistration[] children)
```

or:

```csharp
public MultiChatService(params (string Name, IChatService Service)[] children)
```

Prefer a record/class over tuple if you want clearer tests and future metadata.

Also define valid names as constants:

```csharp
public static class ChatPlatformNames {
    public const string Twitch = "twitch";
    public const string YouTube = "youtube";
}
```

Avoid magic strings scattered across `VoteSession`, `MultiChatService`, and `ModEntry`.

### Medium 17 — YT-only mode is called a non-goal but acceptance Step 1 requires it to work

The non-goal says:

> YouTube-only deployments without Twitch. ... Architecturally supported ... don’t optimise for it.

Acceptance Step 1 is a YT-only smoke test with broken Twitch, and Failure mode #21 says YT votes still flow and control winner selection.

That is fine, but the wording “non-goal” is misleading. It is not first-class UX, but it is a supported degraded mode.

Suggested wording:

- Move “YouTube-only deployments” from **Non-goals** to **Supported degraded modes**.
- State:
  - votes work;
  - in-game label works;
  - receipts do not;
  - skip gate degrades if Twitch terminal.

### Medium 18 — Receipt flap suppression by fixed 30s debounce is arbitrary and underspecified

The spec says one receipt per 30s per platform/stable state. But “stable state” is not defined enough to implement reliably.

Questions:

- If YT goes `ConnectedReadOnly → Reconnecting → ConnectedReadOnly` within 10s, which receipt is emitted?
- If disconnect receipt is suppressed, should reconnect receipt also be suppressed?
- Does startup receipt count against the 30s debounce?
- Is debounce per state, per transition, or per message type?

Suggested implementation rule:

- Maintain last emitted receipt kind and timestamp:
  - `YouTubeConnected`
  - `YouTubeDisconnected`
- Suppress same receipt kind within 30s.
- Do not suppress a reconnect receipt after a disconnect if it represents user-visible recovery, unless the disconnect itself was suppressed.
- Startup receipt should not count as a mid-session state-change receipt.

Alternative: tie debounce to retry cadence:

- 2 × reconnect cadence, e.g. 120s, for repeated disconnects.
- No debounce for the first successful reconnect after a long outage.

I would keep 30s for v1 but specify exact behavior.

## 4. Suggested changes

### Must change before implementation

1. **Verify and explicitly preserve skip-gate Twitch-terminal semantics.**
   - Add a design note under `ModEntry wiring`:
     - `ShouldEnforceSkipGate()` must inspect Twitch state, not aggregate multi state.
   - Add an acceptance/unit test for Twitch terminal + YT connected.

2. **Change `YouTubeChatService.IsConnected`.**
   - `Reconnecting` must not count as connected.

3. **Resolve D10 / `TalliesByPlatform` contradiction.**
   - Decide whether platform rows appear:
     - only when both platforms have votes;
     - when YouTube is configured;
     - when any non-Twitch platform is observed.
   - Update Step 1, Step 3b, Failure mode #9, and `VoteTallyLabel` pseudocode accordingly.

4. **Define YouTube initial backlog and duplicate handling.**
   - Add a “stale/backlog suppression” subsection under `YouTubeChatService` or `YouTubeLiveChatScraper`.
   - Minimum: initial poll establishes cursor and does not emit messages.
   - Add duplicate message ID cache if feasible.

5. **Pin `MultiChatService` constructor/API shape.**
   - Use named child registrations.
   - Define platform name constants.
   - Remove “pin during implementation.”

6. **Make aggregate event state caching explicit.**
   - `MultiChatService` needs `_lastAggregateState`.
   - Add thread-safety note.

7. **Add reason/status tracking for YouTube reconnects.**
   - Avoid receipts that falsely say “no live broadcast” or “live broadcast ended” for network/parser/rate-limit failures.

8. **Fix Super Chat contradiction.**
   - Either paid messages with text count like normal chat, or the non-goal must say they are skipped. I recommend counting text-bearing paid messages.

---

### Should change before implementation

9. **Special-case HTTP 429.**
   - Respect `Retry-After`.
   - Use longer fallback backoff for 429 only.

10. **Strengthen `YouTubeChatService` threading/disposal contract.**
    - State changes through one synchronized method.
    - Timers cancelled on disconnect/dispose.
    - No callbacks after disposed.
    - State events dispatched consistently.

11. **Reduce per-frame allocations in `VoteTallyLabel`.**
    - Cache snapshot/rendered text on tally changes if reasonable.

12. **Add tests for `YouTubeChatService`.**
    - The context says ~8 tests are expected, but the spec’s test tree omits `YouTubeChatServiceTests.cs`.
    - Include state transitions, reconnect timer, dispose race, initial backlog suppression, and dispatcher behavior.

13. **Make log levels less noisy for permanent bad config.**
    - First invalid/no-live warning: Warn.
    - Repeated identical failures: Debug or rate-limited Warn.
    - This still honors D7 without log spam.

## 5. Alternatives

### Alternative A — Keep `MultiChatService` as an event/read/send aggregator, not a lifecycle service

If `ConnectAsync` throwing feels too ugly, introduce an internal narrower interface later:

```csharp
public interface IChatMessageSource {
    event EventHandler<ChatMessage>? MessageReceived;
    ChatConnectionState State { get; }
    bool CanSend { get; }
    Task SendMessageAsync(...);
}
```

Then `VoteCoordinator` could eventually depend on the narrower consumer surface. Not necessary for this slice, but this is the cleaner long-term direction.

---

### Alternative B — Use a `PlatformAwareChatMessage` internally, but keep public `ChatMessage` unchanged

The spec rejects adding `Platform` to `ChatMessage`, which is reasonable. But internally, `MultiChatService` could wrap child messages with metadata before forwarding to vote/session layers.

Example:

```csharp
internal sealed record RoutedChatMessage(
    string Platform,
    ChatMessage Message);
```

This would avoid `PlatformOf(msg)` parsing `"yt:"` in `VoteSession`.

Downside: it likely touches `VoteCoordinator` or `VoteSession` call paths more than D9 wants. For v1, the prefix approach is acceptable.

---

### Alternative C — Per-platform tally collaborator

The spec asks whether `_talliesByPlatform` should be a separate class. I would not require it now, but if `VoteSession` is already large, a small helper could work:

```csharp
internal sealed class PerPlatformVoteTallies {
    void ApplyVote(string platform, int? previousOption, int nextOption);
}
```

Important: it must not own voter identity or latest-wins policy. It should only apply deltas supplied by `VoteSession`.

This avoids duplicating latest-wins logic while keeping `VoteSession` smaller.

---

### Alternative D — Make startup YT status log-only, not receipt-bearing

Given the known Twitch ratelimit issue, an alternative is:

- Startup receipt remains unchanged unless YT is connected.
- Non-connected YT status goes to log only.
- Mid-session “YouTube connected” receipt fires once when YT becomes useful.

This reduces receipt volume and avoids telling chat about implementation details. I do not think this is necessary, but it is worth considering if ratelimit validation fails.

## 6. Additions

Add the following to the spec.

### A. YouTube stale/backlog policy

New subsection under `YouTubeChatService`:

- Initial poll establishes cursor.
- Initial response messages are not emitted unless explicitly known to be live/new.
- Maintain optional bounded recent-message/action ID cache.
- Reconnect does not replay already-seen messages.

### B. Child state reason model

Add a small reason/status concept:

```csharp
public enum YouTubeChatStatusReason {
    None,
    NoLiveBroadcastFound,
    LiveBroadcastEnded,
    NetworkError,
    RateLimited,
    ScraperParseFailed,
    InvalidOrUnavailableChannel,
    UnknownError
}
```

Even if private/internal, this will make D8 receipts and logs much better.

### C. Explicit main-thread/event dispatching contract

State whether each event is raised on main thread:

- `MessageReceived`: yes, via dispatcher.
- `ConnectionStateChanged`: should also be yes unless existing Twitch behavior says otherwise.
- `ChildConnectionStateChanged`: same as aggregate state event.

If not main-thread, say so and require subscribers to dispatch.

### D. Tests missing from the spec

Add:

- `YouTubeChatServiceTests.cs`
  - no-live ⇒ `Reconnecting` + retry armed;
  - initial parse failure ⇒ `Reconnecting`;
  - poll failure ⇒ `Reconnecting`;
  - dispose cancels retry;
  - initial poll messages suppressed;
  - duplicate message IDs suppressed;
  - messages raised via dispatcher;
  - no events after dispose.

- `VoteTallyLabel` tests if feasible, or at least a pure formatter helper test.
  - Current rendering is operator-only; the platform-row contradiction makes unit coverage valuable.

- `ModEntry`/receipt tests if there is an existing bootstrap test seam.
  - Especially debounce and startup suffix logic.

### E. Operator validation for stale message replay

Add to Acceptance Step 3 or Step 4:

- Send `#1` in YT chat before a vote starts.
- Reconnect YT or start mod.
- Start vote.
- Verify the old `#1` does not count.

### F. Configuration documentation

Add a README/settings note:

- `youtubeChannelId` must be the stable `UC...` channel ID, not `@handle`.
- Public live chat required.
- YouTube receipts are not posted.
- Same person voting on both platforms counts twice.
- YT votes may be underrepresented due to lag.

    - one child `AuthenticationFailed`, one child `JoinFailed`;
    - no child connected/reconnecting/connecting;
    - current “all children in JoinFailed” / “all children in AuthFailed” rules would skip both and maybe fall to what?
  - Define mixed-terminal behavior.

- `LastError` on `MultiChatService`:

  ```csharp
  FirstOrDefault(x => x is not null)
  ```

  This is arbitrary. Prefer most recent error if children expose timestamps, or document arbitrary first child.

- `LastMessageReceivedAt` with nullable LINQ is probably okay, but make sure empty sequence returns null and does not throw.

- `Login: msg.AuthorChannelId` for YouTube is acceptable, but consider `Login: $"yt:{msg.AuthorChannelId}"` for consistency if any downstream diagnostics display login. Since `UserId` drives `VoterKey`, this is low priority.

- `YouTubeLiveBroadcastDiscovery` tests mention “mock returning a redirect chain,” but if using `HttpClientHandler.AllowAutoRedirect = true`, a low-level mock may only expose the final response. Keep tests aligned with the actual HTTP abstraction.

- `PollCeiling = 10s` may be too aggressive if YouTube legitimately asks for longer. Fine for v1, but log Debug when clamping unusually high values.

- “Jitter ±10s” should define bounds:
  - 50–70s?
  - or 60s plus random 0–10s?
  - Avoid negative jitter accidentally if implementation changes.

- `OutgoingMessagePriority.Normal` for D8 receipts may worsen the known Twitch rate-limit issue. Consider Low for YT status receipts unless they are operationally important.

- Acceptance Step 3e says “mock the scraper” in an operator-validation gate. That is not really operator validation unless there is a test hook/dev flag. Either add a dev injection hook or move 3e to automated tests.

## 9. Assumptions I’m making

- I do not have direct codebase access, so the skip-gate concern depends on how `ShouldEnforceSkipGate()` currently obtains chat state. If it already holds a Twitch-specific reference, the risk is lower; if it reads aggregate `Voter.Default`/`VoteCoordinator` state, this is critical.

- I assume existing `IChatService.IsConnected` means “currently able to receive chat,” not “will retry eventually.” If the codebase uses a different semantic, rename or document it, because the current proposed YT implementation is surprising.

- I assume `VoteSession` only receives messages while a vote is active, but the chat service itself runs continuously. That makes YouTube initial backlog suppression important when a service connects or reconnects during a vote.

- I assume `TwitchIrcChatService` has some main-thread dispatch convention for `MessageReceived`; the YouTube service should match it. If Twitch raises state events from background threads already, then the spec should explicitly say `ModEntry` subscribers are thread-safe.

- I assume YouTube `get_live_chat` responses can include historical/replayed actions and duplicate-ish actions across continuation boundaries. Even if rare, the cost of a stale `#1` being counted is high enough to specify a policy.

- I assume paid YouTube messages can contain ordinary text and should count if the text is a valid vote command, based on the spec’s own “treated as normal chat messages” non-goal.

- I assume preserving v0.1 Twitch-only behavior is more important than making aggregate-state semantics elegant. Therefore, any ambiguity should be resolved in favor of not changing Twitch/gate behavior when YouTube is absent or broken.
