Now let me read the spec itself:# Review: YouTube chat parallel integration (design v1)

## One-line verdict

Architecturally sound and well-reasoned, with mostly cosmetic-to-medium issues — but there are two real semantic bugs (`IsConnected` includes `Reconnecting`, and `TalliesByPlatform` gates on "observed" rather than "configured") that should be fixed before implementation, plus a handful of contract-shape decisions worth pushing back on.

---

## What's good (preserve)

- **Aggregator pattern is the right shape.** `MultiChatService : IChatService` with `VoteCoordinator` unchanged is the minimum-invasive design that actually works. Single-child passthrough as the Twitch-only case ("always wire a `MultiChatService`") is the right call — one code path beats branching at `ModEntry`. Preserve.
- **D9's `"yt:"` prefix discipline over a `Platform` field.** Avoiding a `ChatMessage` schema change is correct; the prefix is structurally forward-compatible (Discord/Kick/etc. get their own prefix later). The `PlatformOf` helper as a private static keeps the discrimination boundary tight. Preserve.
- **Risk concentration analysis is honest.** The "~30% LOC carries ~70% of redesign risk, all isolated in `Ti/Chat/YouTubeChat/`" framing is realistic and accurately predicts the maintenance shape. Preserve.
- **Side-dict over class extraction for the per-platform tally.** Rule-of-three applies; class extraction is premature here. Preserve until the second use case appears.
- **D7's "one cadence, no permanent/transient disambiguation"** is the right call. The cost (one Warn per minute on typo'd channel ID) is negligible; the alternative (404-shape heuristics) is fragile and would consume disproportionate test surface. Preserve.
- **Failure-modes table at 21 enumerated entries** is thorough — that's the kind of completeness B.2.1 v4 also had. Preserve the discipline.
- **Acceptance gate Step 0 (vanilla regression via new aggregator)** is exactly the right shape to catch "the new code path is subtly different from the old direct path". Preserve.

## Concerns & risks (ranked by severity)

### HIGH — likely real bugs

**1. `YouTubeChatService.IsConnected` includes `Reconnecting` (line 183-185).** This is semantically wrong:

```csharp
public bool IsConnected => _state is
    ChatConnectionState.ConnectedReadOnly or
    ChatConnectionState.Reconnecting;
```

`Reconnecting` is explicitly "I'm not happy but I'm trying" per your own state machine notes (section "Notes on the state choices"). A consumer asking `IsConnected` and getting `true` while the service is mid-reconnect will draw the wrong conclusion. Worse: `MultiChatService.IsConnected => _children.Any(c => c.IsConnected)` will report aggregate "connected" when no child is actually connected — only retrying. The B.2.1 skip gate isn't affected (it reads Twitch-specific terminal states, not `IsConnected`), but anything else reading `IsConnected` is misled. **Either flip the semantic (`IsConnected => ConnectedReadOnly or ConnectedReadWrite` only) or rename to `IsAliveOrRetrying`.** Almost certainly the former.

**2. `_observedPlatforms`-gated `TalliesByPlatform` produces a mid-vote rendering snap.** Section "What changes" in `VoteSession`:

```csharp
public IReadOnlyDictionary<...>? TalliesByPlatform =>
    _observedPlatforms.Count > 1 ? ... : null;
```

Concrete scenario: vote opens, Twitch chat votes first (5s in), YouTube chat votes 8s in (lag). The `VoteTallyLabel` renders single-platform for 8 seconds, then snaps to two-line split rendering when the first YT message lands. That visual snap is jarring and reveals implementation detail. The decision of "show split vs single" should derive from **configuration** (i.e., `youtubeChannelId` is non-null AND YT child is in any non-`Disposed` state), not from "have we seen a YT message yet". Practically: pass a `bool isMultiPlatformConfigured` into the `VoteSession` or expose it from `MultiChatService` and have `VoteTallyLabel` consult that instead. **This wants a fix before implementation.**

### MEDIUM — contract smells worth pushing back

**3. `MultiChatService.ConnectAsync` throws `NotSupportedException` (Q4).** You picked option (b) and documented; I'd push for (a) — split `IChatService` into a consumer interface (`IChatService` minus `ConnectAsync`) and an operator interface (`IConnectableChatService` adding it). `VoteCoordinator` and the multi only see the consumer half; `ModEntry` and the children see the operator half. Concrete benefit: the runtime exception goes away, the test suite can't accidentally call ConnectAsync on the multi, and the type system documents the design ("the multi doesn't own connect lifecycle") rather than a comment. Cost: two interfaces and a small refactor of consumers. Worth it. The current shape is a code smell that future-you will trip over.

**4. `MultiChatService.SendMessageAsync` fan-out semantics are unspecified for partial failures.** Line 360-364 says "Fan-out only to children whose `CanSend == true`. Returns `Task.WhenAll` of each child's send." But: if Twitch sends successfully and a hypothetical future Discord child fails, `WhenAll` throws `AggregateException` — the caller sees "failure" even though the Twitch chatters DID receive the receipt. Today this is hypothetical (only Twitch is `CanSend`), but the spec is for a v1 surface that will be extended. **Specify**: continue past individual failures, log each, return a success summary or `Task.CompletedTask` if at least one child succeeded. Add a test for the two-CanSend-children case where one throws.

**5. Hardcoded YouTube client metadata (`"clientVersion": "2.20240101.00.00"` at line 229).** This string will rot. YouTube can and does reject requests claiming to be from old client versions (this is how their A/B rollout walls work). The `live_chat?v={videoId}` page you already parse contains the current `INNERTUBE_CONTEXT.client.clientVersion` — extract it alongside `INNERTUBE_API_KEY` and pass it through. Tiny LOC delta, removes a bit-rot trap. The fixture-based scraper tests already cover this if you extract from the same page.

**6. Failure mode 17 (`yt:anon-{randomGuid}` fallback for missing `authorChannelId`).** This is a one-message-one-voter pattern; if YouTube ever ships a response shape where `authorChannelId` is systematically missing for some message class (e.g., a new "incognito chat" feature), the per-message-unique-voter behaviour silently swamps the tally with single-use ballots. **Safer default: drop the message entirely with a Debug log**, the way you handle other malformed renderers. The cost of dropping is identical to the cost you already accept for malformed messages; the cost of admitting them is catastrophic in the worst case.

**7. HTTP client lifecycle is unspecified.** Section "Service shape" has `Func<IYouTubeHttp> httpFactory` but no explanation of: who owns the `HttpClient`, what its timeout is, whether one is shared across discovery + scraper or two are spawned. If `IYouTubeHttp` wraps a fresh `HttpClient` per call you'll hit socket exhaustion under any sustained polling load (the classic .NET gotcha); if it's a singleton, where does it get disposed? Specify: shared per-service `HttpClient` instance, `AllowAutoRedirect = true`, per-request timeout (15s? 30s? — pin a number), disposed on `Dispose()`. This is a one-paragraph addition that prevents a real production failure mode.

**8. `MultiChatService.LastError` is non-deterministic.** Line 348: `_children.Select(c => c.LastError).FirstOrDefault(x => x is not null)`. Enumeration order picks the winner. If both children have errored, the consumer sees one and is silently blind to the other. Either document "Twitch errors win" explicitly (which assumes ordering), expose both, or pick the most recent by timestamp. Low-stakes since `LastError` is mainly diagnostic, but worth a one-line decision.

**9. The two-channel event API (`ConnectionStateChanged` + `ChildConnectionStateChanged`) is a smell.** You acknowledge this in Q5. The cleaner shape: a single `ConnectionStateChanged` event whose `ChatConnectionChangedEventArgs` is extended (or subclassed) to carry an optional `ChildName` for child-originated transitions, and `null`/empty for aggregate-only transitions. `ModEntry` filters on `e.ChildName is not null`; the existing Twitch-only consumers see the existing shape. One event channel, no consumer needs to subscribe to two streams to get the full picture. The current two-channel shape forces ModEntry-the-only-consumer to subscribe twice, and confuses any future consumer about which event to listen to.

### LOW — nits and small additions

**10. `ChildName` as a magic string in `ChildConnectionStateChangedEventArgs` (line 393-395).** `"twitch"` and `"youtube"` are bare strings; if anyone typos one in the receipt formatter (`"twitter"`), the receipt silently misroutes and there's no compile-time catch. A small `internal enum ChildKind { Twitch, YouTube }` (or `static class ChildNames { public const string Twitch = "twitch"; ... }`) would lock this down for ~5 LOC.

**11. `Reconnecting` is a misleading name for "no live broadcast yet".** When the streamer hasn't started YouTube broadcasting, you've never connected — there's nothing to *re*-connect. Headline-shift 1 acknowledges this; the disconnect-receipt wording even leaks ("live broadcast ended, will resume...") which is wrong for the never-connected case. Two options: (a) accept the misnomer, sharpen the receipt wording to "YouTube: not currently live, will track when broadcast starts"; (b) introduce a new state `WaitingForBroadcast` — but that's an `IChatService` schema change and goes against your "minimum surface change" discipline. Option (a) is the right call; just fix the receipt strings.

**12. Platform-line ordering in `VoteTallyLabel` (line 513) is enforced by `OrderBy(p => p, StringComparer.Ordinal)` but never unit-tested.** Operator validation Step 4 would catch a regression but slowly. Add a single test in `VoteSessionPerPlatformTallyTests` (or a small `VoteTallyLabelFormatterTests`) pinning "Twitch row comes before YouTube row alphabetically". The formatter is pure enough to test off the Godot tree if you factor the `StringBuilder.Append` block out.

**13. Receipt-flap debounce of 30s (Q6) is arbitrary.** Tie it to the reconnect cadence: `2 × ReconnectBase = 120s`. Rationale: "no more than one receipt per two retry attempts" is a defensible invariant; "30s" is a number. Document the rationale next to the constant.

**14. `_observedPlatforms` lifecycle.** I'm assuming `VoteSession` is constructed per vote (single Open→Close cycle), in which case `_observedPlatforms` is naturally scoped to the vote. If `VoteSession` is reused across votes, the set needs to reset on `Open()`. Worth one line in the spec to pin which it is.

**15. Headline-shifts numbering in Author's note vs Decisions table.** Headline-shift 2 is the channel-ID-invalid edge-case; in the Decisions table D7 is the source. Cross-referencing them ("Headline-shift 2 ↔ D7") would help reviewers tie them together. Minor doc polish.

## Suggested changes

In rough order of "fix this before moving to plan-writing":

1. **Fix `IsConnected` to exclude `Reconnecting`** in `YouTubeChatService` (Concern 1). Add a unit test asserting `IsConnected == false` while in `Reconnecting`. Verify the same property on `TwitchIrcChatService` doesn't have the same bug while you're there.
2. **Change `TalliesByPlatform` gating from "observed" to "configured"** (Concern 2). Pass a `bool` (or `int expectedPlatformCount`) into `VoteSession` at construction, derived from `MultiChatService.Children.Count` or similar. The `_observedPlatforms` set can stay as a "seen" tracker but shouldn't drive the multi-line rendering decision.
3. **Drop the random-GUID fallback for missing `authorChannelId`** (Concern 6). Skip the message; log Debug.
4. **Specify `HttpClient` lifecycle** in the `YouTubeChatService` service-shape section (Concern 7). Pick a timeout number.
5. **Extract `clientVersion` from the live_chat page** instead of hardcoding (Concern 5).
6. **Specify partial-failure semantics for `MultiChatService.SendMessageAsync`** (Concern 4). Add a test.
7. **Carve out HTTP 429 for backoff** (Q3, answered below). One special case for the one failure mode where YouTube has explicitly told you to slow down is worth the asymmetry.
8. **Consolidate the two-channel event API into one** (Concern 9). Extend `ChatConnectionChangedEventArgs` with an optional `ChildName`.
9. **Replace magic strings with constants/enum** for child names (Concern 10).
10. **Fix `Reconnecting`-vs-never-connected receipt wording** (Concern 11).

## Alternatives

- **Split `IChatService` into consumer + operator interfaces** instead of the throw-from-`ConnectAsync` (Q4, Concern 3). I'd push for this. The cost is small; the benefit is the runtime exception goes away and the design becomes self-documenting.
- **Pin `clientVersion` via a periodic refresh** rather than per-page-load, if extraction-per-load is too expensive. Cache it for the lifetime of `_innertubeApiKey`; re-extract on the next page-load (which already happens on reconnect).
- **For the per-platform tally**, a small `PerPlatformVoteTally` collaborator class (Q2 alternative) is in fact cleaner long-term, but I agree with deferring it. When the third platform appears (Discord, Kick, etc.) extract then. v1 side-dict is fine.

## Answers to the Key Questions for Reviewers

1. **All-transient-failures → `Reconnecting` (D7 + headline-shift 2).** Right call. The cost of distinguishing permanent from transient is fragile heuristics on a 404 response shape that YouTube changes silently; the cost of not distinguishing is one Warn line per minute. Easy trade.
2. **Per-platform tally as side-dict vs separate class.** Side-dict for v1; extract when a third platform arrives. Rule of three.
3. **Fixed 60s vs backoff for 429.** Carve out 429 specifically. Use the same 60s base for everything else, but on 429 multiply the next-attempt interval by 2 (cap at, say, 30 minutes) until a non-429 response, then reset. Aggressively retrying a service that has told you "slow down" risks IP bans; the spec's "we never get ratelimited in practice" is hopeful.
4. **`MultiChatService.ConnectAsync` throws.** Push for (a) — split the interface. The throw is a smell; the split is small.
5. **Aggregate best-of-children + separate `ChildConnectionStateChanged`.** Best-of-children for `State` is right. Two event channels is wrong; collapse into one (Concern 9).
6. **30s vs 120s receipt debounce.** Pick 120s (= 2× reconnect cadence) and document the rationale. 30s is a number; 120s is a derivable invariant.
7. **Slice ordering: this vs B.2.2.** Architecturally B.2.2 first. It stays in the well-trodden Harmony path with no new external dependency; this slice introduces live-YouTube-broadcast-required operator validation, which is meaningfully harder to schedule. Practically: FrostPrime drives the decision.
8. **Scraper isolation realism.** Generally yes for shape changes — 1-2 LOC + fixture refresh is realistic. The riskier case the spec correctly flags is **auth-model change** (session cookies, anti-bot challenges), where the whole approach is dead. I'd add: also expect periodic minor breaks where the `liveChatRenderer` JSON path moves by one level (e.g., wraps in a new container). Plan for a fixture refresh per ~quarter, not per ~year.
9. **Worst-of-both-worlds mode.** Accept it as documented-but-degraded. In-game label is sufficient feedback for the rare configuration where Twitch is dead but YT is alive; don't optimise for it.
10. **Noita pattern in this spec or deferred.** Fold a minimum fix in. The lag amplification is real; shipping the YT integration and deferring the Noita fix means the first ~6 months of YT usage is meaningfully worse than the Twitch baseline. A generation-token mechanism on `VoteSession` (vote-id stamped on each `ChatMessage` event subscription, drop messages received after Close) is small; the test surface fits in `VoteSessionTests`.

## Assumptions I'm making (please validate)

These are places where I lacked codebase visibility and inferred:

1. **`VoteSession` is constructed per vote**, not reused across votes. If reused, `_observedPlatforms` needs an explicit reset path. (Concern 14.)
2. **`TwitchIrcChatService.IsConnected` returns `false` during `Reconnecting`** — i.e., that the YouTube service's inclusion of `Reconnecting` in `IsConnected` is the *deviation* from the established pattern, not the pattern itself. If TwitchIrcChatService also does this, my Concern 1 is invalidated — though the semantic concern remains.
3. **`ChatConnectionChangedEventArgs` is a record/class that can be extended** without breaking existing consumers (Concern 9 mitigation). If it's sealed and consumed by reflection elsewhere, that approach needs adjustment.
4. **`IChatService` consumers outside `VoteCoordinator`** are limited to `ModEntry` and the receipt-sender path. The "split into consumer/operator interfaces" alternative (Concern 3) is cheap only if the consumer count is genuinely 1-2.
5. **`MegaCrit.Sts2.Core.Logging.Logger` is thread-safe** — the YT poll loop is on a `Task.Run` thread and calls `TiLog.Warn`/`TiLog.Debug` directly. If the sink isn't thread-safe, the spec needs a marshalling step.
6. **`HttpClient` instances persist across reconnect cycles.** If the spec author plans to construct a new client per `Connecting` transition (which would be wrong), my Concern 7 fix is more invasive than I've described.
7. **The B.2.1 `ShouldEnforceSkipGate()` reads `TwitchIrcChatService.State` directly, not through `VoteCoordinator` or the (new) `MultiChatService`.** Context section 6 says this; the spec preserves it. If that direct read is via `Voter.Default` or some other indirection that I'd see in the code, the per-platform aggregation might leak in unexpected ways.
8. **Operator-validation Step 7 of B.2.1 (the Twitch ratelimit issue mentioned in context section "Known issues")** is well-understood and won't be made worse by D8's startup-and-state-change receipts. The context flags this as "a real risk that operator-validation Step 7 of this spec needs to confirm doesn't worsen the burst problem"; I'm taking that flag at face value but the spec doesn't explicitly tie its Step 7 back to ratelimit-burst measurement. Might be worth an explicit acceptance criterion.
