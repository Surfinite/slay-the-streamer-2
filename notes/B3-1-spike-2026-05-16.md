## Step 1: `NCreatureVisuals.SpineBody` non-null when `HasSpineAnimation` is true

**File:** `decompiled/sts2/MegaCrit/sts2/Core/Nodes/Combat/NCreatureVisuals.cs`

**Findings:**

- Line 114: `public bool HasSpineAnimation => SpineBody != null;` ŌĆö simple null check on the property.
- Lines 145ŌĆō153 in `_Ready()`:
  ```csharp
  if (IsSpineNode)
  {
      SpineBody = new MegaSprite(_body);
      if (SpineBody.GetSkeleton()?.GetData() == null)
      {
          GD.PushWarning(...);
          SpineBody = null;  // <-- can be set back to null here
      }
  }
  ```
- Line 102ŌĆō111: `IsSpineNode` returns `_body.GetClass() == "SpineSprite"` (live Godot instance check).

**Nuance ŌĆö degenerate path:** If `SpineBody.GetSkeleton()?.GetData() == null` after construction, `SpineBody` is set back to `null` and a `GD.PushWarning` fires. So `HasSpineAnimation` would be `false` even though `_body` IS a SpineSprite. This is a defensive path for broken data; it won't happen in production where MegaCrit ships valid Spine data. The guard `if (visuals.HasSpineAnimation)` before calling `monster.GenerateAnimator(visuals.SpineBody)` is correctly placed ŌĆö `SpineBody` is definitively non-null when `HasSpineAnimation` is true.

**Verdict: CONFIRMED.** The spec's guard pattern is correct. One edge case (broken skeleton data) is already handled by the vanilla code and surfaces as `SpineBody = null`, so our `HasSpineAnimation` check gates it correctly regardless.

## Step 2: `NCreatureVisuals.Bounds` non-null after `CreateVisuals`

**Verdict: CONFIRMED** (assuming well-formed scenes; degenerate case would crash loudly, not silently return null).

## Step 4: `PreloadManager.LoadActAssets` covers monster combat scenes

**File:** `decompiled/sts2/MegaCrit/sts2/Core/Assets/PreloadManager.cs` (lines 65ŌĆō73) and
`decompiled/sts2/MegaCrit/sts2/Core/Models/ActModel.cs` (lines 69ŌĆō99).

**Findings:**

`PreloadManager.LoadActAssets` builds `AssetSets.Act` from `act.AssetPaths`. Reading `ActModel.AssetPaths` (lines 69ŌĆō99):

```csharp
List<string> obj = new List<string> { BackgroundScenePath, MapBotBgPath, MapMidBgPath, MapTopBgPath };
// + _rooms.Ancient.MapNodeAssetPaths (if present)
// + _rooms.Boss.MapNodeAssetPaths
// + _rooms.SecondBoss.MapNodeAssetPaths (if present)
return new ReadOnlyList<string>(obj);
```

`MapNodeAssetPaths` (from `EncounterModel`, lines 154ŌĆō168) returns:
- If `BossNodeSpineResource != null`: the boss map-node `.tres` Spine skeleton resource.
- Otherwise: `[BossNodePath + ".png", BossNodePath + "_outline.png"]` ŌĆö the placeholder PNG pair.

**These are the MAP NODE assets (the Spine animation shown on the act map), NOT the combat scene assets.**

`ActModel.AssetPaths` does NOT include `AllMonsters.SelectMany(m => m.AssetPaths)`. There is no reference to `AllMonsters`, `AllEncounters`, or any monster `VisualsPath` in `ActModel.AssetPaths`.

Combat scene loading happens separately via `PreloadManager.LoadRoomCombatAssets` (lines 80ŌĆō83), which calls `encounter.GetAssetPaths(runState)`. That method iterates `MonstersWithSlots` (i.e., the encounter must have already generated its monsters). This is called just-in-time when entering the combat room.

**CONTRADICTION WITH SPEC (v3, line 73):**

The spec states:
> *"if `PreloadManager.LoadActAssets(runState.Act)` has already loaded the act's monster scenes (it does ŌĆö `ActModel.AssetPaths` includes monster `AssetPaths` for all `AllPossibleMonsters`)"*

**This is false.** `ActModel.AssetPaths` does NOT include monster combat scene paths (`creature_visuals/<id>.tscn`). It only includes map-node assets (the `.tres` or PNG shown on the act map) and background images. The combat scenes for boss candidates are not in the preload cache when the boss vote fires.

**Practical implication:** Our pre-warm at vote-start (calling `PreloadManager.Cache.GetScene(monster.AssetPaths.First())` for each candidate) will be a **cold load** on every first boss vote of a run ŌĆö not a silent no-op. Vanilla's `Log.Warn("Asset not cached: <path>")` will fire once per candidate per first vote. On subsequent votes in the same run (if the player saves/loads or the act caches the scenes via another path), it may already be in cache. But the common case for the first boss encounter is a cold synchronous load.

**Impact on the spec's pre-warm design:** The functionality is still correct ŌĆö `GetScene` on a cache miss falls back to synchronous load (verified in the spec via `AssetCache` decompile). The pre-warm still ensures the scene is in cache before the factory closure fires. The `Stopwatch` telemetry will now show real non-zero load time for the first run encounter, which is the right signal. The spec's description of the pre-warm as "typically a no-op" is inaccurate for the first boss encounter.

**Verdict: SPEC CLAIM IS INCORRECT.** Pre-warm will be a cold synchronous load on first boss encounter, not a cache hit. This changes the telemetry expectations but does NOT break the implementation ŌĆö the synchronous fallback in `AssetCache.GetScene` handles cold loads correctly.

## Step 7: Commit decision

**One finding contradicts the spec (Step 4).** Committing spike notes and flagging to the controller.

**Contradiction summary:**

The spec v3 (line 73) claims `ActModel.AssetPaths` includes monster combat scene paths for all possible monsters, making the vote-start pre-warm a no-op cache hit in the common case. Decompile verification shows `ActModel.AssetPaths` only includes map-node assets and background images ŌĆö it never references `AllMonsters` or `AllEncounters`. The combat scenes (`creature_visuals/<id>.tscn`) are loaded just-in-time when entering combat via `LoadRoomCombatAssets`.

**What this means for Task 2:**

1. The pre-warm code in `BossVotePatch.PreWarmBossVisuals` is still correct and necessary ŌĆö it's not a "nice to have" optimization, it's the only thing ensuring the scene is in cache before the factory fires.
2. The `Stopwatch` telemetry will show real (non-trivial) load times on first boss encounters. Operators should not be surprised by this.
3. The spec's phrase "typically hits the cache and is a no-op (no log noise)" should be revised. The accurate statement is: "will be a cold synchronous load on first boss encounter; fires vanilla `Log.Warn("Asset not cached: ...")` once per candidate."
4. No code changes are blocked ŌĆö the implementation design remains valid. Only the commentary in the spec is wrong.

The spec's code examples, API surface claims, and implementation pattern are all unaffected by this finding.
