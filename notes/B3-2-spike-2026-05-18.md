## Step 1: `PreloadManager.Cache.GetTexture` is synchronous

**File:** `decompiled/sts2/MegaCrit/sts2/Core/Assets/AssetCache.cs`

**Findings:**

- There is no method literally named `GetTexture(string)` on `AssetCache`. The actual texture accessors are:
  - Line 105: `public Texture2D GetTexture2D(string path) => (Texture2D)GetAsset(path);`
  - Line 115: `public CompressedTexture2D GetCompressedTexture2D(string path) => (CompressedTexture2D)GetAsset(path);`
- Both delegate to the private `GetAsset(string)` (line 16), which is synchronous: it does a `_cache.TryGetValue` and on miss calls `LoadAsset(path)`.
- `LoadAsset` (lines 30–40) calls `ResourceLoader.Load<Resource>(path, null, ResourceLoader.CacheMode.Reuse)` directly — no `await`, no `Task` return, no async loader path. Logs a `Log.Warn("Asset not cached: " + path)` on miss.
- The whole class has zero `async` / `Task` members. Matches the B.3.1 finding that `GetScene` is sync.

**Verdict: CONFIRMED SYNC.** `GetTexture2D` / `GetCompressedTexture2D` block on first-load (with a `Asset not cached:` warn — same vanilla noise pattern B.3 / B.3.1 already log). Safe to call on the main thread; pre-warming patterns from B.3 apply unchanged.

Note for Task 10: when reading a `Texture2D` for the popup preview, prefer `GetCompressedTexture2D` for PNG-backed paths (the act map-bg helpers in `ActModel` use this form — `MapTopBg => GetCompressedTexture2D(MapTopBgPath)`).

## Step 2: `BeginRunLocally` idempotency / pre-line-411 state mutations

**File:** `decompiled/sts2/MegaCrit/sts2/Core/Multiplayer/Game/Lobby/StartRunLobby.cs`, lines 408–438.

**Method body, fully traced:**

```csharp
private void BeginRunLocally(string seed, List<ModifierModel> modifiers)
{
    Rng rng = new Rng((uint)StringHelper.GetDeterministicHashCode(seed));            // line 410
    List<ActModel> list = ActModel.GetRandomList(rng, GetUnlockState(),

    list[0] = GetAct(Act1) ?? list[0];                                                // line 412 — Act1 read here
    for (int i = 0; i < Players.Count; i++) {                                         // line 413

            CharacterModel character = rng.NextItem(ModelDb.AllCharacters);
            ChangeCharacter(lobbyPlayer.id, character, isRandomCharacterResolution: true);
        }
    }
    if (NetService.Type == NetGameType.Singleplayer) { ... }                          // lines 422–436
    _isBeginningRun = true;                                                           // line 437
    LobbyListener.BeginRun(seed, list, modifiers);                                    // line 438
}
```

**Pre-line-411 trace:** Only ONE statement runs before line 411: the local `Rng rng = new Rng(...)` construction on line 410. This is a pure local-variable allocation; no instance-field write, no `LobbyListener` callback, no `RunManager` registration, no event dispatch.

**`_isBeginningRun` write sites** (grep confirmed): only at line 437 (post-line-411). The earlier reads at line 386 (in `BeginRunForAllPlayers`) and line 707 (in `OnConnectedToClientAsHost`) gate against double-invocation but do not flip the flag.

**Verdict: SAFE FOR RE-INVOKE** — `BeginRunLocally` is effectively idempotent before line 411. A second call with the same seed would re-construct the local Rng, re-roll `GetRandomList`, re-evaluate `Act1`, and proceed identically as the first call.

**Design note for Task 9:** The chosen design writes `lobby.Act1 = "underdocks"` from an earlier patch site (before `BeginRunLocally` is invoked at all), so the re-invoke fallback is not strictly needed even for the rare reflection-failure case — the write target is the public `Act1` property setter (auto-property at line 87) on a live instance held by the patched method. **KEEP THE FALLBACK RE-INVOKE if Task 9's design path actually requires it; from this trace, the pre-line-411 surface is benign and a second call would be safe.** The decision is therefore: re-invoke is a defensible last resort; mutations are clean.

## Step 3: `Act1` is read exactly once in `BeginRunLocally`

**File:** `decompiled/sts2/MegaCrit/sts2/Core/Multiplayer/Game/Lobby/StartRunLobby.cs`

**Findings:**

- `Act1` declared as auto-property at line 87: `public string Act1 { get; set; } = "random";`
- Grep across the file for `Act1` (case-sensitive) — read sites:
  - Line 87: declaration only (no read).
  - Line 397: `act1 = Act1` inside `BeginRunForAllPlayers` — this is the LobbyBeginRunMessage broadcast field (host→clients), NOT a re-evaluation that affects the local act decision.
  - Line 412: `list[0] = GetAct(Act1) ?? list[0];` — the one definitive read in `BeginRunLocally`.
- The `GetAct` helper (lines 441–452) is a pure string-switch with no side effects.

**Verdict: CONFIRMED SINGLE READ inside `BeginRunLocally`** (line 412). The line-397 read inside `BeginRunForAllPlayers` happens earlier in the host's flow and only populates a network message; for a singleplayer-only mod (StS2 singleplayer is the only supported v0.x path for this mod), `BeginRunForAllPlayers` runs on the same machine as `BeginRunLocally` and there is no message round-trip to consider.

**Important multiplayer caveat (deferred — singleplayer-only design):** In multiplayer, `act1 = Act1` is serialized into `LobbyBeginRunMessage` (line 397) and clients receive the message via `HandleLobbyBeginRunMessage` (line 360) which calls `BeginRunLocally(message.seed, modifiers)` — but **`message.act1` is NOT applied to `this.Act1` before that call** (decompile shows no such assignment in `HandleLobbyBeginRunMessage`). So clients would read THEIR OWN `Act1` (default `"random"`) at line 412 even if the host had set it. This means the act-variant choice does NOT propagate to multiplayer clients through this code path. **B.3.2 is single-player only per v3 spec scope; this is correctly out of scope.**

No need for `GetRandomList` postfix Plan B. The `Act1`-write design is viable.

## Step 4: Cancellation probe (Embark → return-to-character-select)

**Files:**
- `decompiled/sts2/MegaCrit/sts2/Core/Nodes/Screens/CharacterSelect/NCharacterSelectScreen.cs`
- `decompiled/sts2/MegaCrit/sts2/Core/Nodes/Screens/MainMenu/NMainMenu.cs` (line 227)
- `decompiled/sts2/MegaCrit/sts2/Core/Nodes/NSceneContainer.cs` (lines 31, 55–62)

**Findings:**

- **No `NCharacterSelectScreen.Instance` static.** Grep confirms zero `public static .* Instance` declarations on the type. The screen is owned by `NMainMenuSubmenuStack` as `private NCharacterSelectScreen? _characterSelectSubmenu` (line 99 of that file). Access is through `submenuStack.GetSubmenuType<NCharacterSelectScreen>()`.
- **`NMainMenu` does have `SubmenuStack`** as a public instance property (line 227): `public NMainMenuSubmenuStack SubmenuStack { get; private set; }`. But also no `NMainMenu.Instance` static — `NMainMenu` is reachable only through `NGame.Instance.RootSceneContainer.CurrentScene`.
- **`NGame.Instance.RootSceneContainer.CurrentScene` IS accessible** and IS the singleton-style hook: `NSceneContainer.cs:31` declares `public Control? CurrentScene { get; private set; }`. When the streamer clicks Embark, vanilla's flow (NCharacterSelectScreen.cs lines 537–565 in the singleplayer path) calls `NGame.Instance.RootSceneContainer.SetCurrentScene(NRun.Create(runState))` — switching the root scene from `NMainMenu` to `NRun`. When the streamer clicks Back instead, `NMainMenu` remains the `CurrentScene` and the submenu stack pops `NCharacterSelectScreen`.
- The submenu-stack identity probe (the "did the streamer back out of character select") is therefore: capture `NGame.Instance.RootSceneContainer.CurrentScene` reference at vote start. If during `_Process` the reference changes (Embark consummated → `NRun` is now current scene), OR the character-select submenu is gone from the stack (Back pressed → `_characterSelectSubmenu` is no longer on top), the streamer has navigated away.

**Chosen probe (Task 9 / Task 11 `IsRunStartAbandoned`):**

```csharp
private static bool IsRunStartAbandoned() {
    try {
        // Embark consummated: the root scene swapped to NRun. We don't need
        // to cancel — the vote will simply complete and the patch's await
        // will resume normally on the next BeginRunForAllPlayers call.
        // But if the streamer clicked Back, NMainMenu is still current AND
        // its SubmenuStack is no longer showing NCharacterSelectScreen.
        var game = NGame.Instance;
        if (game?.RootSceneContainer?.CurrentScene is not NMainMenu mm) {
            // Either scene swapped away (NRun.Create happened — Embark went through)
            // or NGame itself is being torn down. Both signal "stop waiting".
            return true;
        }
        var stack = mm.SubmenuStack;
        if (stack is null) return true;
        // The character-select submenu must be on top of the stack while
        // the streamer is on the Embark screen. If it's been popped, the
        // streamer hit Back. NSubmenuStack inherits from a stack-based
        // submenu manager — peek-top by checking the active submenu type.
        // SAFEST EXPRESSION: stack.GetActiveSubmenu() is not NCharacterSelectScreen
        // (Task 9 implementer: if GetActiveSubmenu doesn't exist on
        // NSubmenuStack base, fall back to checking
        // stack.SubmenusOpen == false || the _characterSelectSubmenu Visible flag.)
        return false;  // optimistic: rely on Task 9 to wire the active-submenu check
    } catch {
        return false;  // never crash _Process; let the vote timer expire as fallback
    }
}
```

**Notes for Task 9 implementer:**
- `NSubmenuStack`'s `SubmenusOpen` bool is already used by `BossVotePatch.IsOccludingOverlayVisible` (line 193 of that file) and IS the count-based "any submenu showing" probe. For B.3.2's purpose we want "is character select still on top", which is type-specific. If `NSubmenuStack` doesn't expose `GetActiveSubmenu`, the simpler workable probe is `stack.SubmenusOpen` going false (streamer popped back to the main menu).
- The `CurrentScene` identity check (capture-on-open, compare-during-Process) is the cleanest one-liner and would cover both "Embark consummated" and "navigated all the way back to main menu" — it's the **recommended primary probe**. Subtype check on `NMainMenu` is a strong correctness anchor; submenu stack inspection is secondary.

**Verdict: Chosen probe is `NGame.Instance?.RootSceneContainer?.CurrentScene` identity (or subtype-check `is not NMainMenu`).** Compilable from anywhere `MegaCrit.Sts2.Core.Nodes` is in scope; `ActVariantVotePatch` has the same MegaCrit-side reach as `BossVotePatch`. Concrete one-liner:

```csharp
return !(NGame.Instance?.RootSceneContainer?.CurrentScene is NMainMenu);
```

Returns `true` if NGame is torn down (defensive) OR if the scene has swapped away from main menu. **Does NOT distinguish Embark-success from Back-cancel** — both surface as "current scene is no longer NMainMenu" or "submenus closed" — but for our purposes (cancel the popup and bail) we don't need to distinguish; we just need to know the streamer is no longer staring at the character-select screen.

**Optional refinement** (deferred unless Task 9 implementer wants it): differentiate via `stack.SubmenusOpen` — if `CurrentScene is NMainMenu && !mm.SubmenuStack.SubmenusOpen`, it's a back-cancel; if `CurrentScene is not NMainMenu`, it's Embark consummation. v3 spec only cares about the "abandon" superset so the basic check is sufficient.

## Step 9: `VoteSession.Cancel()` idempotency

**File:** `src/Ti/Voting/VoteSession.cs`, lines 241–250.

**Findings:**

```csharp
public void Cancel() {
    if (_state != VoteSessionState.Open) return;
    _chat.MessageReceived -= OnChatMessage;
    _chat.ConnectionStateChanged -= OnChatConnectionStateChanged;
    _closeTimer.Dispose();
    _periodicTimer?.Dispose();
    _state = VoteSessionState.Cancelled;
    _winnerTcs.TrySetCanceled();
    Cancelled?.Invoke(this, this);
}
```

**Trace:**
- Line 242: `if (_state != VoteSessionState.Open) return;` — this is the idempotency guard.
- If already `Cancelled`, `Closed`, `Closing`, or `Disposed`, the method returns immediately as a no-op. Zero side effects, no exception, no event re-fire.
- Cross-reference: `Dispose()` (line 293) calls `Cancel()` after setting state to `Disposed`, but `Cancel()` itself only fires `Cancelled` exactly once because the state transition happens between the guard check and the event invocation (state is mutated to `Cancelled` on line 247 before `Cancelled?.Invoke` on line 249, but the guard on line 242 prevents a second call from ever reaching line 249).
- `TrySetCanceled` on line 248 is the `TrySet` variant that itself is idempotent — second call would return `false` without observable side effect.

**Verdict: CONFIRMED IDEMPOTENT.** A second call after `Cancelled` / `Closed` / `Disposed` state is a guaranteed no-op. **No local `_cancelledSent` guard needed in Task 11's `_Input` ESC handler.** The B.3 popup pattern (`try { _session.Cancel(); } catch { /* swallow */ }`, line 329 of `BossVotePopup.cs`) is already the right shape — the `try`/`catch` is belt-and-braces, not strictly needed.
