## Spike 1 — `AllBossEncounters` pool sizes per vanilla act

**Answer**: Every vanilla act has exactly 3 boss encounters. Sufficient for a 3-option chat vote.

| Act | Class | Boss encounters |
|---|---|---|
| Overgrowth (Act 1 standard) | `Acts/Overgrowth.cs` | `VantomBoss`, `CeremonialBeastBoss`, `TheKinBoss` |
| Underdocks (Act 1 alt) | `Acts/Underdocks.cs` | `WaterfallGiantBoss`, `SoulFyshBoss`, `LagavulinMatriarchBoss` |
| Hive (Act 2) | `Acts/Hive.cs` | `TheInsatiableBoss`, `KnowledgeDemonBoss`, `KaiserCrabBoss` |
| Glory (Act 3) | `Acts/Glory.cs` | `QueenBoss`, `TestSubjectBoss`, `DoormakerBoss` |

Source: each act's `GenerateAllEncounters()` override — counting entries with `Boss` suffix name
conventions and cross-checking against `BossDiscoveryOrder`. E.g. `Overgrowth.cs:67-94`
returns 22 encounters; `BossDiscoveryOrder` (line 19-24) lists exactly the three `*Boss`
entries. Same pattern verified in Hive, Glory, Underdocks.

**Implication**: a 3-option poll uses the full boss pool. No need to randomly subsample. The
v3 spec's "sample 3 from the pool" language should be revised to "present all 3 from the pool".

## Spike 2 — `AllBossEncounters` is NOT unlock-filtered

**Answer**: Confirmed. No unlock filter on `AllBossEncounters`.

`ActModel.cs:119`:
```csharp
public IEnumerable<EncounterModel> AllBossEncounters =>
    _allBossEncounters ?? (_allBossEncounters =
        AllEncounters.Where((EncounterModel e) => e.RoomType == RoomType.Boss));
```

Pure RoomType filter; no `UnlockState` parameter. Compare to `GetUnlockedAncients(UnlockState)`
at `ActModel.cs:197` which explicitly takes and applies unlock state. The boss pool is always
the full set of 3.

The note in the v3 spec — "unlocked-everything streamers are unaffected" — still holds for
completeness, but the distinction doesn't matter in practice: `AllBossEncounters` ignores
unlock state entirely.

## Spike 3 — `HasSecondBoss` timing

**Answer**: Set exactly once during `RunManager.GenerateRooms()` at run start, exclusively for
the final act on A10+ (`AscensionLevel.DoubleBoss`). Stable for the rest of the run.

`RunManager.cs:499-503`:
```csharp
if (i == State.Acts.Count - 1 && AscensionManager.HasLevel(AscensionLevel.DoubleBoss))
{
    EncounterModel secondBossEncounter = State.Rng.UpFront.NextItem(
        act.AllBossEncounters.Where((EncounterModel e) => e.Id != act.BossEncounter.Id));
    act.SetSecondBossEncounter(secondBossEncounter);
}
```

`SetSecondBossEncounter` (`ActModel.cs:366`) calls `AssertMutable()` — it can only be called on
a mutable clone, and runs call it exactly once here. No mid-run path exists that calls it again.
`HasSecondBoss` (`ActModel.cs:164`) is a simple property wrapper:
```csharp
public bool HasSecondBoss => _rooms.HasSecondBoss;
```

**Lifecycle**: `HasSecondBoss` is false until `GenerateRooms()` runs for the final act. After that
it is true for the rest of the run. It is stable from the moment the act's rooms are generated.

**B.3 implication**: For v1 (primary-boss vote only), ignore `HasSecondBoss`. If/when B.3 grows
to a second-boss vote, a separate chest-room trigger on Act 3's second chest (if any) would handle
it — but that's explicitly deferred.

## Spike 4 — `OnProceedButtonPressed` signature

**SURPRISE — this changes the implementation shape.**

**Answer**: `OnProceedButtonPressed` is **private** and takes one **`NButton`** parameter.
It is **not** parameterless.

`NTreasureRoom.cs:198`:
```csharp
private void OnProceedButtonPressed(NButton _)
{
    if (_proceedButton.IsSkip) { ... }
    else { TaskHelper.RunSafely(RunManager.Instance.ProceedFromTerminalRewardsScreen()); }
}
```

The method is wired as a delegate:
```csharp
// NTreasureRoom.cs:177
_proceedButton.Connect(NClickableControl.SignalName.Released, Callable.From<NButton>(OnProceedButtonPressed));
```

**Implication**: The patch's synthetic re-call **requires reflection**, same as
`CardRewardVotePatch._selectCardMethod`. Use:
```csharp
private static readonly Lazy<MethodInfo?> _proceedMethod =
    new(() => AccessTools.Method(typeof(NTreasureRoom), "OnProceedButtonPressed",
                                 new[] { typeof(NButton) }));
```

The synthetic re-call passes a null/dummy `NButton` argument (the parameter is discarded —
named `_` — and the method body only uses `_proceedButton` from the instance field, not the
passed argument). Specifically, `_proceedButton` is a field on `NTreasureRoom`, not the arg.

The Harmony patch itself targets `OnProceedButtonPressed(NButton)` — a private instance method.
That is legal under HarmonyLib (`[HarmonyPatch(typeof(NTreasureRoom), "OnProceedButtonPressed")]`
plus a `Prepare` that verifies the parameter list).

## Spike 5 — `BossVotePatch.Prepare` reflection surface

**Answer**: Because `OnProceedButtonPressed` is private with a parameter, `Prepare` needs to:

1. **Hard check**: cache `_proceedMethod` (the `MethodInfo` for `OnProceedButtonPressed(NButton)`)
   and abort if null.
2. **Hard check**: verify the cached method has exactly one parameter of type `NButton`. This is
   the load-bearing shape the synthetic re-call depends on.

No need to reflect `_proceedButton` or `_isRelicCollectionOpen` in `Prepare` — the patch body
doesn't read them via reflection (they're accessible via the `__instance` parameter at Harmony
prefix time, since they are instance fields on `NTreasureRoom` but still private — wait, see
Spike 10 for `_isRelicCollectionOpen` access path).

**Prepare checklist** (mirrors `CardRewardVotePatch.Prepare`):
```
Registration-time (original == null):
  - _proceedMethod.Value is null → abort (hard check)
  - _proceedMethod.Value.GetParameters().Length != 1 → abort (hard check)
  - _proceedMethod.Value.GetParameters()[0].ParameterType != typeof(NButton) → abort (hard check)

Per-method (original != null):
  - Verify original.GetParameters() matches NButton signature → abort if mismatch
  - Log Info on success
```

## Spike 6 — `CurrentActIndex` base

**Answer**: **0-based**. Act 1 = index 0, Act 2 = index 1, Act 3 = index 2.

Evidence 1 — new-run construction (`RunState.cs:179`):
```csharp
RunState runState = CreateShared(..., currentActIndex: 0, ...);
```

Evidence 2 — final-act guard (`RunManager.cs:967`):
```csharp
if (State.CurrentActIndex >= State.Acts.Count - 1) { /* game over flow */ }
else { await EnterAct(State.CurrentActIndex + 1); }
```
`State.Acts.Count - 1` = 2 for a standard 3-act run. `CurrentActIndex >= 2` is the "on last act"
condition. This is 0-based end-to-end.

Evidence 3 — `Acts[CurrentActIndex]` (`RunState.cs:63`):
```csharp
public ActModel Act => Acts[CurrentActIndex];
```
Direct list indexing with no offset.

**v3 spec adjustment**: the popup title `$"Act {runState.CurrentActIndex + 1} Boss Vote"` is
correct as written — `+1` converts 0-based to display-friendly 1-based. No change needed.

## Spike 7 — `BossNodePath` extension convention

**Answer**: `BossNodePath` returns a `.tres` path (no `.png`). The PNG fallback is
`BossNodePath + ".png"` (appended directly, making `.tres.png`).

`EncounterModel.cs:138`:
```csharp
public virtual string BossNodePath =>
    $"res://animations/map/{base.Id.Entry.ToLowerInvariant()}/{base.Id.Entry.ToLowerInvariant()}_node_skel_data.tres";
```

`EncounterModel.cs:154-168` (the `MapNodeAssetPaths` getter):
```csharp
if (BossNodeSpineResource != null)
    return new ReadOnlySingleElementList<string>(BossNodePath);     // Spine path (.tres)
return new ReadOnlyArray<string>(new string[2]
{
    BossNodePath + ".png",           // → "..._node_skel_data.tres.png"
    BossNodePath + "_outline.png"    // → "..._node_skel_data.tres_outline.png"
});
```

**Exact convention**: PNG path = `BossNodePath + ".png"` — the `.png` extension is appended
to the entire `.tres` filename, yielding `..._node_skel_data.tres.png`. This is an unusual
convention (not replacing `.tres` with `.png`), but it is what vanilla uses for the map icon.

**For the popup**: load the PNG path via `ResourceLoader.Exists(BossNodePath)` first; if the
Spine resource exists use MegaSpine; otherwise `ImageTexture.CreateFromImage(Image.LoadFromFile(
BossNodePath + ".png"))` gives a static fallback. For v1, using the PNG fallback is safer and
avoids pulling in MegaSpine bindings.

## Spike 8 — `EncounterModel.Title.GetFormattedText()` plain text vs BBCode

**Answer**: Boss titles are **plain text** — no BBCode in the output. Safe to use in any
`Label`, `RichTextLabel`, or as a chat message string without post-processing.

Reasoning:
1. `EncounterModel.Title` is `new LocString("encounters", base.Id.Entry + ".title")`
   (`EncounterModel.cs:152`). No variables are added to it.
2. `LocString.GetFormattedText()` calls `LocManager.SmartFormat(this, _variables)`
   (`LocString.cs:62-65`). With an empty `_variables` dictionary, SmartFormat acts as a
   passthrough: it reads the raw text and returns it unchanged (no `{variable:formatter}`
   placeholders to expand).
3. BBCode in the localization system is only generated by SmartFormat extension formatters like
   `EnergyIconsFormatter` (which outputs `[img]...[/img]`) or `HighlightDifferencesFormatter`.
   These fire only when the loc string template contains formatter references like
   `{energyCount:energyIcons}`. Boss name strings (e.g., `"Vantom"`, `"The Kin"`,
   `"The Insatiable"`) are simple proper nouns with no such placeholders.
4. `GetRawText()` (`LocString.cs:68`) is the underlying JSON lookup — it returns the raw string
   from the localization table. For boss titles this is the human-readable name with no markup.

**Practical implication**: `encounter.Title.GetFormattedText()` and `encounter.Title.GetRawText()`
return identical output for boss names. Use `GetFormattedText()` for consistency with the rest of
the codebase (e.g., `CardRewardVotePatch` uses `o.Card.Title` which follows the same pattern).
`RichTextLabel` with `BbcodeEnabled = true` is fine for the popup (harmless for plain text) and
is consistent with the rest of the mod's UI layer.

## Spike 9 — Safe `CanvasLayer.Layer` index

**Answer**: **No vanilla C# code sets `CanvasLayer.Layer` at all.** Layer index 100 has no
known conflict. 100 is safe.

Exhaustive search of `decompiled/sts2/` for `.Layer =` patterns found zero matches. All vanilla
UI containers (`NModalContainer`, `NOverlayStack`, `NTreasureRoom`, etc.) extend `Control`, not
`CanvasLayer`. The CanvasLayer abstraction is not used in any decompiled C# source — vanilla
relies on scene-graph ordering and Control z-index for UI layering.

The existing mod code (`VoteTallyLabel.cs:39`) has a comment noting CanvasLayer as a future
option; it currently attaches directly under root. `BossVotePopup` will use `CanvasLayer` with
`Layer = 100` for proper overlay semantics, and no vanilla code will conflict.

If a scene file (`.tscn`) assigns a layer, that would be independent of C# and not visible in
the decompile. However, the standard convention for modding overlays in Godot is high layer
values (100+) precisely because scene-file layers are typically single digits. 100 remains a
safe default.

## Spike 10 — `_isRelicCollectionOpen` interaction with `OnProceedButtonPressed`

**Answer**: `OnProceedButtonPressed` does **not** check `_isRelicCollectionOpen` internally.
However, the Proceed button is **disabled** while the relic collection is open, so the method
will never fire during relic selection unless the vote patch synthesizes a re-call.

Full analysis of `NTreasureRoom`:

`OnProceedButtonPressed` body (`NTreasureRoom.cs:198-213`) only checks `_proceedButton.IsSkip`
to branch between skip-relic and normal proceed. No guard on `_isRelicCollectionOpen`.

But `OpenChest()` (`NTreasureRoom.cs:241-279`) manages Proceed button state:
- Line 244: `_proceedButton.Disable()` — disabled at chest-open start.
- Line 265: re-enabled via `EnableSkipAfterDelay(2.5f, ...)` with "Skip" text (singleplayer).
- Line 270: `_proceedButton.Disable()` again when relic picking begins.
- Line 275: `_proceedButton.Enable()` — re-enabled only after relic picking **finishes** (i.e.,
  player has selected a relic or explicitly skipped it).
- Line 277: `NMapScreen.Instance.SetTravelEnabled(enabled: true)` — travel gate opened.

`_isRelicCollectionOpen` is set to `true` at line 258 (before relic picking) and back to `false`
at line 272 (after picking). During the window when `_isRelicCollectionOpen == true`, the Proceed
button is **disabled** (from line 270). The button is only re-enabled at line 275 after relic
picking is done and `_isRelicCollectionOpen` has been reset to `false`.

**Result**: by the time the player can click Proceed (and our prefix fires), `_isRelicCollectionOpen`
is always `false`. The vote fires post-relic-selection, which is the correct timing.

**Smoke I setup implication**: no special test setup needed. The test should call the prefix
directly (not through the button signal path) with an `NTreasureRoom` instance that has
`_hasChestBeenOpened == true` and `_isRelicCollectionOpen == false` — the natural post-relic state.
Since `_proceedButton.IsSkip` controls the branch inside the method body, the re-call after vote
close should pass `_proceedButton.IsSkip = false` (normal Proceed, not skip) to trigger the
`ProceedFromTerminalRewardsScreen()` path. The `NButton` argument itself is discarded (`_`), so
passing `null` or a dummy `NButton` to the reflected re-call is safe.

**One caveat**: `_proceedButton.Enable()` is called before `NMapScreen.Instance.SetTravelEnabled`,
so by the time our prefix intercepts the button press, the proceed button has just been enabled and
travel is about to be opened. Our prefix should suppress the original call (return `false`) so
travel stays closed during the vote; the synthetic re-call after vote close will run the original
body and open travel normally.
