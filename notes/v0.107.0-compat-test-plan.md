# v0.107.0 compatibility — in-game test plan

**Game update**: Beta v0.106.1 → v0.107.0 (installed 2026-06-05).
**Automated verification**: 12-agent decompile diff + binding audit. 11/12 areas PASS; 1 needs-attention (non-blocking). Clean build against v0.107.0 sts2.dll, 381/381 tests pass. Game now ships **Harmony 2.4.2.0**.

## What actually changed in our surface

The only relevant game-code change is **`MegaSprite.GetAnimationState()`** ([MegaSprite.cs:127](../decompiled/sts2-v0.107.0/MegaCrit.Sts2.Core.Bindings.MegaSpine/MegaSprite.cs#L127)): it now **throws** `InvalidOperationException` when the Spine skeleton isn't initialized yet, where v0.106.1 returned a (possibly null) wrapper. This is on the boss-vote portrait animation path (both the regular-monster animator via `CreatureAnimator` ctor, and the Kaiser Crab direct call). **Both mod call sites are inside try/catch**, so the failure mode is bounded: a caught Warn + a static (first-frame) boss portrait. No crash, no vote-logic impact, no boss-swap impact.

Everything else (`NTreasureRoom`, `MapCmd`, `ActModel`, `NCardRewardSelectionScreen`, `CardReward`, `NEventRoom`, `Pael`, both `OnEmbarkPressed` screens, `NActDropdown`, `NTopBarMapButton.OnRelease`, `NModInfoContainer.Fill`, `RunManager.DebugOnlyGetState/IsAbandoned`, the `NRun...SubmenuStack.Stack.SubmenusOpen` occlusion chain, dev-console registration) is **byte-identical or cosmetic-only** on the members we bind.

## Test plan (tiered — do A and B; C is a quick smoke pass)

Logs: `%APPDATA%\SlayTheSpire2\logs\godot.log`. Connect Twitch/YouTube chat first so votes fire.

### A. Boot sanity (30 seconds)
1. Launch the game modded. In `godot.log` confirm: the mod **version hash matches git HEAD** (else `dist/` is stale — rerun `install.ps1`), a `Harmony patched N method(s)` line, and **NO** lines containing `hard check failed`, `target signature mismatch`, or `Postfix threw`. A boot with all patches bound is the single strongest signal.

### B. Tier 1 — boss-vote portrait animations (the one thing that could regress)
2. Start a single-player run, reach the **Act 1 chest room**, click **Proceed**. Confirm the boss-vote popup appears (3 columns + countdown) and **each portrait animates** (gentle idle loop), not a frozen frame.
3. Tail `godot.log` during that vote. **Red flag**: any `[boss-vote] deferred animation setup failed` or `...skeleton finished initializing` Warn = the new throw fired and that portrait fell back to static. Vote still works, art degraded. (If you see these a lot, that's the cue to take the optional `TryGetAnimationState()` hardening — see below.)
4. Reroll to a **Kaiser Crab** option (dev console `` ` `` → `rerollvote` until `KAISER_CRAB_BOSS` appears). Confirm the crab (body + both claws) renders and idles; no `Kaiser Crab deferred setup failed`.
5. Finish the vote (chat `#0/#1/#2`, or `votenow`). Confirm `[boss-vote] resume: applying boss swap` and the map boss matches the chat pick.

### C. Tier 2 — one quick smoke per patch (confirms runtime Harmony binding)
A game update is the classic patch-breaker, so eyeball each once even though code verification is green:
6. **Card reward**: win a fight → "Add a card" → vote fires, winning card added.
7. **Card skip**: with `cardSkipsPerAct=1`, skip a reward → counter label + "skipped a card reward" receipt; next act → reset receipt.
8. **Ancient**: reach an Act 2/3 ancient (e.g. Pael) → click an option → vote fires; Pael's middle option is never Pael's Wing.
9. **Act-variant**: from character select, click **Embark** → Underdocks/Overgrowth popup → winner loads. (Also test **Custom mode** Embark — separate screen target.)
10. **Map guard**: open any vote, then click the **Map** button / press **M** → blocked while the vote runs; works again after.
11. **Settings panel**: Mods screen → click the mod row → custom panel renders (slider, checkboxes, dropdowns); no `Postfix threw`.
12. **Dev console**: `` ` `` → confirm `rerollvote`, `votenow`, `resetskips` are listed.
